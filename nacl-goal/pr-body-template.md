# /nacl-goal PR body template

The `intake` alias renders the goal-run PR body from `plan.lock.json` at
three points in the lifecycle:

1. **LOCK PLAN** (Flow step 5) — initial WIP body, before any atom ships.
   Written to `.tl/goal-runs/<run_id>/pr-body.md`. `/nacl-tl-ship` reads
   this file when it opens the PR on the first atom push.
2. **Atom state transitions** (Flow step 9) — refresh on `implementing`,
   `shipped`, `verified`, `failed` for each atom.
3. **Terminal state** (`goal_ok` / `goal_blocked`) — final body with
   outcome, links, and audit pointers.

Re-rendering is idempotent: the wrapper regenerates the full body each
time from current state, never patches it in place.

---

## Template

The body is plain Markdown with stable anchor sections so future tooling
can parse it. Every PR body begins with the run-id footer so reviewers
can find the artifact directory.

```markdown
## /nacl-goal intake — autonomous goal run

**Goal:** `<sanitized goal preview, max 300 chars, escape backticks>`

**Run ID:** `<run_id>`
**Branch:** `<branch>`
**Deploy target:** `<staging|dev-only>`
**Status:** `<WIP — pending atoms | WIP — implementing | WIP — delivering | GOAL_OK | GOAL_BLOCKED — <code>>`

### Atoms

| ID | Type | Risk | Linked UC | State | Last commit |
|---|---|---|---|---|---|
| `atom-001` | BUG | L1 | UC-804 | `verified` | `abc1234` |
| `atom-002` | FEATURE_SMALL | L1 | UC-804 | `shipped` | `def5678` |

### Pre-existing commits on this branch

<!-- rendered ONLY when plan.lock.json.branch_mode == "current"
     AND prior_unpushed_commits > 0; omitted otherwise -->

This run executed on an existing feature branch (`branch_mode: current`).
`<prior_unpushed_commits>` commit(s) in range
`<branch_base_sha short>..<first goal-run commit short>` predate the goal
run and are part of the user's ongoing batch on this branch. Goal-run
commits carry the `Goal-run-id:` trailer — `git log --grep "Goal-run-id: <run_id>"`
separates them cleanly.

### Authorization

- Strict mode: `<true|false>`
- Envelope gates pre-authorized: `<spec-first-prerequisite, spec-gap-routing>` (or `none`)
- Owner: `<user_email>`

### Evidence

- CI: `<pending|success|failure>` (link to GitHub Actions run)
- Staging health: `<healthy|degraded|failed|n/a>` (link if available)
- Regression check: `<clean|N regressions detected|best-effort>` (link to `regression-diff.json` if non-empty)
- Deployed SHA matches PR head: `<true|false|n/a>`

### Artifacts (gitignored — local only)

- Plan lock: `.tl/goal-runs/<run_id>/plan.lock.json`
- Progress log: `.tl/goal-runs/<run_id>/progress.jsonl`
- Exception audit: `.tl/goal-runs/<run_id>/exceptions.log`
- All wrapper exception YAMLs: `.tl/exceptions/goal-runs/<run_id>/`

---

🤖 Generated by `/nacl-goal intake` (2.10.1)
Goal-run-id: `<run_id>` — please leave intact for traceability
```

## Sanitization rules for the goal preview

The PR is publicly visible (typically a GitHub repo, possibly internet-facing
on an open source project). The original free-text goal MUST NOT be embedded
raw. The same sanitization rules as the exception envelope apply:

- Max 300 characters (PR bodies have more room than YAML reason fields)
- Newlines collapsed to single spaces
- Markdown-control chars escaped or fenced: backticks (`` ` ``) escaped
  by wrapping the preview in a code span; angle brackets escaped to `&lt;`
  / `&gt;`; pipe characters in table cells escaped to `\|`
- Non-printable / control bytes stripped

The full goal text remains in the gitignored `request.json`. The PR body
preview is for orientation, not the canonical record.

## Per-atom status badges in the table

| State value | Badge in table |
|---|---|
| `pending` | `⏳ pending` |
| `implementing` | `🔧 implementing` |
| `shipped` | `📤 shipped` |
| `verified` | `✅ verified` |
| `failed` | `❌ failed` |

Emojis are user-visible PR content; they are NOT used in any internal
artifact (JSON, YAML, JSONL) per NaCl's general no-emoji rule. The PR
body is the one place reviewers benefit from quick visual scanning so
they're allowed here.

## Authorization section

When `strict_mode: true`:

```markdown
### Authorization

- Strict mode: **true** (--strict)
- Envelope gates pre-authorized: **none** (--strict refuses pre-flight if needed)
- Owner: `<user_email>`
```

When `strict_mode: false` AND no envelope gates were needed:

```markdown
### Authorization

- Strict mode: false
- Envelope gates pre-authorized: **none required for this plan**
- Owner: `<user_email>`
```

When `strict_mode: false` AND some gates were materialized:

```markdown
### Authorization

- Strict mode: false
- Envelope gates pre-authorized: `spec-first-prerequisite`, `spec-gap-routing`
- Owner: `<user_email>`
- Audit trail: `.tl/goal-runs/<run_id>/exceptions.log` (2 entries)
```

## Evidence section in terminal state

### `GOAL_OK` body

```markdown
### Evidence

- CI: ✅ success (https://github.com/.../actions/runs/12345)
- Staging health: ✅ healthy (https://staging.example.com/api/health)
- Regression check: ✅ clean (43 passed, no diff vs baseline)
- Deployed SHA matches PR head: ✅ true
- Mode: `stable_ids`
```

### `GOAL_BLOCKED_<code>` body

```markdown
### Evidence

- Block code: **GOAL_BLOCKED_NEW_REGRESSIONS_DETECTED**
- Regressions: 2 (see `.tl/goal-runs/<run_id>/regression-diff.json`)
- Resumable: false — re-run requires `--new-run` after manual investigation
- Next step: review the diff and the failing tests; either fix and re-run, or accept the regression manually
```

The user-facing block code, the resumable flag, and the next step are
copied verbatim from `refusal-catalog.md`. This is intentional — reviewers
on GitHub get the same explanation the local user got.

## Rendering implementation note

The wrapper does not call `gh pr edit` directly. It writes the body to
`.tl/goal-runs/<run_id>/pr-body.md` and `/nacl-tl-ship` (in append mode,
recognized via `NACL_SHIP_MODE=append` env var) is responsible for syncing
to GitHub on push.

This keeps the GitHub API surface owned by `/nacl-tl-ship` (and means the
PR2 inner-skill change is minimal: just read this file when opening or
updating the PR, instead of computing a body from commit history).

---

## Footer line invariant

Every PR body MUST end with:

```
Goal-run-id: `<run_id>` — please leave intact for traceability
```

This line is parsed by `intake.sh` as a fallback signal when `pr.json` is
missing (e.g. user manually closed and reopened the PR). The exact string
`Goal-run-id: ` followed by the run_id in backticks is the stable
anchor — reviewers may edit other parts of the body, but stripping the
footer breaks the traceability check.

If a reviewer needs to add commentary, it goes above the auto-generated
section (the wrapper's regeneration replaces only the autogenerated block,
bounded by `## /nacl-goal intake — autonomous goal run` header and the
footer line).
