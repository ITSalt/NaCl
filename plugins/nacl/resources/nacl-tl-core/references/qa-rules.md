# Правила E2E-тестирования через MCP Playwright (QA Rules)

## 1. Назначение

QA-тестирование -- это проверка реализованного Use Case "глазами пользователя". Агент открывает браузер, выполняет сценарий из UC (навигация, заполнение форм, клики, проверка результатов) и фиксирует результат скриншотами.

**Ключевой принцип**: QA не проверяет код -- он проверяет поведение. Тесты идут по acceptance criteria из `acceptance.md` и верифицируют, что пользователь видит ожидаемый результат при каждом шаге.

**Инструмент**: MCP Playwright -- набор инструментов для управления браузером из Claude Code.

---

## 2. Подготовка к QA-сессии

### a. Изучить acceptance criteria

Прочитать файл `.tl/tasks/UC###/acceptance.md` -- это главный чек-лист для QA. Каждый критерий из этого файла должен быть проверен.

### b. Изучить сценарий

Прочитать task-файлы для полного понимания контекста:

- `task.md` -- основной поток, альтернативные потоки, входные/выходные данные
- `impl-brief.md` -- API-эндпоинты, URL-маршруты, имена компонентов
- `result.md` -- что именно было реализовано, какие файлы затронуты

### c. Убедиться, что серверы запущены

Перед началом QA проверить, что dev-серверы работают:

```
# Проверить backend
curl http://localhost:3000/api/health

# Проверить frontend
curl http://localhost:5173
```

Если серверы не запущены, использовать `preview_start` для их запуска (конфигурация в `.claude/launch.json`).

### d. Подготовить директорию для скриншотов

```
# Очистить директорию перед запуском
rm -rf .tl/qa-screenshots/UC###/
mkdir -p .tl/qa-screenshots/UC###/
```

Директория `.tl/qa-screenshots/` должна быть в `.gitignore` -- скриншоты не коммитятся.

## 3. Построение тестового сценария

### Источник сценария

Тестовый сценарий строится на основе **Main Flow** из `task.md` и **Acceptance Criteria** из `acceptance.md`.

### Структура шагов

Каждый шаг QA-сценария состоит из:

| Элемент | Описание | Пример |
|---------|----------|--------|
| **Номер шага** | Порядковый номер | Step 01 |
| **Действие** | Что делает пользователь | Навигация на страницу создания заказа |
| **MCP-команда** | Какой инструмент Playwright использовать | `playwright_navigate` |
| **Ожидаемый результат** | Что должен увидеть пользователь | Форма с полями "Клиент" и "Товары" |
| **Проверка** | Как верифицировать результат | `playwright_get_visible_text` -- текст содержит "Создать заказ" |
| **Скриншот** | Имя файла скриншота | `step-01-navigate-to-form.png` |

### Пример сценария для UC001 (Create Order)

```
Step 01: Навигация на страницу создания заказа
  Действие: Открыть /orders/new
  Ожидание: Форма создания заказа отображается
  Проверка: Заголовок содержит "Создать заказ"
  Скриншот: step-01-navigate-to-form.png

Step 02: Выбор клиента
  Действие: Кликнуть на dropdown "Клиент", выбрать "ООО Рога и Копыта"
  Ожидание: Клиент выбран, dropdown закрыт
  Проверка: Поле "Клиент" содержит "ООО Рога и Копыта"
  Скриншот: step-02-select-client.png

Step 03: Добавление товара
  Действие: Кликнуть "Добавить товар", выбрать товар, ввести количество
  Ожидание: Товар появился в таблице, сумма рассчитана
  Проверка: Таблица содержит строку товара, итог > 0
  Скриншот: step-03-add-item.png

Step 04: Отправка формы
  Действие: Кликнуть "Создать заказ"
  Ожидание: Переход на страницу заказа, статус "Новый"
  Проверка: URL содержит /orders/, статус = "Новый"
  Скриншот: step-04-order-created.png

Step 05: Проверка альтернативного потока -- пустые товары
  Действие: Открыть /orders/new, не добавлять товары, кликнуть "Создать"
  Ожидание: Ошибка валидации
  Проверка: Текст ошибки "Добавьте хотя бы один товар"
  Скриншот: step-05-validation-error.png
```

### Шаги для альтернативных потоков

Каждый альтернативный поток из `task.md` тестируется отдельным шагом:

- Валидационные ошибки (пустые обязательные поля, невалидные данные)
- Edge cases (граничные значения, нулевые количества)
- Ошибки сервера (если можно спровоцировать через UI)

## 4. Использование MCP Playwright

### Навигация

```
playwright_navigate
  url: "http://localhost:5173/orders/new"
```

Используется для перехода на нужную страницу. Всегда указывать полный URL с портом dev-сервера.

### Заполнение форм

```
playwright_fill
  selector: "#client-select"
  value: "ООО Рога и Копыта"

playwright_fill
  selector: "input[name='quantity']"
  value: "5"
```

Используется для ввода значений в текстовые поля и поля ввода. Селекторы -- CSS-селекторы.

### Клики

```
playwright_click
  selector: "button[type='submit']"

playwright_click
  selector: ".add-item-btn"

playwright_click
  selector: "[data-testid='save-order']"
```

Для нажатия на кнопки, ссылки, элементы навигации. Предпочтительные селекторы: `data-testid` > `id` > CSS class > tag + text.

### Скриншоты

```
playwright_screenshot
  name: "step-01-navigate-to-form"
  savePng: true
```

Снимок текущего состояния страницы. Каждый значимый шаг фиксируется скриншотом.

### Проверка контента

```
playwright_get_visible_text
```

Возвращает весь видимый текст на странице. Используется для проверки:
- Заголовков и лейблов
- Сообщений об ошибках
- Данных в таблицах
- Статусов и уведомлений

### Проверка динамического состояния

```
playwright_evaluate
  script: "document.querySelector('.order-status').textContent"

playwright_evaluate
  script: "document.querySelectorAll('.order-items tr').length"

playwright_evaluate
  script: "window.location.pathname"
```

Используется для проверки:
- Значений, недоступных через visible text (например, скрытых полей)
- Количества элементов
- Текущего URL
- Состояния JavaScript-переменных

### Дополнительные инструменты

```
# Выбор из dropdown
playwright_select
  selector: "#status-filter"
  value: "NEW"

# Нажатие клавиш
playwright_press_key
  key: "Enter"

# Загрузка файлов (если форма требует)
playwright_upload_file
  selector: "input[type='file']"
  filePath: "/path/to/test-file.pdf"
```

## 5. Правила работы со скриншотами

### Расположение

```
.tl/qa-screenshots/
  UC001/
    step-01-navigate-to-form.png
    step-02-select-client.png
    step-03-add-item.png
    step-04-order-created.png
    step-05-validation-error.png
  UC002/
    step-01-...
```

### Именование

Формат: `step-NN-description.png`

| Компонент | Формат | Пример |
|-----------|--------|--------|
| Префикс | `step-` | `step-` |
| Номер | Двузначный `NN` | `01`, `02`, `10` |
| Описание | kebab-case, краткое | `navigate-to-form`, `validation-error` |
| Расширение | `.png` | `.png` |

### Правила

1. **Очистить директорию** перед каждым прогоном -- старые скриншоты удаляются
2. **Скриншот на каждом значимом шаге** -- навигация, действие, результат, ошибка
3. **Не коммитить** -- `.tl/qa-screenshots/` добавлена в `.gitignore`
4. **Скриншоты -- доказательство** -- они ссылаются из `qa-report.md` как подтверждение

### Запись в .gitignore

```
# QA screenshots (temporary, not committed)
.tl/qa-screenshots/
```

## 7. Правила вынесения вердикта

### PASS

Задача проходит QA, если **все** условия выполнены:

- Все acceptance criteria, проверяемые через UI, имеют статус PASS
- Не обнаружено критических багов (CRITICAL)
- Не обнаружено major-багов, блокирующих основной поток (MAJOR в main flow)

### FAIL

Задача не проходит QA, если **хотя бы одно** условие выполнено:

- Хотя бы один acceptance criterion не пройден (кроме N/A)
- Обнаружен критический баг (CRITICAL)
- Основной поток (main flow) не выполняется до конца
- Страница не загружается или возвращает 500-ошибку

### Severity багов

| Severity | Описание | Влияние на вердикт |
|----------|----------|-------------------|
| **CRITICAL** | Приложение крэшится, данные теряются, security-уязвимость | FAIL -- блокирует |
| **MAJOR** | Основная функция не работает, но приложение не крэшится | FAIL -- если в main flow |
| **MINOR** | Косметические дефекты, неудобство, не влияет на функцию | PASS -- фиксируется как замечание |

### Критерии N/A

Некоторые acceptance criteria нельзя проверить через UI:

- Транзакционность базы данных (AC09) -- проверяется в unit-тестах
- Покрытие кода (AC12) -- проверяется в nacl-tl-review
- Производительность SQL-запросов -- проверяется нагрузочными тестами

Такие критерии помечаются как `N/A` и не влияют на вердикт QA.

## 8. Повторный запуск (Re-run)

### Когда нужен повторный запуск

- Вердикт FAIL -- разработчик исправил баги и запросил перепроверку
- Acceptance criterion не пройден -- реализация доработана
- Среда была нестабильна -- необходимо убедиться в корректности результатов

### Процедура повторного запуска

1. **Очистить скриншоты** -- удалить `.tl/qa-screenshots/UC###/`
2. **Повторить только неуспешные шаги** -- не нужно перепроверять все шаги, достаточно:
   - Шаги, отмеченные FAIL
   - Шаги, связанные с исправленными багами
   - Step 01 (навигация) -- для подтверждения работоспособности
3. **Обновить qa-report.md** -- не создавать новый файл, а обновить существующий:
   - Изменить `qa_date` на текущую дату
   - Обновить статусы шагов
   - Обновить секцию багов (помечить исправленные)
   - Обновить вердикт
   - Добавить секцию "Re-run History"

### Секция Re-run History

```markdown
## Re-run History

### Re-run #1 -- 2025-01-31

**Причина**: Исправлен BUG-001 (валидационная ошибка не отображалась)
**Проверено**: Steps 05, 06, 07
**Результат**: BUG-001 исправлен, все шаги PASS
**Новый вердикт**: PASS
```

### Правила обновления

- Первоначальный `qa-report.md` не удаляется -- он дополняется
- История re-run сохраняется для аудита
- Скриншоты при re-run перезаписывают старые (одинаковые имена)
- Если при re-run найден новый баг -- он добавляется в секцию багов с новым ID

## Чек-лист QA-сессии

### Перед тестированием

- [ ] Прочитан `acceptance.md` -- понятно, что проверять
- [ ] Прочитан `task.md` -- понятен сценарий
- [ ] Dev-серверы запущены (BE + FE)
- [ ] Директория `.tl/qa-screenshots/UC###/` очищена
- [ ] `.tl/qa-screenshots/` в `.gitignore`

### Во время тестирования

- [ ] Каждый шаг сценария выполнен
- [ ] На каждом шаге сделан скриншот
- [ ] Альтернативные потоки проверены
- [ ] Валидационные ошибки проверены
- [ ] Результат каждого шага зафиксирован

### После тестирования

- [ ] `qa-report.md` создан в `.tl/tasks/UC###/`
- [ ] Все acceptance criteria отмечены (PASS/FAIL/N/A)
- [ ] Баги описаны со скриншотами
- [ ] Вердикт вынесен (PASS/FAIL)
- [ ] Рекомендация указана
