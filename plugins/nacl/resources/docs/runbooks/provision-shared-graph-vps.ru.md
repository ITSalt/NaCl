# Runbook: развёртывание графа проекта на общем VPS

**Для кого.** Для владельца сервера, который разворачивает контейнер Neo4j 5 Community отдельного
проекта на доступном VPS для распределённой команды. Проект разворачивается один раз. Доступ
разработчика выдаётся на весь сервер, а не отдельно на проект.

**Результат.** Проект получает собственный контейнер Neo4j, постоянные volumes и mTLS-gateway.
Приватный CA выпускает персональные сертификаты разработчиков — отзываемые «API-ключи». Серверный
inventory регистрирует все gateway проектов и проецирует в каждый один набор server-wide grants.

**Обязательные ограничения.** Neo4j не публикуется напрямую на host; наружу открыт только gateway
каждого проекта. Приватный ключ CA не покидает VPS/admin-host. Секреты генерируются и не коммитятся.
Текущая граница авторизации — сервер: разработчик с доступом к серверу получает доступ ко всем
зарегистрированным gateway проектов на нём.

## Предварительные требования

- VPS с DNS-именем, например `graph.example.com`, Docker и Docker Compose, `node`, `openssl` и,
  опционально, `ufw`.
- Checkout NaCl на VPS со скриптами развёртывания, шаблонами и схемой.
- SSH-доступ администратора сервера с возможностью выполнять `sudo`.

## Развёртывание контейнера проекта

1. Запустите provisioner на VPS или через SSH:

   ```sh
   sh <NaCl>/graph-infra/vps/provision-vps.sh \
     --skills-dir <NaCl> --host graph.example.com \
     --project-scope acme-billing --prefix acme-billing \
     --gateway-port 7687 --first-developer alice@example.com \
     --state-dir /etc/nacl-graph
   ```

   `--host` также становится идентификатором сервера в access controller. Provisioner резервирует
   scope и gateway-порт проекта, создаёт отдельное хранилище Neo4j, загружает схему, регистрирует
   gateway, выдаёт первому разработчику доступ на уровне сервера и согласует все зарегистрированные
   gateway. Итог: `NACL_VPS_RESULT: status=READY|FAILED`.

2. При `FAILED` проверьте `failed_check`. Для `need-docker`, `need-compose`, `need-node` или
   `need-openssl` установите зависимость. При `container-health` проверьте
   `docker logs <prefix>-neo4j`. Загрузка схемы идемпотентна и допускает повтор. Ошибка авторизации
   или reconcile gateway не является частичным успехом: затронутые gateway помещаются в карантин,
   а операция остаётся заблокированной.

3. Передайте bundle первого разработчика по защищённому каналу. Файлы находятся в
   `/etc/nacl-graph/clients/alice@example.com/`: `client.crt`, `client.key` и `ca.crt`.

4. Коммитьте только несекретные метаданные маршрута проекта, описанные в
   [конфигурации](../configuration.ru.md#graph-обязательно-для-graph-aware-скиллов). Никогда не
   коммитьте сырой пароль Neo4j, приватный клиентский ключ или приватный ключ CA.

## Выдача и отзыв server-wide доступа

Используйте тот же `--server-id`, который был зарегистрирован через `provision-vps.sh --host`.
Выпуск или отзыв сертификата изменяет server-wide grant trusted CN и перед успешным завершением
согласует все зарегистрированные gateway проектов.

```sh
# Выпустить персональный сертификат и дать разработчику доступ ко всем gateway сервера.
sh <NaCl>/graph-infra/vps/issue-client-cert.sh bob@example.com \
  --state-dir /etc/nacl-graph --server-id graph.example.com \
  --host graph.example.com --gateway-port 7687

# Отозвать сертификат и удалить разработчика из всех gateway сервера.
sh <NaCl>/graph-infra/vps/revoke-client-cert.sh bob@example.com \
  --state-dir /etc/nacl-graph --server-id graph.example.com
```

`issue-client-cert.sh` успешен только после обновления проекций всех зарегистрированных gateway.
`revoke-client-cert.sh` требует `--server-id`: он обновляет audit-запись CA, удаляет CN из серверного
grant и согласует все зарегистрированные gateway. Если проекцию или reload нельзя проверить,
команда возвращает `BLOCKED` и физически помещает неопределённые gateway в карантин вместо ложного
успеха. Эти операции не меняют пароль Neo4j и не перезапускают Neo4j.

## Критерии готовности

- [ ] Для развёртывания проекта получен `NACL_VPS_RESULT: status=READY`.
- [ ] В server inventory есть gateway проекта с ожидаемым `server_id`.
- [ ] Bolt/HTTP-порты Neo4j недоступны извне; опубликованы только mTLS gateway-порты.
- [ ] Bundle первого разработчика безопасно передан и подключается через sidecar.
- [ ] Тестовый выпуск или отзыв согласует все зарегистрированные gateway и возвращает проверенный
      терминальный статус, а не частичный успех.
- [ ] В репозитории находятся только несекретные метаданные маршрута и непрозрачная ссылка на секрет.
