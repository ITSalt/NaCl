# Апгрейд графа проекта до структур 2.15 (Фазы 0–4): инструкция для агента-оркестратора

> **Классификация: исторический pre-2.15 Claude/manual/internal runbook.** Это не
> текущий onboarding Codex-плагина и не доказательство того, что плагин умеет
> выполнить описанный ниже domain graph upgrade. Обычный пользователь Codex
> начинает с `docs/setup/install-codex-plugin.ru.md`, проходит
> `docs/quickstart.ru.md` и запрашивает поддерживаемый migration plan
> через публичный `$nacl-migrate`. Текущий плагин принимает только allow-listed
> named gateway tools и не исполняет произвольный Cypher. Его migration route
> ограничен фиксированным schema-recovery corridor и подтверждённой file-only
> BA/SA-конвертацией; отсутствующая domain migration закрывается статусом
> `BLOCKED/DOMAIN_MIGRATION_RESOURCE_UNAVAILABLE`.

Все slash-команды вида `/nacl-*` ниже — интерфейсы исторических
Claude/manual/internal leaf-скиллов. Они не являются дополнительными публичными
скиллами Codex и не разрешают обходить packaged gateway contract. APOC, прямой
доступ к контейнеру и произвольные graph statements в этой инструкции относятся
только к совместимому legacy-runtime соответствующей версии.

**Кому предназначено.** Эту инструкцию пользователь NaCl отдаёт агенту с чистым
контекстным окном в проекте, чей граф создан ДО релиза 2.15 — то есть без слоёв
change-propagation/provenance (Decision, staleness), машин состояний экранов,
behavior slices, таксономии доменных ошибок и политик кэша/деградации.

**Что получится в конце.** Граф проекта дополнен теми из новых слоёв, которые
пользователь принял; `/nacl-sa-validate` проходит без CRITICAL; код проекта
сверен с новой спецификацией, расхождения оформлены отчётом с маршрутизацией
(что чинить в коде, что — в графе).

> **Смежный апгрейд (привязка требований).** Если на графе сработал новый
> CRITICAL L3.7 — `Requirement` без ребра `REALIZED_BY` к реализующему его
> шагу/полю/форме — это отдельная самодостаточная инструкция:
> [`docs/runbooks/requirement-anchoring-upgrade.md`](requirement-anchoring-upgrade.md).
> Её можно запускать независимо от слоёв 2.15.

## 0. Твоя роль: оркестратор, не исполнитель

Ты управляешь процессом и принимаешь решения. Свой контекст береги для анализа,
плана и контроля — всю тяжёлую работу делают сабагенты:

- **чтение кода, графа, длинных документов** — только через сабагентов,
  возвращающих структурированную выжимку (counts, списки id, найденные факты),
  а не сырые дампы;
- **прогон скиллов по UC** — каждый прогон в отдельном сабагенте; тебе
  возвращается итог: созданные/изменённые узлы поштучно, варнинги, вопросы;
- **сверка код↔спецификация** — параллельные сабагенты по слоям.

Правила, которые нельзя нарушать:

1. **Не пиши в граф мимо скиллов.** Скиллы несут идемпотентность, направленные
   staleness-штампы и Decision-provenance; ручной Cypher-write всё это обходит
   и создаёт дрейф, ради устранения которого затевался апгрейд.
2. **verify-before-bulk.** Каждый класс правок сначала на ОДНОМ UC; убедился,
   что валидатор чист и результат осмыслен, — масштабируй.
3. **Бэкап до первой записи.** APOC-экспорт графа (или снапшот контейнера) —
   обязательный шаг 1; точку отката укажи в финальном отчёте.
4. **Git-дисциплина проекта** — по его CLAUDE.md. Ничего не пушь без явного
   запроса пользователя.

## 1. Шаг 0 — настройки проекта и доступ

Сабагентом-разведчиком собери (и держи при себе как «паспорт проекта»):

- `CLAUDE.md` проекта: правила, особые ограничения, git-политика.
- Подключение к графу для этого legacy-процесса: доступен ли MCP-интерфейс,
  поставлявшийся с соответствующей версией Claude/manual? Если нет — **СТОП**.
  Не используйте raw `docker exec`, `cypher-shell`, credential в argv или
  данные из project config как fallback. Codex-плагин не предоставляет
  произвольный Cypher и может использовать только allow-listed gateway tools.
- Версия NaCl-скиллов: в `nacl-sa-validate/SKILL.md` должны существовать уровни
  L8–L13, в `nacl-sa-uc` — команды `slices`, `errors`, `resilience`, в
  `nacl-sa-ui` — команда `state-machine`. Если их нет — **СТОП**: сначала
  обновить скиллы до 2.15+, апгрейд графа старыми скиллами невозможен.
- Есть ли BA-слой в графе (для финального `ba-cross`-прогона валидатора).

Сделай бэкап графа и зафиксируй его расположение.

## 2. Развилка «источник правды» — ОБЯЗАТЕЛЬНЫЙ вопрос до любой работы

Задай пользователю один вопрос и работай по выбранному режиму до конца:

> Что считать источником правды при доспецификации: **написанный код** проекта
> или **ваши ответы**?

**Режим А — код.** Поведение извлекают сабагенты-археологи из кода (экраны и их
состояния, коды ошибок в обработчиках, кэш-слои, fallback-ветки). Пользователю
задаются вопросы ТОЛЬКО при противоречиях: код↔код (две ветки делают разное),
код↔граф (спецификация утверждает иное), или когда код молчит (поведение не
реализовано вовсе). Молчание кода — это тоже факт: фиксируй гэп, не выдумывай.

**Режим Б — ответы пользователя.** Код считается возможно отстающим; вопросы
задаются в сутевых местах. Правила вопросов:

- **Спрашивай про работу функций и бизнес-процессы**, на языке пользователя:
  - «Что должен увидеть пользователь, когда платёжный провайдер недоступен?»
  - «Какие отказы при создании заказа клиент должен уметь различать?»
  - «Допустимо ли офлайн показывать вчерашний остаток квоты, или лучше ничего?»
  - «После сбоя загрузки список можно перезапросить кнопкой, или экран тупиковый?»
- **НИКОГДА не спрашивай про технику и методологию.** Тип ребра, `invalidation_kind`,
  state_kind, в какой узел что класть — это ТВОЯ работа: выводи из сутевых
  ответов и правил скиллов. Вопрос «какой тип связи мне указать?» — признак, что
  ты переложил свою работу на пользователя.
- **Батчируй.** Вопросы по слою собирай в один пакет на модуль/группу UC, не
  дёргай пользователя по одному вопросу.

В обоих режимах: ответы пользователя на сутевые вопросы **записываются в граф
скиллами** (требования, slice.then, behavior деградаций) — не остаются в чате.

## 3. Шаг 1 — аналитика текущего состояния (read-only)

Запусти сабагентов-аналитиков (параллельно, все read-only):

1. **Pre-flight и валидация:** `/nacl-sa-validate` (internal). Pre-flight Step
   0a/0b скажет, канонична ли схема и каковы counts по всем меткам, включая
   новые (Decision, Screen*, Slice, DomainError, ErrorPresentation, CachePolicy,
   DegradationRule). Полный прогон даст findings L1–L13. На старом графе новые
   уровни L10–L13 пройдут вакуумно (0 узлов = 0 findings) — это норма, opt-in.
2. **Инвентаризация по слоям** (только legacy: read-only Cypher через
   version-matched Claude/manual MCP, без shell fallback):
   - Фаза 0: есть ли `:Decision`-узлы; у скольких FeatureRequest нет
     `IMPLEMENTS→Decision` (гэп L9); есть ли `review_status`/`spec_version` поля.
   - Фаза 1: UI-UC (`has_ui` ≠ false) без `HAS_SCREEN`.
   - Фаза 2: UC без `HAS_SLICE` (и был ли проект вообще заинтересован в слайсах).
   - Фаза 3: модули без `HAS_ERROR`; эндпоинты без `MAY_RAISE`.
   - Фаза 4: модули без `HAS_CACHE`; UC без `HAS_DEGRADATION`.
   - NULL exemption-флаги (`has_ui`, `system_only`, `shared`, `internal`,
     `field_category`) — кандидат на `/nacl-sa-flags`.
3. **Карта кода** (только режим А, иначе отложи до Шага 4): экраны/состояния в
   UI-коде; коды ошибок в API-ответах; кэширующие слои и offline-ветки.

⚠️ **Критично для апгрейда:** если в проекте используется `/nacl-tl-plan`,
проверь `planned_from_version` на UC. На апгрейде существующего проекта baseline
проставляется **один раз** (см. правило Фазы 0 в `docs/graph-schema.md`), иначе
первый же `tl-plan` пометит устаревшим всё подряд.

## 4. Шаг 2 — gap-список и план (одно подтверждение пользователя)

Собери таблицу по слоям: **слой → что даёт проекту → текущее состояние →
рекомендация (внедрять / пропустить / частично)**. Помни:

- **Слои opt-in.** Проект без слайсов или без политик кэша — валиден (вакуумный
  проход валидатора). Не навязывай слой, которому в проекте нечего описывать:
  backend-сервису без UI не нужны машины экранов; приложению без офлайна и
  внешних провайдеров может быть не нужна Фаза 4.
- **Порядок принятия фиксирован зависимостями:**
  1. **Фаза 0 первой и обязательно** (provenance + staleness — фундамент, на
     который штампуют все producer'ы);
  2. машины экранов (Фаза 1) — ДО слайсов: UI-слайсу нужен якорь COVERS,
     producer остановится без машины;
  3. таксономия ошибок (Фаза 3) — ДО resilience: error-правила деградации
     требуют `ON_ERROR` на каталогизированные ошибки (chaining-правило);
  4. слайсы (Фаза 2) и resilience (Фаза 4) — поверх.
- Масштаб: UC-by-UC, группируя по модулям. В план включи поимённый список
  «UC → какие команды скиллов запускаем».

Покажи пользователю таблицу слоёв + план + оценку числа сутевых вопросов
(режим Б) и получи **одно** подтверждение. Дальше работай автономно, прерываясь
только на сутевые вопросы и противоречия.

## 5. Шаг 3 — исполнение (скиллы через сабагентов)

Порядок и инструменты:

| # | Гэп | Инструмент | Примечание |
|---|---|---|---|
| 1 | FR без Decision (L9) | runbook `nacl-tl-core/references/provenance-gap-closure.md` + `/nacl-sa-finalize` FR-backfill | честный backfill, не выдумывание rationale |
| 2 | NULL exemption-флаги | `/nacl-sa-flags` | чистит ложные срабатывания L4–L7 |
| 3 | baseline staleness | по правилу Фазы 0 | один раз, до первого tl-plan |
| 4 | UI-UC без машины | `/nacl-sa-ui state-machine UC-NNN` | по одному UC на сабагента |
| 5 | модуль без error-каталога | `/nacl-sa-uc errors UC-NNN` | мульти-модульные UC скилл сам эскалирует |
| 6 | UC без слайсов | `/nacl-sa-uc slices UC-NNN` | UI-UC — только после п.4 |
| 7 | кэш/деградация | `/nacl-sa-uc resilience UC-NNN` | только после п.5 для error-правил |

Правила исполнения:

- **Первый UC каждого класса — отдельно** (verify-before-bulk): прогнал скилл →
  scoped-прогон соответствующего L-уровня валидатора → посмотрел глазами на
  созданное → только потом ставь остальные UC в очередь сабагентам.
- После каждого модульного блока — scoped `/nacl-sa-validate` затронутых
  уровней; **0 CRITICAL прежде чем идти дальше**. WARNING фиксируй в журнал
  апгрейда (многие закрываются позже штатными потребителями, например L11.4 —
  перепланированием tl-plan).
- Скиллы сами зададут свои подтверждающие вопросы; сутевые вопросы пользователю
  (режим Б) готовь заранее пакетом на модуль — из источников скилла
  (requirements, BA-правила, RC-строки) и карты кода.
- Сабагенту-исполнителю давай: id UC, режим источника правды, релевантные
  сутевые ответы пользователя. Получай от него: дельты графа поштучно,
  вопросы/варнинги, вердикт scoped-валидации.

## 6. Шаг 4 — финальная сверка кода со спецификацией

Последний шаг — всегда, в обоих режимах:

1. Полный `/nacl-sa-validate` (internal + `ba-cross`, если есть BA-слой):
   **0 CRITICAL обязателен**, WARNING — перечислены и атрибутированы.
2. Параллельные сабагенты-сверщики код↔граф по принятым слоям:
   - **экраны:** состояния/переходы в UI-коде vs Screen-машины (нет ли в коде
     состояний, которых нет в графе, и наоборот);
   - **ошибки:** коды ошибок в API-обработчиках и клиентской обработке vs
     каталог DomainError + ErrorPresentation (необработанные каталожные ошибки
     уже подсвечивает L12.7 — сверь его вывод с кодом);
   - **кэши/деградации:** реальные кэш-слои, offline- и fallback-ветки кода vs
     CachePolicy/DegradationRule (`behavior` правила — это наблюдаемое поведение:
     его можно проверить по коду и тестам);
   - **слайсы:** существующие тесты/acceptance vs `slice.then`.
3. Каждое расхождение классифицируй и маршрутизируй:

| Класс расхождения | Режим А (код = правда) | Режим Б (пользователь = правда) |
|---|---|---|
| код делает X, спека молчит | дополнить граф (повторный прогон скилла) | спросить пользователя: X — норма или случайность? |
| спека требует X, код молчит/делает иначе | проверить: может, спека зафиксировала желаемое — спросить | задача разработки: `/nacl-tl-fix` (баг) или `/nacl-tl-plan` (доработка) |
| код противоречив сам себе | сутевой вопрос пользователю | сутевой вопрос пользователю |

Расхождения «код отстаёт» НЕ чинят правкой графа под код, если правда —
пользователь; и наоборот.

## 7. Definition of Done и отчёт пользователю

- [ ] Бэкап-точка графа указана.
- [ ] Режим источника правды зафиксирован в отчёте.
- [ ] Принятые слои: counts узлов/рёбер до → после, по слоям.
- [ ] Пропущенные слои: перечислены с причиной («проекту нечего описывать в…»).
- [ ] `/nacl-sa-validate`: 0 CRITICAL; список WARNING с владельцами/планом.
- [ ] Сверка код↔спецификация: таблица расхождений с маршрутизацией.
- [ ] Список заданных сутевых вопросов и полученных ответов (они уже в графе).
- [ ] Открытые вопросы, на которые ответа нет, — отдельным списком, не потеряны.

Отчёт — компактный, со ссылками на узлы графа по id, без сырых дампов.
