# Verification Playbook — «как я пойму, что вынесенный тул дал точный И лучший результат»

Протокол, который вы запускаете сами на **живом проекте**, чтобы для каждого вынесенного
тула получить однозначный вердикт: тул **корректен** (даёт правильный ответ) и **строго
лучше** prose-пути (точнее и/или детерминированнее). Без самообмана.

---

## 0. Главная ловушка: тул не может судить сам себя

Если «правильный ответ» = «то, что вывел тул», то «тул прав» — тавтология. Поэтому
основа протокола — **независимый оракул** (ground truth, выведенный НЕ из тула). Всё
остальное сверяется с ним.

Три способа получить оракул, по возрастанию строгости — берите доступный:

| Оракул | Как получить | Для чего годится |
|---|---|---|
| **O1. Ручной вывод из спецификации** | Возьмите ~8–12 входов, для каждого посчитайте ожидаемый выход ВРУЧНУЮ, строго по правилу из `SKILL.md` (таблица/шаги). | Детерминированные правила: slug, wave, severity-rollup, ID-формат. |
| **O2. Независимая реализация** | Второй человек (или второй агент с ДРУГИМ промптом, без вашего тула) считает ответ. Совпали → это оракул. | Когда правило сложное и ручной счёт ненадёжен. |
| **O3. Downstream-исход** | Прогнать выход через реальное использование и проверить, что он РАБОТАЕТ (см. §5). | Сильнейший: «лучше» = «не ломается на практике». |

> Правило чистоты: **оракул пишет не тот, кто писал тул** (test-author isolation — у вас
> это уже есть в `nacl-tl-regression-test`). Иначе ошибка автора протечёт в обе стороны.

---

## 1. Соберите входы — обязательно с edge-кейсами

Самая частая причина ложного вывода «тул не нужен»: все тест-входы лёгкие, на них даже
prose не ошибается. **Намеренно включите входы, где правило тонкое:**

- из **реальных данных проекта** (например, реальный UC-граф зависимостей, реальный набор
  validation-находок) — чтобы распределение было продакшеновым;
- из **границ правила**: ровно пороговое значение (5 warnings), пустой/максимальный вход
  (slug на 50 символов, имя из одной пунктуации), исключения (exempt-поля, shared-сущности),
  «ромб» зависимостей, цикл, backend-only без FE.

Минимум ~10 входов, из них ≥4 — edge.

---

## 2. Зафиксируйте оракул (до прогонов!)

Для каждого входа запишите ожидаемый выход (O1/O2) в таблицу `oracle.csv`:
`input_id, expected_output`. **Раньше**, чем запускаете модель — иначе соблазн подогнать.

Заодно это сразу проверяет сам тул: прогоните тул на этих входах. **Тул обязан == оракул
на 100%.** Если нет — баг в туле (или в тесте), чините до продолжения. Этот шаг и есть
«тул действительно точен» — независимо от модели.

```bash
# пример для wave-plan: тул против ручного оракула
node nacl-tl-plan/scripts/wave-plan.mjs '<input>'   # сверить с expected_output
node --test nacl-tl-plan/scripts/wave-plan.test.mjs # тесты = оракул, авторованный из спеки
```

---

## 3. Прогоните обе руки (как в `ab.mjs`)

На тех же входах, той(ими) моделью(ями), что реально используете (включите слабую — на
ней разрыв больше). По N≥20 на каждый (вход × рука × модель):

- **OLD (prose):** дайте правило из до-выносного `SKILL.md` + вход, попросите ТОЛЬКО решение.
- **NEW (tool):** попросите запустить скрипт и вернуть его вывод.

Харнесс `bench/skill-tools/ab/ab.mjs` это уже делает — адаптируйте блок `CASES` под свой
тул (вход, oldPrompt, newPrompt, normalize, correct) и `git show <main>:<path>/SKILL.md`
для текста OLD-руки. Метрики на ячейку:

- **accuracy** = доля прогонов, где `normalize(вывод) == оракул`;
- **distinct** = число различимых нормализованных выходов на N (1 = детерминировано);
- **deferred?** (для NEW) = в транскрипте реально есть вызов тула (а не «угадал»).

---

## 4. Вердикт — когда тул «победил»

Тул подтверждён как **точный И лучший**, если выполнены ВСЕ три:

| # | Условие | Проверка |
|---|---|---|
| **V1. Корректен** | тул == оракул на **100%** spec-входов | §2 |
| **V2. Детерминирован на деле** | NEW-рука: `distinct == 1` **и** `deferred == true` на каждом входе | §3 |
| **V3. Строго лучше prose** | на ≥1 edge-входе OLD-рука даёт `accuracy < 100%` **или** `distinct > 1` | §3 |

Тонкости честной интерпретации:
- **V3 не выполнился на каком-то туле** (prose тоже 100%/1 везде) → тул НЕ даёт прироста
  точности на вашем распределении; остаётся выигрыш в *гарантии* детерминизма и токенах.
  Это слабее — так и пишите, не натягивайте «точнее».
- Разрыв обычно растёт на **слабых моделях** и **алгоритмических/граничных** правилах,
  и почти исчезает на лёгких входах с сильной моделью. Если хотите громкий результат —
  edge-кейсы и слабая модель; если консервативный — сильная модель.
- **Контринтуитив допустим:** сильная модель может быть вручную ХУЖЕ слабой на конкретной
  рубрике (у нас Opus 40% vs Haiku 100% на classify-findings). Это не шум — это аргумент
  «нельзя предсказать, что сломается», т.е. за детерминизм.

---

## 5. Downstream-проверки (оракул O3 — сильнейшее «реально лучше»)

Где можно — проверьте, что выход тула РАБОТАЕТ, а prose-выход ломается. Это снимает любые
споры про «а правда ли тул прав»:

| Тул | Downstream-проверка |
|---|---|
| **wave-plan** | Cypher по плану: нет задачи раньше своей зависимости (`from.wave > to.wave` на всех DEPENDS_ON), все TECH в волне 0, BE<FE. Prose-план, нарушивший зависимость → конкретный дефект, который тул предотвращает. |
| **slug / branch-guard** | `git check-ref-format --branch "$slug"` = 0; guard реально блокирует коммит в base (проверить exit-код на сценарии «на base + feature-branch»). |
| **classify-findings** | Сверить overall-вердикт с РУЧНЫМ аудитом тех же находок: посчитать non-exempt CRITICAL и WARNING руками; exempt-фильтры применены? |
| **nacl-ids** | Сгенерированный ID уникален и парсится обратно (`MATCH {id}` не коллизит; формат проходит регэксп схемы). |

---

## 6. Защита от ложного «выигрыша» (чек-лист честности)

- [ ] Оракул выведен из **спеки вручную**, не скопирован из вывода тула (иначе круговая логика).
- [ ] Оракул и тул писали **разные авторы**.
- [ ] Во входах есть **edge-кейсы** (иначе prose не ошибётся и вы зря решите «тул не нужен»).
- [ ] Часть входов — **реальные данные проекта**, не только синтетика.
- [ ] Для NEW-руки подтверждено, что модель **реально запускала тул** (транскрипт), а не угадала.
- [ ] Baseline (prose) зафиксирован **до** правок, метрики посчитаны на одинаковых входах.
- [ ] N достаточно (≥20), результат описан как индикативный, без поддельной p-value-строгости.

---

## TL;DR

1. Напиши оракул вручную из `SKILL.md` (с edge-кейсами и реальными данными) — **до** прогонов.
2. Тул обязан совпасть с оракулом на 100% (это «точен»).
3. Прогони prose-руку N≥20 — измерь accuracy и distinct против оракула.
4. **Победа = V1 (тул==оракул 100%) ∧ V2 (NEW distinct=1 & делегировал) ∧ V3 (prose где-то <100% или distinct>1).**
5. Где можешь — добей downstream-проверкой (плана/слага/вердикта), что prose-выход реально ломается, а тул — нет.
