# Бенчмарк Фазы 2: Behavior Slice в графе NaCl

**Дата:** 2026-06-05 · **Статус:** ПОДТВЕРЖДЕНО · **Цель:** `family-cinema` (граф 1913 узлов / 2693 рёбер), изолированный клон.

> ℹ️ **О приватности.** `family-cinema` — собственный демо-проект автора; имя, код, UC-id
> называются открыто. Абсолютные локальные пути заменены на плейсхолдеры. Клиентских данных нет.

---

## 0. Что проверяли

Фаза 2 добавила в SA-слой **Behavior Slice** — graph-native сценарий приёмки (Given/When/Then),
вертикальный срез наблюдаемого поведения UC: слой ниже UC (у UC несколько слайсов), выше Task
(задачи остаются per-UC — слайсы это **overlay**, per-Slice задачи сознательно out of scope).
Узел `Slice` + рёбра `HAS_SLICE` (required parent) / `COVERS → ScreenState|Transition` (UI-якорь
в хаб Фазы 1) / `CALLS → APIEndpoint` (backend-якорь; имя намеренно разделено с
`ScreenEffect→APIEndpoint`) / `VERIFIED_BY → Task` (TL-overlay). Производитель —
`nacl-sa-uc slices`, потребитель — `nacl-tl-plan` (секция в task-файлы + перепривязка
VERIFIED_BY), валидатор — **новый уровень L11** в `nacl-sa-validate` (L8/L9/L10 заняты).

Бенчмарк, как и в Фазах 0–1, меряет **сценарий изменения**: дотягивается ли «потянул за нитку»
до нового слоя, остаётся ли гейт-штамп точечным, ловят ли валидаторы реальные поломки — и,
специфично для Фазы 2, **не стреляет ли namespace-шеринг `CALLS` в обе стороны**.

### Гипотезы (фальсифицируемые)

- **H0 (vacuous pass, дважды).** (a) На графе без единого Slice-узла все L11-чеки дают ноль;
  (b) на графе с **31 машиной экранов и всё ещё нулём слайсов** — тоже ноль: overlay строго
  opt-in, внедрение машин само по себе L11 не включает.
- **H1 (достижимость, по классам якорей).** По **старому** (до Фазы 2) allow-list
  `sa_impact_closure` COVERS-only слайс от изменённого UC **недостижим**; CALLS-якорные слайсы
  «случайно» достижимы и по старому (CALLS зарегистрирован ещё Фазой 1) — меряем честно оба
  класса. По **новому** списку изменение достигает **все** слайсы от UC (1 hop), от
  DomainAttribute, от APIEndpoint (1 hop) и от Task (1 hop через VERIFIED_BY).
- **H2 (точечный штамп).** После authoring слайсов направленный штамп (контракт 3g) помечает
  ровно задачи UC + задачи транзитивных `DEPENDS_ON*1..5`-зависимых + сам UC; широкий
  undirected-обход по расширенному Фазой 2 набору рёбер пометил бы кратно больше.
- **H3 (детекторы детектируют).** Каждый из 14 классов инжектированных дефектов поджигает ровно
  свои чеки по **полной матрице L10+L11 (21 чек)**, без cross-talk — в частности, мусорное
  `(sl:Slice)-[:CALLS]->` не поджигает ScreenEffect-чеки (L10.2/L10.7a) и наоборот — это
  гарантия label-qualified namespace-шеринга.

## 1. Установка (воспроизводимо)

Клон тот же, что в Фазах 0–1: изолированный `nacl-bench-neo4j` (bolt 7798, neo4j:5-community +
APOC), потоковый APOC-экспорт рабочего fc-графа через HTTP → `cypher-shell -f` (команды — §1
phase0-отчёта). Применены constraints Фазы 2 (`constraint_slice_id` + индексы из
`sa-schema.cypher`).

**Выборка:** все **31 UC** клона с `USES_FORM` (не backend-only) — не cherry-picked. Каждому
программно создаётся каноническая машина Фазы 1 (4 состояния / 3 события / 4 перехода /
1 load-эффект) и **три канонических слайса ровно тем Cypher, что прописан в
`nacl-sa-uc slices`**: `HappyPath` (happy; COVERS Loading+Loaded+переход; CALLS эндпоинт),
`EmptyResult` (alternate; **COVERS-only** — контрольный класс для H1), `LoadFailureRetry`
(error; COVERS Error+2 перехода; CALLS). `VERIFIED_BY` — по default-правилу «все
`GENERATES`-задачи UC» (task-id клона неканоничны: `TASK-FR029-W4-*`/`TECH-*`, мульти-owner).
**Полный сброс** между каждым измерением.

## 2. Результаты

### H0 — vacuous pass ✓ (дважды)

На чистом клоне (0 Slice) все 10 L11-чеков = 0. После авторинга **31 машины без единого
слайса** — снова все 10 = 0. Проекты, внедрившие машины, но не слайсы, от L11 не страдают.

### H1 — достижимость изменения ✓ (31/31)

| Откуда тянем | Старый allow-list | Новый allow-list |
|---|---|---|
| UC (31 проб) | **2/3** слайсов (только CALLS-якорные) | **3/3** (HappyPath в 1 hop) |
| UC → COVERS-only слайс (31 проб) | **недостижим** (31/31) | достижим (31/31) |
| APIEndpoint (31 проб) | 2/3 | **3/3** (CALLS-якорные в 1 hop) |
| DomainAttribute (18 проб*) | 2/3 | **3/3** |
| Task (18 проб**) | 2/3 | **3/3** (1 hop через VERIFIED_BY) |

\* у 13 из 31 форм нет ни одного `MAPS_TO`-атрибута — проба честно пропущена (как в Фазе 1).
\** задачи есть у 18 из 31 UI-UC (ещё 3 task-bearing UC — backend-only).

Два честных вывода: (1) регистрация `HAS_SLICE|COVERS|VERIFIED_BY` в allow-list **обязательна** —
без неё чисто-UI-поведение (COVERS-only слайс) структурно невидимо для импакт-анализа;
(2) переиспользование имени `CALLS` дало CALLS-якорным слайсам достижимость «бесплатно» ещё до
регистрации — приятный, но недостаточный побочный эффект namespace-шеринга.

### H2 — штамп остаётся точечным ✓ (31/31)

| Метрика | Значение |
|---|---|
| Помечено точечным штампом, всего (31 UC) | **58** узлов (27 задач + 31 UC) |
| Широкий undirected-радиус от UC (Task-узлы), всего | **1391** |
| Широкий радиус от одного слайса, всего | **1326** |
| Среднее на UC: точечный / широкий | **0.87 / 44.9 задачи** (~**51×**) |

Тренд подтверждается третий раз и ухудшается монотонно: Фаза 0 — ~20×, Фаза 1 — ~49×, Фаза 2 —
**~51×** ложного радиуса у широкого обхода. Каждый новый тип рёбер делает broad-штамп опаснее;
правило «широкий обход — только разведка/отображение, гейт-штамп — только направленный 3g»
обязательно навсегда.

### H3 — детекторы детектируют ✓ (14/14, cross-talk 0 по матрице из 21 чека)

| Инжектированный дефект | Ожидаемые чеки | Сработало |
|---|---|---|
| Осиротевший слайс | L11.0 + L11.1 + L11.2 | ✓ ровно они |
| Слайс без родителя (с якорем) | L11.1 | ✓ |
| Слайс без якорей (COVERS снят) | L11.2 + L11.7 (×2: открылся coverage-gap) | ✓ |
| COVERS в чужой UC | L11.3 | ✓ |
| COVERS в не-ScreenState/Transition | L11.3 | ✓ |
| VERIFIED_BY снят на planned UC | L11.4 | ✓ |
| VERIFIED_BY в задачу чужого UC | L11.5 | ✓ |
| VERIFIED_BY в не-Task | L11.5 | ✓ |
| **Slice-CALLS в не-APIEndpoint** | **L11.5; L10.2/L10.7a молчат** | ✓ шеринг не стреляет |
| Пустой `then` | L11.6a | ✓ |
| `slice_kind` вне словаря | L11.6b + L11.8 (happy перестал быть happy) | ✓ |
| Раскрыт coverage-gap (COVERS Loading снят) | L11.7 | ✓ |
| Не осталось happy-слайса | L11.8 | ✓ |
| **Реверс: ScreenEffect-CALLS в Form** | **L10.7a + L10.2; все L11 молчат** | ✓ шеринг не стреляет |

На целых слайсах перед каждой инжекцией — все 21 чек = 0 (ложных срабатываний нет).

## 3. Что бенчмарк нашёл и что мы починили

1. **Одно неверное ожидание в первом прогоне H3** (дефект модели автора, не детектора — тот же
   класс, что два неверных ожидания Фазы 1): `slice_kind='smoke'` у happy-слайса корректно
   поджигает **оба** чека — L11.6b (мусорный словарь) и L11.8 (UC лишился единственного
   happy-слайса). Ожидание исправлено, повторный прогон — чистый.
2. **Канон data-loading-разложения уточнён до бенчмарка** (на демо UC-006): первая редакция
   не покрывала `Loading`-состояние ни одним слайсом → канонические 3 слайса оставляли бы
   L11.7-gap на каждом экране. Теперь `HappyPath` покрывает и `Loading` — 3 канонических
   слайса дают полное покрытие 4-состоянийной машины.

## 3-bis. Skill-level подтверждение (независимые агенты на клоне)

Поверх графового бенчмарка — два **независимых агента** (test-author isolation: исполнитель
читает только skill-текст, верификатор не видел вывода исполнителя), через
`docker exec cypher-shell` (в сессии нет MCP-neo4j, `itsalt-pinch` не установлен — sanctioned
Agent-инструмент, как в Фазах 0–1):

- **Исполнитель** прочитал обновлённый `nacl-sa-uc` и исполнил `slices` для **двух** UC:
  `UC-014` «Проведение AI-интервью» (UI-кейс: машина существует, 3 задачи, неканоничные id) и
  `UC-033` «Обработка и монтаж видеоролика» (**backend-only**: `has_ui=false`, 5 задач,
  ни одного эндпоинта → provisional-путь). Результат: 3 слайса UC-014 (полное покрытие машины
  8/8) + 5 CALLS-only слайсов UC-033 из Requirements; 2 provisional-эндпоинта (run + status,
  заякорены `EXPOSES`); 34 VERIFIED_BY по default-правилу; направленные штампы 3+1 и 5+1;
  scoped-L11 PASS дважды; **0 ошибок cypher-shell** — все шаблоны исполнились дословно.
- **Верификатор** независимо считал граф и сверил с контрактом: вердикт **PASS, 0
  CRITICAL/WARNING/INFO**. Особо: общий stale-набор графа — **ровно 10 узлов** (2 UC + 8 их
  задач), без фан-аута; cross-talk CALLS подтверждён на живых данных (1 ScreenEffect-CALLS и
  9 Slice-CALLS в одном графе, строгое разделение по label).

**Прогон поймал 5 реальных дефектов skill-текста + 1 дефект запроса — все исправлены:**

| # | Дефект (нашли независимые агенты) | Фикс |
|---|---|---|
| 1 | Backend-UC без RuntimeContract и с placeholder-шагами: источник слайсов не определён; queue-UC без RC не сигналится | явный приоритет источников для backend-only (RC → Requirements → Steps) + warning-инструкция «рекомендуй detail Phase 4.5, контракт не выдумывай» |
| 2 | Placeholder-ActivitySteps (`"--"`, без order) не оговорены — скилл оговаривал только пустые acceptance_criteria | явная оговорка «считать отсутствующими» |
| 3 | Гранулярность provisional-эндпоинтов для backend-UC не определена (пример в 3.3 — только GET-чтение) | правило «эндпоинт = отдельная backend-операция (trigger vs status), слайсы шарят эндпоинты, никогда 1:1» |
| 4 | `slice_kind` для recovery/degrade/idempotency-сценариев не маппится (таблицы кинды дают только для UI-архетипов) | гайд: recovery/degradation → alternate, наблюдаемый отказ → error, идемпотентность/гонки/границы → edge |
| 5 | Отчётная строка VERIFIED_BY не масштабируется (5 задач × 5 слайсов = 25 рёбер списком) | агрегированный формат `{N slices} × {M tasks}` |
| 6 | `sa_uc_slices`: map-collect над несматченным OPTIONAL даёт `covers=[{NULL,NULL}]` вместо `[]` у CALLS-only слайсов | null-фильтр в comprehension (фикс в sa-queries + копии запроса в tl-plan) |

Зафиксированное наблюдение без правки: L11 не проверяет **формат id** слайса (нелатинский
PascalName прошёл бы все чеки) — сознательно симметрично Фазе 1 (id-дисциплина SCR-семьи тоже
держится на producer, отдельного L10-чека формата нет).

## 4. Демо на одном реальном UC (verify-before-bulk)

До массового прогона слайсы были вручную созданы для **UC-006 «Просмотр результата»**
(машина `SCR-ResultViewer` из демо Фазы 1, задача `TASK-FR029-W4-T3-result-viewer`): 3
канонических слайса → все 10 L11-чеков = 0; closure достигает слайс от UC (1 hop), от атрибута
(4 hops — короче ожидаемых 5: путь идёт через `USES_FORM`, мост `RENDERS` не нужен слайсам),
от эндпоинта (1 hop), от задачи (1 hop); COVERS-only слайс по старому списку недостижим;
штамп точечный (1 задача + 1 UC); клон после сброса — байт-в-байт 1913/2693.

## 5. Граница честности

- **CALLS-якорные слайсы достижимы и по старому allow-list** (CALLS зарегистрирован Фазой 1) —
  H1-различие старый/новый доказано на COVERS-only классе, а не на всех слайсах разом. Это
  отражено в таблице, а не спрятано в агрегате.
- `VERIFIED_BY` в H1-пробах «от Task» существует только у 18 UI-UC с задачами; для остальных
  проба честно null (как DA-пробы Фазы 1).
- Канонические слайсы бенчмарка — data-loading-архетип; backend-only путь (CALLS-only,
  provisional-эндпоинты) покрыт skill-level прогоном (UC-033), не графовым харнессом.
- Цифры специфичны графу family-cinema (особенно «broad»). Харнесс воспроизводим на любом графе.
