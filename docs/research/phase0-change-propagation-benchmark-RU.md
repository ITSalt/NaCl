# Бенчмарк Фазы 0: распространение изменений и provenance в графе NaCl

**Дата:** 2026-06-05 · **Статус:** ПОДТВЕРЖДЕНО · **Цель:** `family-cinema` (граф 1913 узлов / 2693 рёбер), изолированный клон.

> ℹ️ **О приватности.** `family-cinema` — собственный демо-проект автора; имя, код, UC/FR/REQ-id
> называются открыто. Абсолютные локальные пути в командах ниже заменены на плейсхолдеры
> (`<repo>`, `<tmp>`) — это шум, не приватность. Клиентских данных нет.

---

## 0. Что проверяли

Фаза 0 добавила в NaCl механизм «потянул за нитку → видно всё, что надо поправить» и graph-native
ответ «почему так решили год назад»:

- **staleness-флаг** (`review_status`/`stale_origin`/`stale_since`), который write-скиллы ставят на
  зависимые артефакты при изменении upstream-узла; читается через `coalesce(...,'current')`;
- **L8 (Staleness Closure)** в `nacl-sa-validate` — закрытие (release/conductor) блокируется, пока
  есть `stale`-узлы;
- **идемпотентный `nacl-tl-plan`** — `spec_version` vs `planned_from_version`, перегенерирует только
  изменившееся, при этом снимает флаг;
- **`:Decision`** + `JUSTIFIES`/`SUPERSEDES`/`FeatureRequest-[:IMPLEMENTS]` — provenance в графе.

Бенчмарк измеряет **сценарий изменения** (не greenfield), потому что именно он был главным риском:
не растеряем ли мы зависимые артефакты, меняя UC.

### Гипотезы

- **H1.** *До* Фазы 0 изменение UC оставляет его зависимые task-снапшоты молча устаревшими, а
  релизный гейт проходит «вслепую», не замечая дрейфа.
- **H2.** *После* Фазы 0 то же изменение помечает **все** зависимые, гейт **блокируется**, а
  перепланирование очищает до нуля — молчаливого дрейфа нет.
- **H3.** Штамп **точечный**: изменение одного UC помечает только настоящих downstream, а не пол-проекта
  (широкий `sa_impact_closure` — только для разведки/отображения, не для гейта).

## 1. Установка (воспроизводимо)

Клон делается **без остановки и без мутации** рабочего графа family-cinema — потоковый экспорт APOC
через HTTP в свежий изолированный контейнер:

```bash
# 1) изолированный scratch-Neo4j (bolt 7798), APOC, отдельный контейнер
docker run -d --name nacl-bench-neo4j -p 7798:7687 -p 7799:7474 \
  -e NEO4J_AUTH=neo4j/neo4j_graph_dev -e NEO4J_PLUGINS='["apoc"]' neo4j:5-community

# 2) экспорт fc-графа потоком (read-only для fc), jq собирает чистый cypher-shell скрипт
curl -s -u neo4j:<pw> -H 'Content-Type: application/json' \
  http://localhost:<fc-http>/db/neo4j/tx/commit \
  -d '{"statements":[{"statement":"CALL apoc.export.cypher.all(null,{stream:true,format:\"cypher-shell\",useOptimizations:{type:\"unwind_batch\",unwindBatchSize:200}}) YIELD cypherStatements RETURN cypherStatements"}]}' \
  | jq -r '.results[0].data[].row[0]' > <tmp>/fc-clone.cypher

# 3) загрузка в клон
docker cp <tmp>/fc-clone.cypher nacl-bench-neo4j:/tmp/fc-clone.cypher
docker exec nacl-bench-neo4j cypher-shell -u neo4j -p <pw> -f /tmp/fc-clone.cypher
```

Паритет клона подтверждён: **1913 узлов, 2693 ребра, 31 Decision, 37 UseCase** — байт-в-байт исходный
граф (включая 31 `:Decision`, закрытые на шаге gap-closure до клонирования).

Перед прогоном применён **один baseline gap-closure**: `planned_from_version = coalesce(uc.spec_version,0)`
на 37 задачах (иначе первый инкрементальный `tl-plan` пометил бы всё как дрейф — см. §4).

## 2. Метод

Выборка — **все 21 UC, у которых есть сгенерированные задачи** (не отобранные вручную). Каждый UC
проходит два «рукава» с полным сбросом графа к baseline между любыми измерениями (рукава и итерации
независимы):

- **BASELINE (имитация до-Фазы-0):** меняем UC (`spec_version → 1`), штамп **не** ставим. Считаем,
  сколько его задач теперь устарели по версии, и есть ли хоть какой сигнал (его нет — механизма не
  было). Вердикт гейта = `PASS(blind)`.
- **TREATMENT (Фаза 0):** то же изменение **+ точечный штамп** (как в `nacl-sa-feature` шаг 3g после
  фикса). Считаем помеченные задачи и вердикт L8 (`BLOCKED`), затем имитируем инкрементальный
  `nacl-tl-plan` (снять флаг, `planned_from_version = spec_version`) и перечитываем L8 (`0`).
- **H3:** дополнительно считаем, сколько задач пометил бы **старый широкий** undirected-обход (для
  сравнения «радиуса взрыва» точечного штампа против широкого).

Гейт = `MATCH (n) WHERE coalesce(n.review_status,'current')='stale' RETURN count(n)` (это и есть L8.1).

## 3. Результаты

**Агрегат по 21 UC:**

| Метрика | Значение |
|---|---|
| Молчаливо устаревших задач, BASELINE | **37** |
| Релиз прошёл «вслепую» при наличии дрейфа, BASELINE | **21 / 21** |
| Помечено задач, TREATMENT | **37** (ровно те же 37) |
| Гейт заблокировался, TREATMENT | **21 / 21** |
| Очищено после перепланирования, TREATMENT | **21 / 21** |
| Молчаливо устаревших задач, TREATMENT | **0** |
| Радиус штампа: точечный (всего) | **37** |
| Радиус штампа: широкий обход (всего) | **760** |

**H1 ✓** — до Фазы 0: 37 снапшотов молча устарели, гейт прошёл вслепую на всех 21 изменениях.
**H2 ✓** — после Фазы 0: помечены ровно 37, гейт заблокирован 21/21, очищен 21/21, молчаливого дрейфа 0.
**H3 ✓** — точечный штамп 37 против широкого 760 → **в ~20.5× меньше** ложного радиуса (в среднем
1.76 задачи на изменение против 36.2). На графе family-cinema широкий обход через общий `ACTOR`
(одна роль → каждый UC с этой ролью) и общие `Requirement` пометил бы почти весь admin-кластер при
правке одного UC — то есть блокировал бы релизы по ложной устарелости.

**Фрагмент по-UC** (полная таблица — в `<tmp>/bench_results.json`):

```

UC-033   5     5            PASS(blind)   5        BLOCKED→PASS(clean)  5      39
UC-032   4     4            PASS(blind)   4        BLOCKED→PASS(clean)  4      39
UC-004   3     3            PASS(blind)   3        BLOCKED→PASS(clean)  3      39
```

Точечный штамп всегда равен числу реально зависимых задач; широкий — почти константа ~36
(весь связный кластер), что и есть проблема, ради которой штамп сделан направленным.

## 4. Что бенчмарк нашёл и что мы починили

Бенчмарк — не только подтверждение, но и детектор дефектов. Два реальных, исправленных по ходу:

1. **Первый прогон на проекте после апгрейда помечал ВСЁ.** Без `planned_from_version` у задач и
   `spec_version` у UC условие `0 > -1` истинно для каждой пары → первый `tl-plan` пере-генерировал бы
   все задачи и сбросил бы in-progress. **Фикс:** обязательный one-time baseline в gap-closure +
   guard `planned_from_version IS NOT NULL` в детекторе (реальное изменение на не-baseline-проекте всё
   равно ловится сигналом `review_status='stale'`). Коммит `5b2f0fc`.
2. **Штамп фанился через общие узлы.** Шаг 3g `nacl-sa-feature` штамповал по широкому undirected-обходу
   `*1..6`, расходясь через общий `ACTOR`/`Requirement` — это и есть «broad=760». **Фикс:** штамп
   теперь направленный, по списку затронутых UC (как в `nacl-tl-fix`): задачи затронутых UC + задачи
   UC, которые `DEPENDS_ON` их, + сами UC. `sa_impact_closure` остался широким — он для разведки, не
   для гейта. Коммит `e610046`.

## 4-bis. Skill-level подтверждение (независимые агенты на клоне)

Поверх графового бенчмарка — два **независимых агента** (не автор скиллов) исполнили
реальные скиллы против клона, через `docker exec cypher-shell` (в сессии нет MCP-neo4j,
`itsalt-pinch` не установлен, поэтому не `claude -p`, а sanctioned Agent-инструмент):

- **`nacl-sa-feature`** — фича-изменение UC-033 («пересборка ролика с другим саундтреком»).
  Результат (проверено независимым read-back): UC-033 `spec_version` 0→1; новый `REQ-033-16`;
  **5 задач** помечены `stale` (origin `FR-039`); создан `FR-039` `INCLUDES_UC{modified}`→UC-033;
  записан `DEC-039` с непустым rationale, `IMPLEMENTS`+`JUSTIFIES`. Штамп остался **точечным**
  (ровно поддерево UC-033: 5 Task + 1 UC, без фан-аута), **L9 остался чистым** (скилл сам
  записал Decision).
- **`nacl-tl-plan`** (инкрементальный) — обнаружил UC-033, перегенерировал его 5 задач
  (`MERGE`, **дубликатов нет: 5→5**), `planned_from_version` 0→1, снял флаги; глобальный
  stale **6→0**; другие UC не тронуты; повторный прогон детектора → 0 (идемпотентно).

**Прогон поймал 3 реальных дефекта скиллов — все исправлены:**

| # | Дефект (нашёл независимый агент) | Фикс |
|---|---|---|
| 1 | Штамп шага 3g — широкий undirected-обход, фан-аут через общий ACTOR/Requirement (broad=760) | направленный штамп по списку затронутых UC (коммит `e610046`) |
| 2 | Запрос штампа 3g — `WITH 1 AS _` давал картезиан, row-count ≠ число помеченных задач (вводит в заблуждение) | два чистых statement'а, `RETURN count(DISTINCT t)` |
| 3 | Детектор tl-plan — арм «FR-modified + spec-complete» переметил **12 UC** (sticky-статус), хотя дрейфнул только 1 → перегенерация current-задач, сброс in-progress | убран авто-арм; два drift-confirmed сигнала (version-drift + stale-stamp); `INCLUDES_UC` только для `--feature` scoping |

Плюс мелкие заметки (нет MCP-fallback в прозе скилла; формат свойств Requirement/ActivityStep
оставлен на convention-matching; противоречие в чеклисте 3g — поправлено).

> Это и есть ценность skill-level прогона: графовый бенчмарк доказал, что *механизм* работает;
> независимые агенты доказали, что *текст скиллов* действительно приводит агента к нужным
> записям — и попутно вскрыли 3 дефекта, которые графовый прогон (использовавший мой Cypher)
> не мог поймать, потому что я же его и писал.

## 5. Граница честности

- Графовый бенчмарк (§1–3) измеряет **графовое поведение** механизма (штамп → блок → очистка),
  применяя ровно те записи, что делают скиллы. Skill-level подтверждение (§4-bis) дополняет его:
  реальные скиллы исполнены независимыми агентами против клона. Запуск был не `claude -p` (нет
  `itsalt-pinch`), а через sanctioned Agent-инструмент с записью в граф через `docker exec` —
  не полноценный CLI-subprocess, но независимый агент, читающий и исполняющий SKILL.md.
- Прогон на **одном реальном проекте** (family-cinema). Цифры специфичны его графу (особенно «broad»,
  который зависит от плотности общих ролей/требований). Воспроизводимо на любом графе тем же харнессом.
