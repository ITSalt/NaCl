## 0. Где мы остановились и что такое resilience-контракт

Фаза 3 дала отказам имена. Но что система ДЕЛАЕТ, когда названный отказ случился — где лежит
кэш, который спасёт оффлайн; когда он перестаёт врать; что видит пользователь вместо упавшего
медиа — всё это по-прежнему жило прозой в требованиях. У family-cinema таких требований ~30 у
19 UC: «кэширование в LocalStorage», «fallback при недоступности AI», «Retry-After», «офлайн:
восстановление после перезагрузки». Изменение эндпоинта, ошибки или состояния экрана не
дотягивалось ни до одного из них.

Фаза 4 поднимает это в два узла. **`CachePolicy`** — политика кэширования одной серверной
поверхности данных: `storage_kind` (где), **`invalidation_kind` — REQUIRED** (когда кэш
перестаёт врать: «забыли инвалидацию» — это фабрика staleness-багов, зеркало цепочки
rationale→then→code/message), `serves_stale` (можно ли показывать устаревшее).
**`DegradationRule`** — правило деградации опыта одного UC: `trigger_kind` (отказ / оффлайн /
нет capability), **`behavior` — REQUIRED** (что наблюдает пользователь вместо полного опыта —
зеркало `slice.then`), `fallback_kind` (чем система отвечает: cached_data, static_content,
alternate_provider, alternate_ui, skip_unit, backoff — все шесть значений эмпирически выросли
из требований fc).

```
(:Module)-[:HAS_CACHE]->(:CachePolicy)              // каталог модульный: кэши шарятся
(:CachePolicy)-[:CACHES]->(:APIEndpoint)            // anchor: какую поверхность кэшируем
(:UseCase)-[:HAS_DEGRADATION]->(:DegradationRule)   // правила — per-UC
(:DegradationRule)-[:ON_ERROR]->(:DomainError)      // причина (точка прицепки Фазы 3)
(:DegradationRule)-[:DEGRADES_TO]->(:ScreenState)   // место (точка прицепки Фазы 1)
```

У деградации впервые есть и причина (ошибка Фазы 3), и место (состояние Фазы 1), и
наблюдаемое поведение — ровно то, что обещал финал предыдущей статьи.

## 1. Эмпирика раньше схемы — третий раз как штатный процесс

Разведка по клону до фиксации дизайна — и снова все находки изменили дизайн:

1. **~30 requirements у 19 UC несут кэш/fallback/offline-семантику** — богатейшие: UC-006
   (4 offline-требования), UC-033 (7 resilience), UC-026 (rate-limit с Retry-After).
   Следствие: producer-источник №1 — требования, формулировки verbatim.
2. **Деградация уже живёт на BA-уровне.** BusinessRule BRQ-010 «Восстановление сессии» и
   BRQ-011 «Fallback при ошибке» реализованы требованиями ПЯТИ разных UC — requirements
   цитируют BRQ-id прямо в тексте. Следствие: новый класс producer-источника
   (`IMPLEMENTED_BY`-бэкреф) — и решающий довод в споре о родителе правила: один BA-принцип
   даёт ПЯТЬ разных SA-деградаций (fallback-изображение ≠ смена провайдера ≠ skip-клип) —
   правило UC-скоупно.
3. **TTL в реальном графе почти не существует.** Единственная явная инвалидация — событийная
   (redeem промокода → invalidateQuotaCache). Следствие: `ttl_seconds` опционален и REQUIRED
   только при `invalidation_kind='ttl'`; словарь обязан нести `event`/`manual`/`session`;
   правило «никогда не выдумывай TTL, которого нет в требовании».
4. **Кэши шарятся между UC** (квотный кэш инвалидирует один UC, читает хедер всех экранов;
   медиа-кэш фигурирует у двух UC). Следствие: каталог политик — модульный, как HAS_ERROR.
5. **Три UC живут с ДВУМЯ модулями** (по два `CONTAINS_UC`). Фазы 1–3 на это не напарывались;
   для module-парентного каталога это новый guard: «2+ модуля → спроси владельца каталога».
6. **Все пять имён рёбер свободны** — вторая фаза подряд без namespace-шеринга (включая
   проверку, что grep-совпадения `ON_ERROR` — это подстрока `VALIDATION_ERROR` в прозе).

Два решения, на которых дизайн держится:

**1.1. Асимметричные родители.** CachePolicy — Module (shared-инфраструктура, MERGE по id,
shared-stamp по образцу ошибок); DegradationRule — UseCase (per-опыт, инфикс `DEG-NNN-` для
scoped-фильтра, как SLC). Один слой — два скоупа, и оба выведены из эмпирики, не из теории.

**1.2. Anchor-инвариант ≥1-из-2 вместо тотального ON_ERROR.** Оффлайн и отсутствие capability
не порождаются доменной ошибкой — «оффлайн» не поднимает ни один эндпоинт, и форсить
ERR-OFFLINE-узлы значило бы плодить мёртвый словарь в каждом модуле. Поэтому error-правила
якорятся ON_ERROR (kind-required, зеркало L10.2), UI-правила — DEGRADES_TO, правило без обоих —
проза = запрещено (зеркало L11.2, exemption нет). И решение «не делать»: **ребра
CachePolicy↔DegradationRule нет** — join существует через эндпоинт и экран; L13.8 (INFO)
подсвечивает несхождение. Это уже третий отказ от дублирующего ребра за роадмап (Slice→Error
в Фазе 3, Screen→CachePolicy здесь же) — каждый сэкономил объект валидации консистентности.

## 2. Валидатор L13: задел Фазы 3 получает потребителя

Одиннадцать чеков по выверенной калибровке: структура — CRITICAL (orphan'ы, родители, якоря,
same-UC + канал-правило для DEGRADES_TO, целостность целей пяти рёбер, пустые
`invalidation_kind`/`behavior`, ttl без секунд), полнота — WARNING с именем команды в
сообщении (закрыватель — человек, урок L12.7), несхождение слоёв — INFO.

Два чека стоят отдельного абзаца. **L13.6** — первый потребитель `retryable`, задела,
оставленного Фазой 3 «специально для Фазы 4»: backoff-правило на ошибке с `retryable=false` —
спекфикция «ретраим неретраябельное» (и срабатывает только на явном false: NULL — это
«неизвестно», не находка). **L13.7** — «кэшированная поверхность без истории деградации» —
обязан якориться на `CACHES`, а не на наличие ошибок: completeness-чек, заякоренный на
Phase-3-артефактах, зажёгся бы на каждом error-only графе и сломал бы vacuous-арм №2. Гейт
зажигается только ПОСЛЕ принятия кэш-слоя — opt-in остаётся opt-in даже в полноте.

## 3. Демо на одном UC: реальный оффлайн-кейс

Verify-before-bulk на UC-006 «Просмотр результата» — богатейшем offline-UC графа: политики из
реальных REQ (LocalStorage-сессия, IndexedDB-медиа), офлайн-рестор деградирует В content-стейт
(показ stale-данных — штатная деградация, не ошибка), media-fallback по BRQ-011 присоединяется
к silent-презентации Фазы 3 («пользователь никогда не видит сообщение об ошибке» — теперь это
триада: правило + ошибка + презентация). Все 11 чеков — ноль; по старому allow-list — ноль
отовсюду; штамп — ровно 1 задача + 1 UC.

Демо снова уточнило карту путей: DA→CachePolicy за **3 hops** через `Module-HAS_CACHE` —
третий каталожный шорткат подряд. А DA→DegradationRule — за **4 hops** через `USES_FORM`,
не за 5 через экран: урок Фазы 2 («overlay-слоям мост RENDERS не нужен») воспроизвёлся и здесь.

## 4. Бенчмарк: четыре гипотезы на 29 реальных UC — и первый чистый run 1

| Гипотеза | Результат |
|---|---|
| **H0** Vacuous ×2 (0 узлов; 31 машина + 93 слайса + **58 ошибок** без кэша) | ✓ 11/11 — ноль, оба арма; матрица L10+L11+L12 не шелохнулась |
| **H1** Старый allow-list не дотягивается; новый — дотягивается | ✓ 29/29, бинарно: по старому 0 отовсюду; UC→CACHE 2 hops, UC→DEG 1, api→CACHE 1, ERR→DEG 1, DA→CACHE 3 (шорткат) / 5, DA→DEG 4 |
| **H2** Штамп точечный | ✓ 29/29: **0.86** задачи/UC против широкого **44.9** (~**52×**) + shared-арм: ровно consumer-UC, origin = id политики |
| **H3** 27 классов дефектов | ✓ 27/27 против **полной матрицы L10+L11+L12+L13 (42 чека)**, cross-talk 0 в обе стороны |

**Честная остановка, которой не случилось.** Впервые за роадмап run 1 — OVERALL PASS без
единой находки. Это не везение: копилка граблей трёх фаз (AS-алиасы многострочных выражений,
агрегация с группирующим литералом, `[{NULL,…}]` у map-collect, заранее заявленные
мульти-детекции) была применена в харнессе ДО прогона. Лабораторный журнал — это страховка,
которая в четвёртый раз окупилась тем, что выплата не понадобилась.

**Плато вместо роста.** Тренд широкого штампа остановился: 20× → 49× → 51× → 52× → **52×**.
Широкий радиус насытился — 1301 задача от UC и 1257 от политики байт-в-байт повторяют цифры
Фазы 3: через каталоги ошибок широкий обход УЖЕ видел весь кластер, и новые рёбра не нашли
ему новых жертв. Монотонное удорожание сменилось насыщением; вывод не меняется — «широкий
обход — только разведка».

## 5. Skill-level прогон: adoption-order на самом неудобном кейсе

Схема прежняя: исполнитель читает только скилл-тексты, слепой верификатор — только граф.
Кейсы — UC-006 (UI, реальные офлайн-требования) и UC-026 (неудобный: backend-only, ноль
форм/задач/эндпоинтов, таксономии нет, RuntimeContract — легаси flat-строки).

UC-026 дал лучший момент прогона: guard «error-правила без таксономии не якорятся → сначала
errors» **сработал как задуман** — исполнитель легально исполнил `errors UC-026` (два
provisional-эндпоинта по операциям, `ERR-GENERATION_RATE_LIMIT_EXCEEDED` rate_limit/429/
retryable, MAY_RAISE-only), вернулся в resilience и довёл цепочку: `CACHE-QuotaRedis`
(server/event «суточный reset UTC») + `DEG-026-RateLimitBackoff` (error/backoff на retryable
ошибке — L13.6 проверен на живом кейсе). Самое содержательное резильенс-знание UC-026 лежало
в легаси-RC-строке («backend возвращает Retry-After; серверного повтора нет») — правило
Фазы 3 «строки RC — подсказка, не якорь» оказалось ровно тем, что нужно.

Верификатор: **PASS, 0 CRITICAL / 0 WARNING**; ровно 3 stale-узла во всём графе; дельты
сошлись поштучно (+9 узлов / +18 рёбер); выборочная сверка behavior-текстов с требованиями —
выдумок нет; **0 ошибок cypher-shell** у исполнителя.

Журнал неоднозначностей — 10 пунктов, 8 стали фиксами текста (роль context-query;
boundary-инвалидация = event с именем границы; Redis → `server`; chaining-правило для
errors-first detour; канал-правило явно SCREEN-scoped; tie-breaker цели DEGRADES_TO —
«состояние, в котором пользователь проживает деградированный опыт»; мульти-модульный шаблон
отчёта; принцип serves_stale — false там, где stale ведёт к неверному решению). Два пункта
сознательно не правились — паритет с предыдущими фазами (NNN-паддинг по примеру, как SLC;
no-op-детекция прозой, как errors). Фиксы текстовые — механика не менялась, перепрогон
бенчмарка не потребовался.

## 6. Сквозные уроки

- **Каталожный parent воспроизводим как класс**: `Module-HAS_CACHE` дал тот же 3-hop шорткат
  и тот же фан-аут, что `Module-HAS_ERROR`. Это уже не наблюдение — это свойство паттерна.
- **Широкий радиус насыщается.** Пять точек: 20× → 49× → 51× → 52× → 52×. Монотонный рост
  кончился, когда каталоги покрыли кластер; закон «широкий обход — только разведка» от этого
  только окреп.
- **Задел работает, если у него есть имя и потребитель.** `retryable` ждал одну фазу и получил
  два потребителя (L13.6, L13.7) — против INCLUDES_UC Фазы 0, которое три фазы писалось в
  пустоту, пока его не прочитал tl-plan.
- **Копилка граблей — это амортизация.** Первый чистый run 1 куплен тремя фазами
  задокументированных провалов харнессов.
- **Чужой прогон на неудобном кейсе — главный поставщик фиксов, четвёртую фазу подряд**:
  8 фиксов текста, и половина — с backend-кейса UC-026.
- **Асимметрия скоупов — это нормально**, когда выведена из данных: module-каталог кэшей +
  UC-скоупные правила в одном слое; оба решения дала эмпирика (§1), не симметрия с прошлым.

## 7. Что мы намеренно НЕ сделали

- **Не ввели ребро CachePolicy↔DegradationRule** (join через endpoint/экран; L13.8 INFO).
- **Не ввели UI-ребро Screen→CachePolicy** — видимость кэша в UI выводима.
- **Не сделали слой обязательным**: vacuous-pass доказан двойным армом, включая «все три
  предыдущих слоя есть, кэша нет».
- **Не моделируем rate-limit-квоты узлами** — Retry-After живёт строкой `behavior`.
- **Не бэкфиллим ~30 resilience-требований миграцией** — producer предлагает per-UC.
- **Не пушим.** Роадмап Фаз 0–4 уезжает одним релизом; Фаза 4 — последняя серия локальных
  коммитов перед ним.

## 8. Итог роадмапа

Фаза 4 закрывает роадмап: пять фаз, пять слоёв, один инвариант. Изменение любого узла —
UC, атрибута, эндпоинта, ошибки, политики — дотягивается до всех зависимых артефактов
направленным точечным штампом (0.86 задачи против 44.9 у широкого обхода), «почему» отвечается
цепочкой Decision-узлов, а каждый новый тип узла рождается с required-родителем, cross-layer
якорем и своим L-уровнем валидатора — осиротеть структурно нельзя. 42 чека L10–L13 ловят
27+21+14+8 классов поломок без единого cross-talk. Резильенс-слой Фазы 4 впервые связал
троицу «причина (ошибка) → место (состояние) → ответ (кэш/фолбэк)» в графе — и сделал это
тем же циклом, что и все фазы до него: эмпирика → дизайн → демо → бенчмарк → независимый
прогон → фиксы. Роадмап готов к релизу; сам релиз — отдельное решение владельца.
