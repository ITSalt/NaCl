# Отчёт об эксперименте: Claude Code Dynamic Workflows на реальном фреймворке (NaCl)

> ℹ️ **О приватности.** `family-cinema` — **собственный проект автора, который он готов показывать**, поэтому
> его имя и специфику кода (kie.ai/Suno, имена сервисов, REQ-id, найденные баги) **можно называть в статье** —
> это добавляет реалистичности. Единственное косметическое: абсолютные локальные пути (`/Users/...`,
> `/tmp/...`) стоит подчистить/обобщить в публикуемом тексте — это шум, а не приватность. Никаких клиентских
> данных в эксперименте нет.
>
> **Назначение документа:** максимально технический, причинно-следственный лог эксперимента для агента,
> который будет писать статью «Пока все рассказывают про Claude Code Workflow, мы попробовали и делимся
> результатами». Не литература — факты, цифры, цепочки «причина → следствие → что изменили».

## 0. Среда и действующие лица

- **Claude Code 2.1.156** (фича Dynamic Workflows доступна с 2.1.154), модель **Opus 4.8**.
- **NaCl** — наш фреймворк разработки на базе Claude Code: ~57 markdown-«скиллов» (`SKILL.md`), трёхслойный
  пайплайн BA → SA → TL с графом Neo4j как источником истины. Оркестраторы (`*-full`, `nacl-tl-conductor`,
  `nacl-migrate` и т.д.) делегируют под-агентам через Task tool, **последовательно**, с человеческими
  гейтами. Своих JS-воркфлоу до эксперимента не было (каталог `.claude/workflows/` отсутствовал).
- **Dynamic Workflow** (фича Anthropic) — JS-скрипт, который рантайм Claude Code исполняет в фоне; примитивы:
  `phase()`, `agent(prompt, {schema, model, …})`, `parallel([...])` (барьер), `pipeline(items, …stages)`
  (поточный конвейер без барьера), `log()`, глобал `args`, `budget`. Скрипт начинается с
  `export const meta = {...}`. Запуск — через инструмент Workflow (инлайн `script` либо `scriptPath`),
  исполнение фоновое, по завершении приходит уведомление.

## 1. Исходная задача (с чего начали)

Пользователь прислал пост из Telegram-канала про релиз Opus 4.8 и Dynamic Workflows и поставил задачу:
1. **Проверить факты** из поста по свежим источникам Anthropic.
2. Понять, **как новые возможности применить к нашему фреймворку**.
3. Отдельно: **стоит ли использовать workflow вместо «параллельного вызова скиллов»**, предложить сценарии.

Слово «workflow» в запросе → среда явно предписала использовать инструмент Workflow. Мы были в plan mode
(требуется согласованный план перед действиями).

## 2. Фаза 0 — проверка фактов

**Метод:** 3 параллельных Explore-агента (веб-источники + чтение нашего кода) + сверка с собственным
описанием инструментов (примитивы воркфлоу документированы в системном промпте, поэтому часть утверждений
проверяется напрямую).

**Итог: пост точен примерно на 90%.** Подтвердилось: Opus 4.8 (релиз 2026-05-28, цена $5/$25, ~в 4 раза реже
«молча» пропускает баг в собственном коде); фича workflows с примитивами phase/agent/parallel/pipeline;
триггер по слову «workflow»; сохранение прогона как slash-команда; `/deep-research` из коробки; требование
2.1.154+; Max/Team — по умолчанию, Pro — тумблер в `/config`; подтверждение перед первым запуском; Fast mode
для Opus 4.8 (~2.5× быстрее, $10/$50); ползунок «effort» на claude.ai.

**Корректировки (важно для честности статьи):**
- **Desktop — уже доступен (GA), а не «скоро».**
- Конкретный рецепт `/deep-research` («5 углов / 3 критика / 2 из 3 отклоняют») — реконструкция блогера, в
  документации нет (есть «fan-out по углам, кросс-проверка, голосование»).
- «70% лимита на тривиальном запросе» — непроверенный анекдот; токенов воркфлоу ест существенно больше — это
  правда, но конкретной цифры в источниках нет.

## 3. Что выяснили концептуально (две поправки, определившие весь дизайн)

1. **NaCl не делает «параллельных вызовов скиллов».** Оркестраторы — это в основном *последовательная*
   делегация через Task tool с человеческими гейтами. Значит, вопрос не «заменить параллельные вызовы на
   workflow», а «внести параллелизм/голосование/циклы туда, где проза заставляет идти последовательно — и
   заплатить за это токенами».
2. **Нельзя раздавать механическую работу LLM-агентам.** Гонять 50 Cypher-проверок как 50 *агентов* —
   расточительно. Воркфлоу окупается там, где работа **требует суждения И параллелится И выигрывает от
   независимых перспектив / состязательной проверки**: ревью, аудиты, триаж находок валидатора, диагностика.

**Правило принятия решения (вывели и зафиксировали):** workflow — только когда работа judgment-heavy +
параллелится + выигрывает от состязательной проверки. Иначе — markdown-скилл (интерактив, гейты,
последовательность) или **детерминированный код** (механический Cypher/git/файлы). Рычаг стоимости («маленькие
модели — большие задачи»): механических агентов сажать на **Haiku 4.5**, синтез/верификацию — на Opus.

## 4. Постановка условий задачи (о чём договорились)

Через серию вопросов пользователю зафиксировали:
- **Что произвести:** документ-стратегия **+ один рабочий пилот** (а не только рассуждения).
- **Цель пилота:** **критик-панель код-ревью** (`nacl-tl-review`) — у него чистое разделение
  детерминированных гейтов и LLM-суждения, и есть готовый контракт вывода.
- **Способ интеграции:** **аддитивный опциональный слой** — воркфлоу лежат в `.claude/workflows/`,
  markdown-скиллы остаются каноном и фоллбэком (портируемость: workflows есть не у всех планов/версий).
- **Дисциплина бенчмарка (гейт принятия):** ничего не катим широко, пока не измерим воркфлоу head-to-head
  против markdown-базлайна. Это прямое следствие наших правил: «verify before bulk changes»,
  «publishable benchmarks».

**Вмешательство пользователя (ключевое):** «фикстур и рассуждений мало — без теста на реальном проекте все
гипотетические рассуждения бесполезны». → В план добавили **двухфазную валидацию**: Фаза 1 — герметичная
фикстура (дёшево, go/no-go), Фаза 2 — head-to-head на **реальном проекте, который назовёт пользователь**.
Это сохранили в долговременную память как принцип.

## 5. Архитектура пилота `nacl-review-panel`

Файл: `.claude/workflows/nacl-review-panel.js`. Замысел: **drop-in producer** того же вывода, что
`nacl-tl-review` (тот же контракт — заголовок/статус-строка/имя артефакта), но ревью разложено в
критик-панель с состязательной проверкой.

**Ключевое разделение:** детерминированные гейты `nacl-tl-review` (repo-wide `pnpm -r lint/typecheck/test`,
stub-gate, nav-actions Cypher, baseline-worktree, author-independence) — **механические, остаются
детерминированными** (один агент/скрипт, Haiku); параллелится только **LLM-суждение** — 8 BE / 10 FE
категорий чеклиста.

**Поток (фазы воркфлоу):**
1. `Context` — получить дифф + спеку (Haiku).
2. `Gates` — `parallel()` (это правомерный барьер: нужны все результаты, чтобы решить про short-circuit).
   Если сработал блокирующий гейт (repo RED / nav-actions / critical stubs / NO_INFRA / REGRESSION) — панель
   пропускается, сразу синтез с соответствующим `REVIEW APPLIED — BLOCKED (…)`.
3. `Review`+`Verify` — `pipeline(DIMENSIONS, reviewStage, verifyStage)` (**без барьера**: находки каждой
   категории проверяются, как только категория закончила). reviewStage — по агенту на категорию, вывод через
   **schema** (`{category, triState, findings[]}`, finding = severity/file/line/title/description/fix/rationale).
   verifyStage — на каждую BLOCKER/CRITICAL/MAJOR-находку отдельный «скептик».
4. `Synthesize` — барьер-**dedup** (агент схлопывает одинаковые корневые причины из разных категорий — это
   возвращает естественную дедупликацию, которую один markdown-агент делает неявно, а fan-out ломает) →
   **чистая JS-функция `assignHeadline()`** (порт ~12-строчной таблицы приоритетов Step 8b из
   `nacl-tl-review/SKILL.md` + правило P4: `APPROVED` только при `REVIEW COMPLETE`) → агент-писатель рендерит
   артефакт по шаблону `nacl-tl-core/templates/review-template.md`.

**Почему так:** механическую таблицу заголовков (12 строк приоритета) проза заставляет агента перереференсить
каждый прогон — в JS это детерминированная тестируемая функция. Это и есть «выигрыш кода над прозой».

**Фикстура** (`bench/fixtures/review-panel/`): крошечный NestJS-бэкенд UC с **размеченной истиной** — один
*реальный* BLOCKER (SQL-инъекция через интерполяцию строки) и один *ложный* (будто «нет авторизации на
deleteOrder», на деле гард на уровне контроллера) — чтобы проверить обе стадии (находим реальное, отбраковываем
ложное).

## 6. Итерация 1 — Фаза 1, герметичный прогон на фикстуре

**Прогон №1 (без тюнинга):** 16 агентов, **592 743 токена**, 96 tool-uses, **345 c**. Вывод: `REVIEW COMPLETE`
/ `CHANGES REQUESTED`, **26 находок** (2 BLOCKER / 0 CRIT / 14 MAJOR / 10 MINOR).
- ✅ SQL-инъекция-BLOCKER **пережила** состязательную проверку (скептик её не отбраковал).
- ✅ Состязательная проверка **отбраковала 4 завышенных CRITICAL**, в т.ч. два «надо 404, а не null» — что
  *корректно по спеке* (acceptance EH01: «нет записи → null»); скептик оказался spec-aware.
- ✅ **Неожиданно поймала кросс-категориальный BLOCKER**, который я не закладывал: в фикстуре сервис
  реализует `GET /orders` (listOrders), но в контроллере нет роута `@Get()` → метод недостижим. Это ровно
  тот класс «метод есть, но не подключён», который per-метод-ревью пропускает.
- ❗ **Проблема — шум:** 24 MAJOR/MINOR на дифф в 95 строк. verify шёл только по BLOCKER/CRITICAL, MAJOR/MINOR
  не фильтровались; часть MAJOR — педантизм (нет JSDoc, нет миграции индекса, «нет commit message в диффе»).
- ❗ **Нет дедупликации:** одна и та же первопричина приходила от 2–3 категорий.

**Тюнинг (следствие шума):** (а) ужесточили scope reviewer-промпта (только дифф, не выдумывать про
миграции/CI/commit-метаданные); (б) распространили verify на MAJOR; (в) добавили **барьер-dedup** в синтезе.

**Прогон №2 (с тюнингом):** 16 агентов, **521 836 токенов**, 419 c, **12 находок** (1B/1C/3Maj/7Min),
лог `Dedup: 14 → 12`. → Тюнинг **срезал шум ~на 54%** (26 → 12), дедуп сработал. *(Этот прогон оказался
случайным чистым A/B на той же фикстуре — см. промах #1.)*

## 7. Промах #1 — `args` не прокидываются через `scriptPath`

При запуске Фазы 2 я передал в инструмент Workflow `args` (taskId, diffPath на реальный проект и т.д.), но
**прогон №2 отревьюил снова фикстуру** (в логе `task=UC001 … Context: 3 changed file(s)`). Причина: при запуске
воркфлоу через **`scriptPath`** объект `args` **не дошёл** до глобала `args` скрипта — сработали дефолты
(дефолты были настроены на фикстуру). На CC 2.1.156 воспроизвелось стабильно.

**Следствие/обходной путь:** для параметризованного прогона генерировали **вариант-скрипт с зашитым
конфигом** (заменяли `const a = args || {}` на литерал-объект) и запускали его через `scriptPath`. Записали как
gotcha в `.claude/workflows/README.md`. *(Побочный плюс: прогон №2 дал бесплатный замер эффекта тюнинга.)*

## 8. Итерация 2 — Фаза 2, head-to-head на реальном проекте (UC-037-BE)

**Цель:** проверить пилот на реальном коде. **Проект — `family-cinema`** (собственный проект автора, готов к
показу: pnpm-монорепо, Fastify/Next, ~6000 TS-файлов, свой Neo4j). **Таск — UC-037-BE** (админ-дашборд пайплайнов: REST list/detail +
SSE, репозиторий, Zod-схемы), дифф реализации **1484 строки**, 7 файлов, запинен на wave-tip SHA.

**Дизайн сравнения (важно для статьи — мы намеренно держали гейты вне сравнения):** гейты детерминированы и
одинаковы при любой архитектуре ревьюера, поэтому сравнивали **только слой суждения**: панель vs **один сильный
агент-базлайн** (Opus, тот же дифф+спека+чеклист). Панель работала в режиме «как оттюнинговано» (diff-only),
базлайн — Task-агент со **свободным доступом к репозиторию**.

**Результат — самый важный негативный результат эксперимента:**

| | Панель (как оттюнинговано, diff-only) | Базлайн (один агент, доступ к репо) |
|---|---|---|
| Вердикт | **`APPROVED`** (НЕВЕРНО) | **`CHANGES REQUESTED`** (верно) |
| Токены / время | 1 072 026 / ~619 c / 22 агента | 89 417 / ~170 c / 1 агент |
| Находки | 0B / 1C / 8Maj / 9Min | 2B / 3C / 3Maj / 3Min |

**Решающее событие (цепочка причин):** reviewer панели **нашёл** реальный production-баг (рассинхрон словаря
статусов шагов: дифф эмитит `'running'/'completed'`, а рантайм пишет `'processing'/'ready'` — SSE/прогресс
сломаны) и оценил его как **BLOCKER**. Затем **собственная стадия verify его отбраковала** (`refuted`,
confidence *medium*) и выкинула. Причина — скептик в режиме **diff-only** не видел рантайм-файл
(`pipeline-orchestrator.service.ts`), доказывающий баг, а промпт говорил «default to refuted, если не уверен».
→ С исчезновением BLOCKER `assignHeadline` увидел 0 блокеров → `APPROVED`. Базлайн же со свободным доступом
прочитал рантайм, подтвердил баг (и что тестовые фикстуры используют тот же фейковый словарь, *маскируя* баг) и
дал `CHANGES REQUESTED`. **On-disk пост-фикс этого файла в проекте уже исправлен** — т.е. баг был исторически
реальным, а панель его «одобрила».

**Сравнение находок:** совпали (LIKE-эскейпинг, проблемы SSE-auth/JWT_SECRET, тяжёлый detail-запрос, дыры
покрытия тестов). **Только базлайн:** словарь-статусов-BLOCKER (панель отбраковала), фейковые фикстуры-BLOCKER,
**REQ-037-05 не реализован** (CRITICAL, пропущенное требование), 401-vs-403. **Только панель:** секрет
`ADMIN_API_KEY` в URL `?token=` → утечка в логи (CRITICAL, сильная), утечка heartbeat `setInterval`
(MAJOR), non-constant-time сравнение токена, и т.п. (выигрыш в *широте*).

**Выводы итерации 2 (причинно):**
1. **Скоупинг важнее архитектуры.** Базлайн поймал худшие баги *потому что читал за пределами диффа*. Наш
   анти-шумовой тюнинг «только дифф» + «refute-if-uncertain» сделал панель структурно слепой к кросс-файловым
   и пропущенным-требованиям дефектам и заставил её **их активно выбрасывать**.
2. **Ложный `APPROVED` — худший тип отказа для ревью-гейта.**
3. **Баг калибровки вердикта:** `APPROVED` требовал только «0 блокеров»; выживший CRITICAL не блокировал.
4. Стоимость реальна: ~12× токенов и ~3.6× времени за неверный вердикт.

## 9. Промах #2 (методологический) — асимметрия доступа к репозиторию

В итерации 2 базлайн имел свободный доступ к репо, а панель — diff-only. Это **нечестно**, но именно это и
вскрыло вывод (скоупинг доминирует). Честный следующий эксперимент — панель *с доступом к репо* vs агент *с
доступом к репо*. Это и сделали в итерации 3.

## 10. Три фикса (что изменили по итогам провала)

1. **Доступ к репозиторию для reviewer'ов И verifier'ов.** Убрали «review ONLY the diff»; добавили
   `repoAccessNote`: «репо здесь — читай зависимости диффа: импорты, вызываемые функции, типы, рантайм,
   миграции, контракт, потребителей изменённых значений».
2. **Verifier: отбраковывать только по позитивному доказательству.** Промпт: «refuted=true ТОЛЬКО если нашёл
   доказательство, что находка неверна (прочитал гард/обработчик/потребителя); если не смог подтвердить —
   `needs_context=true`, refuted=false, **СОХРАНИТЬ** находку; никогда не отбраковывать на одной
   неуверенности; BLOCKER — только при high-confidence». В схему добавили `needs_context`. **Логика отброса:**
   выкидываем находку **только** при `refuted && confidence==='high'`; иначе сохраняем (помечая needs_context).
3. **Калибровка вердикта + reviewer требований.** `computeVerdict`: `APPROVED` только при 0 BLOCKER **и 0
   CRITICAL**. Добавили **девятый «dimension» — Requirements Traceability** (читает спеку, по каждому REQ/AC
   проверяет реализацию end-to-end; отсутствие → BLOCKER/CRITICAL/MAJOR).

Каждый фикс синтаксически верифицировали (оборачивали скрипт в async-функцию через `new Function` —
top-level `return` валиден в рантайме, но не в голом ESM; первая наивная проверка `node --check` дала ложную
ошибку «Illegal return statement» — это тоже зафиксировано).

## 11. Промахи #3–#4 — git-археология при подготовке большого таска

Пользователь попросил **таск побольше**. Выбрали **UC033** (самый большой: FULL BE+FE, ~7K строк диффа, 50
файлов; к тому же **QA-failed-after-approved** — есть QA-отчёт с реальными багами). Ревьюили **BE-сторону**
(пайплайн-сервисы, ~2751 строк — логика ffmpeg/оркестрации/ретраев).

- **Промах #3:** все 12 BE-файлов UC033 **отсутствуют на HEAD** — пайплайн с тех пор переписан («durable
  pipeline на pg-boss»). → Ревью на HEAD невозможно; нужен исторический коммит.
- **Промах #4:** `git diff <empty-tree> <commit> -- <files>` дал **0 строк** (тихий quirk). → Обошли через
  `git diff --no-index -- /dev/null <file>` внутри запиненного worktree.
- **Решение (правильное):** создали **`git worktree --detach` на build-коммите `d4f13b4`** в `/tmp/fc-uc033`,

## 13. Выводы (эмпирически, на двух реальных тасках)

1. **Режим отказа закрыт.** Доступ к репо + verifier-по-доказательствам + калибровка вердикта превратили
   ложный `APPROVED` в корректный `CHANGES REQUESTED`, ловящий тот же кросс-файловый BLOCKER, что и сильный
   одиночный агент. Планка гейта («ловить блокеры, не одобрять лишнего») достигнута.
2. **Fan-out даёт не только широту, но и глубину.** Выделенный Correctness-reviewer нашёл гонку, которую
   универсал явно отмёл как «sound». SSRF и AF-2-баг — реальные и severe.
3. **Но ни один не является надмножеством другого, и цена ~15×.** Базлайн поймал 2 реальных бага, которые
   панель пропустила, при том же вердикте, за 1/15 токенов и 1/4.5 времени.
4. **Раннее «одиночный агент выигрывает» — это на самом деле «выигрывает доступ к репо», а не архитектура.**
   Самый дешёвый высокоценный ход — просто дать markdown-ревьюеру доступ к репозиторию.

**Обновлённое правило:**
- **Дефолт для рутинного per-UC ревью — один сильный агент *с доступом к репо*** (≈15× дешевле, тот же
  вердикт, хедлайн-баг пойман в обоих прогонах).
- **Панель — opt-in инструмент *аудита*** для high-stakes/широты (когда выявить SSRF + гонку + AF-2 +
  пробелы требований за один проход стоит 15× токенов).
- **Перед полным `bench/`-замером — срезать стоимость панели:** dimension-reviewer'ы на Sonnet (Opus оставить
  requirements/verifier/dedup), verify ограничить BLOCKER/CRITICAL (MAJOR-проход дал основной вклад в 15× и
  мало пользы). Затем мерить cost-adjusted качество (precision/recall vs размеченный набор).

**Прочие концептуальные выводы:**
- **workflow ≠ skill ≠ детерминированный код** — три разных инструмента; раздавать механику агентам нельзя.
- **workflow ≠ `/goal`** — комплементарны (у `/goal` свой transcript-only evaluator + GOAL_PROOF), не
  замена; компонуемы.
- **Дисциплина бенчмарк-гейта сработала:** именно она остановила преждевременное принятие после ложного
  `APPROVED` и потребовала фиксов+перезамера.

  Вопрос 1: годятся ли workflow для самого NaCl?

  Сначала — те четыре кандидата из «Bucket A» (workflows-for-nacl.md), которые мы наметили в начале:

  1. Критик-панель код-ревью (nacl-tl-review) — наш пилот.
  2. Аудит/пост-мортем панель (3-агентный ретроспективный аудит миграции + «skill-postmortem»).
  3. Триаж находок валидатора (nacl-sa-validate / nacl-ba-validate).
  4. Diagnose-свип (nacl-tl-diagnose: история git / дрейф доков / здоровье кода / паттерны регрессий).

  Теперь — переоценка каждого через главный измеренный факт: панель стоит ~15× токенов одиночного агента с доступом к репо, при том же вердикте по
  хедлайн-багу. То есть решающий ингредиент корректности — не fan-out, а доступ к контексту + проверка по доказательствам. Fan-out добавляет глубину

  Вопрос 2: вне NaCl — когда «дикий расход» оправдан и что в workflow делать?

  Принцип один: 15× оправданы, когда ценность широты/глубины/параллелизма/независимой верификации в этот момент превышает стоимость. Конкретно —
  расход адекватен при совпадении нескольких условий:

  - A. Редко + high-stakes. Запуск не на каждый коммит, а изредка, и цена ошибки высока: предрелизный аудит безопасности, пост-мортем инцидента,
  due-diligence чужого кода, разовый аудит миграции. Амортизированная цена низкая.
  - B. Работа реально параллельная и декомпозируемая. N независимых элементов/измерений/углов, каждый требует суждения, и важно wall-clock. Workflow
  превращает «сумму задержек» в «максимум». (Ревью 50 PR, аудит 200 файлов, fact-check 30 утверждений — это и есть встроенный /deep-research.)
  - C. Независимые перспективы / состязательная проверка реально повышают корректность. Когда у одного прохода есть слепые зоны, которые ловят разные
  специалисты или скептики (мы это видели: гонка + отбраковка ложных находок). Берите, когда ложные срабатывания или пропущенные углы дороги.
  - D. Масштаб больше одного контекстного окна. Миграции/аудиты на тысячи файлов, которые один агент не удержит → fan-out + детерминированный харнес
  (pipeline, loop-until-dry) — единственный способ.
  - E. Нужна детерминированная оркестрация (циклы, барьеры, дедуп, голосование, бюджет-капы), которую проза не исполнит надёжно.

  Что делать внутри workflow, чтобы расход был адекватен (рычаги, которые мы вывели на своей шкуре):

  1. Механику — вон из агентов. Детерминированный код/Cypher/grep/git — в один агент или скрипт; fan-out — только на LLM-суждение. (Гонять 50
  проверок как 50 моделей — деньги на ветер.)
  2. Дай агентам контекст (доступ к репо/файлам), который им нужен. Иначе они выдают неверное / переотбраковывают — и весь расход потрачен впустую.
  Дешёвый контекст бьёт дорогую слепоту — это наш центральный урок.
  3. Сажай дешёвые модели туда, где суждение мелкое (Haiku/Sonnet на сбор контекста, механику, простые проверки), Opus — только на синтез и
  состязательную верификацию. Рычаг «маленькие модели — большие задачи».
  4. Состязательная проверка — только по позитивным доказательствам, не «refute-if-uncertain» (иначе убивает реальные находки), плюс барьер-дедуп,
  чтобы снять дубли fan-out.
  5. Явно ограничивай стоимость: бюджет-aware циклы, verify только по high-severity, ширина fan-out под задачу; логируй, что отбросил (молчаливое
  усечение читается как «всё покрыто»).
  6. Ширину масштабируй под ставки: пара «искателей» для «найди баги», большая панель + мульти-голосование для «тщательный аудит».

  Хорошие сценарии для не-NaCl-пользователя:
  - Предрелизный/предмердж глубокий аудит крупного/рискового изменения (безопасность + корректность + перф + a11y параллельно + состязательная
  проверка) — раз перед поставкой.
  - Deep research / fact-checking с кросс-проверкой (встроенный /deep-research).
  - Крупная миграция/рефактор по тысячам файлов: discover → transform каждого (worktree-изоляция) → verify.
  - Пост-мортем инцидента/регрессии: мульти-модальный свип (логи, история git, диффы, метрики) → синтез.
  - Онбординг в незнакомую кодовую базу: параллельные читатели по подсистемам → структурированная карта.
  - Турнир решений: N независимых попыток с разных углов → судейская панель → синтез лучшего.

  Когда НЕ надо (инверсия): рутинное, per-commit, однофайловое, гейт-интерактивное или механическое — там одиночный хорошо-промптованный агент (или
  просто скрипт) дешевле и не хуже. 15× выброшены, когда задача мелкая, частая или ответ локален в диффе.

  Одной фразой: workflow — это редкий силовой инструмент для глубины/широты/масштаба под высокие ставки, а не дефолтный цикл работы. Планка
  «оправдано» намеренно высокая — наш же эксперимент показал, что бо́льшую часть корректности можно получить от одного агента с доступом к репо, а
  fan-out добавляет убывающую отдачу за большие деньги.
