# Бенчмарк Фазы 4: политики кэша и деградации в графе NaCl

**Дата:** 2026-06-06 · **Статус:** ПОДТВЕРЖДЕНО · **Цель:** `family-cinema` (граф 1913 узлов / 2693 рёбер), изолированный клон.

> ℹ️ **О приватности.** `family-cinema` — собственный демо-проект автора; имя, код, UC/REQ-id
> называются открыто. Абсолютные локальные пути заменены на плейсхолдеры. Клиентских данных нет.

---

## 0. Что проверяли

Фаза 4 — последний слой роадмапа — добавила в SA-слой **resilience-контракт**: `CachePolicy`
(политика кэширования одной серверной поверхности данных: где хранится, когда инвалидируется —
`invalidation_kind` REQUIRED, можно ли отдавать устаревшее) и `DegradationRule` (как деградирует
опыт одного UC при отказе / оффлайне / отсутствии capability: `behavior` REQUIRED — наблюдаемое
деградированное поведение, зеркало `slice.then`). Рёбра: `HAS_CACHE` (required parent — каталог
**модульный**, кэши — shared-инфраструктура bounded context, зеркало HAS_ERROR), `CACHES`
(anchor-инвариант: политика без поверхности — мёртвый словарь; provisional-эндпоинты легальны),
`HAS_DEGRADATION` (required parent — правила **UC-скоупны**, зеркало Slice: один BA-принцип
деградации даёт несколько per-опыт правил), `ON_ERROR` (причина — точка прицепки Фазы 3;
kind-required для `trigger_kind='error'`), `DEGRADES_TO` (место — точка прицепки Фазы 1;
same-UC-правило + канал-правило для error-правил). Anchor-инвариант правила: ≥1 из
(ON_ERROR, DEGRADES_TO), exemption нет. Ребра CachePolicy↔DegradationRule НЕТ намеренно
(join через endpoint/экран; L13.8 INFO подсвечивает несхождение — зеркало решения №5 Фазы 3).
Производитель — `nacl-sa-uc resilience`, второй производитель — `nacl-tl-fix` (L2/L3
cache/degradation-ветки), потребитель — `nacl-tl-plan` (секция «Cache & Degradation»),
валидатор — **новый уровень L13** (L8–L12 заняты). `DomainError.retryable` — задел Фазы 3 —
получил штатного потребителя: L13.6 (backoff-правило на retryable=false-ошибке) и фильтр L13.7.

Особенность Фазы 4: **все пять имён рёбер снова новые** (проверено по обеим схемам,
скилл-текстам и `db.relationshipTypes()` живого графа; grep-совпадения `ON_ERROR` — подстрока
`VALIDATION_ERROR` в прозе, не тип ребра) — вторая фаза подряд без namespace-шеринга, H1
бинарный.

### Гипотезы (фальсифицируемые)

- **H0 (vacuous pass, дважды).** (a) На графе без единого Phase-4 узла все L13-чеки дают ноль;
  (b) на графе со **всеми тремя предыдущими слоями** (31 машина + 93 слайса + 58 доменных
  ошибок) и нулём кэша/деградации — тоже ноль, причём полная матрица L10+L11+L12 не меняется:
  внедрение трёх предыдущих слоёв само по себе L13 не включает.
- **H1 (достижимость).** По **старому** (до Фазы 4) allow-list `sa_impact_closure` ни
  CachePolicy, ни DegradationRule недостижимы ни от UC, ни от эндпоинта, ни от DomainError,
  ни от DomainAttribute; по **новому** — изменение достигает политики (UC: 2 hops; эндпоинт:
  1 hop) и правил (UC: 1 hop; ошибка: 1 hop); от DomainAttribute — с измерением фактических
  min-hops (потолок `*1..6` не должен быть упёрт).
- **H2 (точечный штамп + shared-арм).** После authoring resilience-слоя направленный штамп
  (контракт 3g) помечает ровно задачи UC + задачи `DEPENDS_ON*1..5`-зависимых + сам UC; широкий
  undirected-обход по расширенному набору рёбер пометил бы кратно больше (пятая точка тренда
  20×→49×→51×→52×→?). **Плюс новая семантика Фазы 4:** модификация contract-свойств shared-кэша
  помечает ровно consumer-UC (+ их задачи) с `stale_origin` = id политики — направленно.
- **H3 (детекторы детектируют).** Каждый из 27 классов инжектированных дефектов поджигает ровно
  свои чеки по **полной матрице L10+L11+L12+L13 (42 чека)**, cross-talk 0 в обе стороны:
  мусорные `CACHES`/`ON_ERROR`/`DEGRADES_TO` не поджигают L10/L12, а L10/L11/L12-дефекты не
  поджигают L13.

## 1. Установка (воспроизводимо)

Клон тот же, что в Фазах 0–3: изолированный `nacl-bench-neo4j` (bolt 7798, neo4j:5-community +
APOC), потоковый APOC-экспорт рабочего fc-графа через HTTP → `cypher-shell -f` (команды — §1
phase0-отчёта). Применены constraints Фазы 4 (`constraint_cachepolicy_id`,
`constraint_degradationrule_id` + 4 индекса из `sa-schema.cypher`).

**Выборка:** все UC клона с `USES_FORM`, `has_ui` и **модулем** — 29 из 31 (UC-039/UC-040 без
`CONTAINS_UC` честно пропущены — producer-guard «нет модуля → degradation-only»). Каждому
программно создаются каноническая машина Фазы 1, три слайса Фазы 2, две ошибки Фазы 3 и
**канонический resilience-слой ровно тем Cypher, что прописан в `nacl-sa-uc resilience`**:
одна module-парентная политика (`indexed_db`/`manual`/`serves_stale`, CACHES на
provisional-эндпоинт) + два правила — offline/`cached_data` → DEGRADES_TO content-состояние
(деградация В content со stale-данными — штатный случай) и error/`static_content` → ON_ERROR
retryable-external-ошибки + DEGRADES_TO Error-состояние (канал-правило выполнено: эффект экрана
зовёт тот же эндпоинт). **Полный сброс** между каждым измерением.

## 2. Результаты

### H0 — vacuous pass ✓ (дважды)

На чистом клоне (0 Phase-4 узлов) все 11 L13-чеков = 0. После авторинга **31 машины + 93
слайсов + 58 доменных ошибок** и нуля кэша/деградации — все 11 снова 0, и **вся матрица
L10+L11+L12 без изменений** (все нули). Проекты, внедрившие машины, слайсы и таксономию ошибок,
но не resilience-слой, от L13 не страдают.

### H1 — достижимость изменения ✓ (29/29, снова бинарный)

| Откуда тянем | Старый allow-list | Новый allow-list (min hops) |
|---|---|---|
| UC (29 проб) | **0** политик, **0** правил | 1/1 политика (**2 hops**), 2/2 правила (**1 hop**) |
| APIEndpoint (29 проб) | **0** | 1/1 политика (**1 hop**) |
| DomainError (29 проб) | **0** | 2/2 правила (**1 hop** до error-правила) |
| DomainAttribute (18 проб*) | **0** | 1/1 политика + 2/2 правила; CACHE: **3 hops** ×11 / **5 hops** ×7; DEG: **4 hops** ×18 |

\* у 11 из 29 форм нет ни одного `MAPS_TO`-атрибута — проба честно пропущена (как в Фазах 1–3).

Три вывода: (1) регистрация всех пяти рёбер в allow-list **обязательна** — без неё весь
resilience-слой структурно невидим для импакт-анализа; (2) у DA→CachePolicy два пути:
**3 hops через `Module-HAS_CACHE`** (когда сущность атрибута в том же модуле — третий
каталожный шорткат подряд после HAS_ERROR) и 5 hops через `USES_FORM`+`EXPOSES`; DA→DEG —
**4 hops через USES_FORM** равномерно (урок №2 Фазы 2 воспроизводится: для overlay-слоёв
USES_FORM короче экранного пути; экранный путь через DEGRADES_TO = 5 — потолок `*1..6` нигде
не упёрт, запас ≥1 hop); (3) отсутствие шеринга снова сделало H1-таблицу бинарной.

### H2 — штамп остаётся точечным ✓ (29/29 + shared-арм)

| Метрика | Значение |
|---|---|
| Помечено точечным штампом, всего (29 UC) | **54** узла (25 задач + 29 UC) |
| Широкий undirected-радиус от UC (Task-узлы), всего | **1301** |
| Широкий радиус от одной политики, всего | **1257** |
| Среднее на UC: точечный / широкий | **0.86 / 44.9 задачи** (~**52×**) |

Пятая точка тренда: 20× → 49× → 51× → 52× → **52×**. Впервые тренд НЕ вырос — широкий радиус
**насытился**: 1301 задача от UC — байт-в-байт цифра Фазы 3, и 1257 от политики — байт-в-байт
фан-аут Module-каталога ошибок. Новые рёбра Фазы 4 не добавили широкому обходу новых жертв,
потому что через каталоги Фазы 3 он уже видел весь кластер. Урок «каталожный parent = шорткат
И фан-аут» воспроизведён на втором каталоге подряд; правило «широкий обход — только
разведка/отображение» остаётся законом.

**Shared-арм:** политика `CACHE-Bench003IndexedDb` кэширует поверхности UC-003 (invoked) и
UC-004 (consumer); модификация её contract-свойства (`serves_stale` против before-image;
bookkeeping `updated` — не сигнал) через ран UC-003 → помечено **ровно 3 задачи** (объединённый
affected-set обоих UC, посчитан множеством) и **ровно 2 UC**; у consumer-UC и задач
`stale_origin = CACHE-Bench003IndexedDb` (lineage указывает на общий артефакт). Семантика
дословно повторяет shared-error штамп Фазы 3 — на этот раз без правок формы: форма 3g была
выверена там.

## 3. Что бенчмарк нашёл и что мы починили

**Ничего — run 1 чистый: OVERALL PASS с первого прогона.** Это первая фаза без находок
бенчмарка (Фаза 1 — 2 неверные экспектации; Фаза 2 — 1; Фаза 3 — дефект формы 3g + грабля
cypher-shell + 1 экспектация). Причина прозаична и воспроизводима: копилка граблей трёх фаз
была применена в харнессе ДО прогона — обязательные `AS`-алиасы у многострочных
COUNT{}-выражений, обёртка агрегаций с группирующим литералом, comprehension-фильтры
map-collect'ов, мульти-детекции заявлены заранее. Накопленный лабораторный отчёт — это
дешёвая страховка следующего харнесса.

## 3-bis. Skill-level подтверждение (независимые агенты на клоне)

Исполнитель (читал только скилл-тексты) исполнил `resilience` для **UC-006** «Просмотр
результата» (UI-кейс: 4 реальных offline/fallback-требования REQ-UC006-R07/R09/R10/R11,
BA-правила BRQ-010/011, машина и таксономия предсозданы по adoption-order) и **UC-026**
«Управление rate limit» (неудобный кейс: backend-only, 0 форм / 0 задач / 0 эндпоинтов,
таксономии нет, RC — легаси flat-строки). Результат UC-006: 2 политики
(`CACHE-ResultLocalStorage` local_storage/session, `CACHE-ResultMediaIndexedDb`
indexed_db/session) + 2 правила (`DEG-006-MediaFallback` error/static_content с
ON_ERROR+DEGRADES_TO Loaded — behavior цитирует BRQ-011 дословно; `DEG-006-OfflineRestore`
offline/cached_data). Результат UC-026: **adoption-order guard сработал** — исполнитель
легально исполнил сначала `errors UC-026` (2 provisional-эндпоинта по операциям +
`ERR-GENERATION_RATE_LIMIT_EXCEEDED` rate_limit/429/retryable, MAY_RAISE-only), затем
`resilience`: `CACHE-QuotaRedis` (server/event «суточный reset UTC + инкремент») +
`DEG-026-RateLimitBackoff` (error/backoff, ON_ERROR, без DEGRADES_TO — backend-режим).
Направленные штампы 1+1 и 0+1; spec_version 0→1 и 0→2 (два рана); scoped-L13 PASS дважды;
**0 ошибок cypher-shell**.

**Слепой верификатор** (не видел вывода исполнителя) независимо считал граф: вердикт **PASS,
0 CRITICAL / 0 WARNING** (1 INFO-наблюдение). Особо: ровно **3 stale-узла** во всём графе
(1 задача + 2 UC), ноль фан-аута; дельты узлов/рёбер сошлись поштучно (+9 узлов / +18 рёбер);
выборочная сверка behavior/invalidation с текстами требований — выдумок нет.

**Прогон принёс 10 пунктов в журнал неоднозначностей; 8 стали фиксами скилл-текста:**

| # | Дефект (нашли независимые агенты) | Фикс |
|---|---|---|
| 1 | Inline context-query не назван; роль `sa_uc_resilience` (MODIFY-load + read-back) неявна | роли названы явно в Phase 1.1 |
| 2 | Выбор `invalidation_kind` для boundary-инвалидации (суточный reset, сессия) не определён | правило: именованная граница = `event` с именем в `invalidation_event`; `session` = до перезаписи следующей сессией; `manual` = явные purge |
| 3 | `storage_kind` для Redis не определён | маппинг: Redis/memcached → `server`, process-local → `memory` |
| 4 | errors-first detour: рекомендация без определённого автономного пути | chaining-правило: интерактив — стой на гейте; санкционированный chain — исполни errors и вернись в Phase 1; error-правило без ON_ERROR не пишется |
| 5 | Проза канал-правила читалась state-scoped, валидатор screen-scoped | проза явно SCREEN-scoped как L12.3 (любой эффект экрана; не обязан вести в целевое состояние) |
| 6 | Выбор цели DEGRADES_TO при расхождении handling-state и «места проживания» | tie-breaker: состояние, в котором пользователь ПРОЖИВАЕТ деградированный опыт |
| 8 | Шаблон отчёта хардкодил один модуль при живом 2+-module guard | шаблон поддерживает мульти-модульный UC |
| 10 | `serves_stale` без принципа выбора | true только где stale допустим по требованию; false где ведёт к неверному решению (квоты/лимиты) |

Пункты 7 (NNN-паддинг задан примером — паритет с SLC) и 9 (no-op-детекция без канонического
diff-запроса — паритет с errors) сознательно не правились: симметрия с предыдущими фазами,
дисциплина на producer. Фиксы — текстовые уточнения; канонический Cypher, валидатор и
id-форматы не менялись, поэтому **перепрогон бенчмарка после фиксов не требовался** (в
Фазе 3 перепрогон был нужен, потому что фикс менял форму 3g — эталонную механику).

## 4. Демо на одном реальном UC (verify-before-bulk)

До массового прогона resilience-слой был вручную создан для **UC-006** ровно шаблонами
producer'а: 2 политики из реальных требований (LocalStorage-сессия + IndexedDB-медиа),
2 правила (offline-рестор в Loaded — деградация В content; media-fallback по BRQ-011 c
silent-презентацией Фазы 3). Все 11 L13-чеков = 0; санити L10/L12 = 0; closure достигает
CACHE от UC (2 hops), от эндпоинта (1 hop), от DomainAttribute (**3 hops** — каталожный
шорткат `Module-HAS_CACHE`, видный ещё на демо); по старому списку — 0 отовсюду; штамп —
ровно 1 задача + 1 UC; `sa_uc_resilience` рендерится. Клон после сброса — байт-в-байт
1913/2693.

## 5. Граница честности

- Канонические политики/правила бенчмарка — синтетические по содержанию (`Bench*`-id),
  канонические по форме; **реальные** формулировки (BRQ-010/011, Redis-квоты, Retry-After)
  покрыты демо (UC-006) и skill-прогоном (UC-006 + UC-026), не графовым харнессом.
- `H2_shared` измерен на одной паре UC (UC-003/UC-004) — достаточно для семантики штампа,
  статистики по парам нет.
- UC-039/UC-040 исключены из выборки осознанно (нет модуля — для кэшей guard деградирует в
  degradation-only; в харнессе пропущены целиком).
- DA-пробы есть у 18 из 29 UC (у остальных формы без `MAPS_TO` — известный долг fc-графа).
- «Плато 52×» — свойство графа family-cinema (широкий радиус насытился его кластером);
  на большем графе тренд может продолжить рост. Харнесс воспроизводим на любом графе.
- Фиксы skill-прогона применены ПОСЛЕ эталонного прогона без перепрогона — обоснование выше
  (текстовые уточнения, механика не менялась).
