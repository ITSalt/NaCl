# Бенчмарк Фазы 3: таксономия доменных ошибок в графе NaCl

**Дата:** 2026-06-06 · **Статус:** ПОДТВЕРЖДЕНО · **Цель:** `family-cinema` (граф 1913 узлов / 2693 рёбер), изолированный клон.

> ℹ️ **О приватности.** `family-cinema` — собственный демо-проект автора; имя, код, UC/REQ-id
> называются открыто. Абсолютные локальные пути заменены на плейсхолдеры. Клиентских данных нет.

---

## 0. Что проверяли

Фаза 3 добавила в SA-слой **транспортно-независимую таксономию доменных ошибок**: `DomainError`
(именованный, каталогизированный режим отказа — `PROMO_NOT_FOUND`; `code` — источник истины,
HTTP-статус — только проекционная подсказка) и `ErrorPresentation` (как ошибка предъявляется
пользователю: текст на языке пользователя + вид предъявления). Рёбра: `HAS_ERROR`
(required parent — каталог **модульный**, ошибки — shared-словарь bounded context),
`MAY_RAISE` (anchor-инвариант: ошибка без поднимающего эндпоинта — мёртвый словарь; легален на
provisional-эндпоинтах), `HANDLES` (канал-правило вместо same-UC: экран обязан реально звать
поднимающий эндпоинт), `PRESENTED_AS` (required parent презентации), `SHOWS` (замыкание
треугольника state→presentation→error). Производитель — `nacl-sa-uc errors`, второй
производитель — `nacl-tl-fix` (L2/L3 error-ветки), потребитель — `nacl-tl-plan` (секция
«Domain Errors» в task-файлы), валидатор — **новый уровень L12** (L8–L11 заняты).

Особенность Фазы 3: **все пять имён рёбер новые** (проверено по обеим схемам, скилл-текстам и
`db.relationshipTypes()` живого графа) — первая фаза без namespace-шеринга. Это значит, что
впервые нет «случайно достижимого» класса в H1, а cross-talk-армы H3 проверяют изоляцию
матрицы, а не шеринг конкретного имени.

### Гипотезы (фальсифицируемые)

- **H0 (vacuous pass, дважды).** (a) На графе без единого DomainError-узла все L12-чеки дают
  ноль; (b) на графе с **31 машиной экранов И 93 слайсами** и нулём ошибок — тоже ноль, причём
  полная матрица L10+L11 не меняется: внедрение двух предыдущих слоёв само по себе L12 не
  включает.
- **H1 (достижимость).** По **старому** (до Фазы 3) allow-list `sa_impact_closure` ни
  DomainError, ни ErrorPresentation недостижимы ни от UC, ни от эндпоинта, ни от
  DomainAttribute; по **новому** — изменение достигает всех ошибок (UC: 2 hops; эндпоинт:
  1 hop) и презентаций (UC: 3 hops); от DomainAttribute — с измерением фактических min-hops
  (потолок `*1..6` не должен быть упёрт).
- **H2 (точечный штамп + shared-арм).** После authoring ошибок направленный штамп (контракт 3g)
  помечает ровно задачи UC + задачи `DEPENDS_ON*1..5`-зависимых + сам UC; широкий
  undirected-обход по расширенному набору рёбер пометил бы кратно больше. **Плюс новая
  семантика Фазы 3:** модификация свойств shared-ошибки помечает ровно raiser-UC (+ их задачи)
  с `stale_origin` = id ошибки — направленно, без широкого обхода.
- **H3 (детекторы детектируют).** Каждый из 21 класса инжектированных дефектов поджигает ровно
  свои чеки по **полной матрице L10+L11+L12 (32 чека)**, cross-talk 0 в обе стороны: мусорные
  `HANDLES`/`SHOWS` (источник — ScreenState, территория L10) не поджигают L10-чеки, а
  L10/L11-дефекты не поджигают L12.

## 1. Установка (воспроизводимо)

Клон тот же, что в Фазах 0–2: изолированный `nacl-bench-neo4j` (bolt 7798, neo4j:5-community +
APOC), потоковый APOC-экспорт рабочего fc-графа через HTTP → `cypher-shell -f` (команды — §1
phase0-отчёта). Применены constraints Фазы 3 (`constraint_domainerror_id`,
`constraint_errorpresentation_id` + 3 индекса из `sa-schema.cypher`).

**Выборка:** все UC клона с `USES_FORM`, `has_ui` и **модулем** — 29 из 31 (UC-039/UC-040 без
`CONTAINS_UC` честно пропущены: producer-guard «UC без модуля → STOP» снят с них и в харнессе).
Каждому программно создаётся каноническая машина Фазы 1, три канонических слайса Фазы 2 и
**две канонические доменные ошибки ровно тем Cypher, что прописан в `nacl-sa-uc errors`**:
`*_NOT_FOUND` (not_found, 404) и `*_UPSTREAM_FAILED` (external, 503, retryable) — обе
module-parented, MAY_RAISE на provisional-эндпоинте, HANDLES от Error-состояния (канал-правило
выполнено: SCREF-эффект зовёт тот же эндпоинт), презентации с русскими user-language
сообщениями, SHOWS замыкает треугольник. **Полный сброс** между каждым измерением.

## 2. Результаты

### H0 — vacuous pass ✓ (дважды)

На чистом клоне (0 ошибок) все 11 L12-чеков = 0. После авторинга **31 машины + 93 слайсов и
нуля ошибок** — все 11 снова 0, и **вся матрица L10+L11 без изменений** (32/32 нулей).
Проекты, внедрившие машины и слайсы, но не таксономию ошибок, от L12 не страдают.

### H1 — достижимость изменения ✓ (29/29, впервые без «случайного» класса)

| Откуда тянем | Старый allow-list | Новый allow-list (min hops) |
|---|---|---|
| UC (29 проб) | **0** ошибок, **0** презентаций | 2/2 ошибки (**2 hops**), 2/2 презентации (**3 hops**) |
| APIEndpoint (29 проб) | **0** | 2/2 (**1 hop**) |
| DomainAttribute (18 проб*) | **0** | 2/2 ошибки + 2/2 презентации; ERR: **3 hops** ×11 / **5 hops** ×7; ERRP: **4 hops** ×11 / **5 hops** ×7 |

\* у 11 из 29 форм нет ни одного `MAPS_TO`-атрибута — проба честно пропущена (как в Фазах 1–2).

Три вывода: (1) регистрация всех пяти рёбер в allow-list **обязательна** — без неё весь
error-слой структурно невидим для импакт-анализа (по старому списку — нули отовсюду);
(2) у DA→ERR два пути: **3 hops через `Module-HAS_ERROR`** (когда сущность атрибута в том же
модуле — родительское ребро каталога даёт шорткат, открытый ещё на демо) и 5 hops через
экран — потолок `*1..6` нигде не упёрт, запас ≥1 hop; (3) отсутствие namespace-шеринга
сделало H1-таблицу бинарной — нет класса, который «случайно работал до регистрации», как
CALLS-якорные слайсы Фазы 2.

### H2 — штамп остаётся точечным ✓ (29/29 + shared-арм)

| Метрика | Значение |
|---|---|
| Помечено точечным штампом, всего (29 UC) | **54** узла (25 задач + 29 UC) |
| Широкий undirected-радиус от UC (Task-узлы), всего | **1301** |
| Широкий радиус от одной ошибки, всего | **1257** |
| Среднее на UC: точечный / широкий | **0.86 / 44.9 задачи** (~**52×**) |

Тренд подтверждается четвёртый раз и растёт монотонно: 20× (Фаза 0) → 49× (Фаза 1) → 51×
(Фаза 2) → **~52×** (Фаза 3). Широкий обход через `Module-HAS_ERROR` теперь дотягивается от
любой ошибки почти до всего кластера (1257 задач с 29 ошибок) — ещё один аргумент, почему
правило «широкий обход — только разведка/отображение» обязано оставаться законом.

**Shared-арм:** ошибка `ERR-BENCH003_UPSTREAM_FAILED` поднята эндпоинтами UC-003 (invoked) и
UC-004 (raiser); модификация её contract-свойства через ран UC-003 → помечено **ровно 3
задачи** (= объединённый affected-set обоих UC; задачи мульти-owner — пересечение посчитано
множеством, не суммой) и **ровно 2 UC**; у raiser-UC и его задач `stale_origin =
ERR-BENCH003_UPSTREAM_FAILED` (lineage указывает на общий артефакт). Примечание о порядке:
если задача принадлежит обоим UC, raiser-штамп перезаписывает origin своего предшественника —
оба штампа точечные, побеждает последний; семантической потери нет (оба origin ведут к одному
изменению).

### H3 — детекторы детектируют ✓ (21/21, cross-talk 0 по матрице из 32 чеков)

| Инжектированный дефект | Ожидаемые чеки | Сработало |
|---|---|---|

| Осиротевшая презентация | L12.0 + L12.1 + L12.2 | ✓ |
| Ошибка без модуля-родителя | L12.1 | ✓ |

| Unraisable-ошибка (MAY_RAISE снят) | L12.2 + L12.3 (канал HANDLES стал фикцией) | ✓ |
| Unshown-презентация (SHOWS снят) | L12.2 + L12.8 | ✓ |
| HANDLES в не-DomainError | L12.3; **все L10/L11 молчат** | ✓ |
| HANDLES от чужого экрана (канал-нарушение) | L12.3 + L12.8 | ✓ |

## 3. Что бенчмарк нашёл и что мы починили

1. **Дефект формы 3g в первом драфте shared-штампа (пойман ревью харнесса ДО эталонного
   прогона):** второй statement штамповал UC-узлы всех affected (включая
   `DEPENDS_ON`-зависимых), тогда как контракт 3g штампует узлы только самих изменённых UC —
   зависимые получают штамп задач, не узла. Разнесено на два statement по форме базового
   контракта (фикс в скилле, зеркале и харнессе).
2. **Харнесс-грабля cypher-shell (run 1):** безымянное многострочное выражение
   `RETURN COUNT {…} + COUNT {…}` даёт **многострочный заголовок колонки** в plain-формате —
   парсер скаляра читал строку заголовка и молча возвращал 0 для L12.3/L12.4 (5 армов H3
   «прошли» вслепую, 2 упали). Фикс — обязательный `AS`-алиас; в копилку к грабле Фазы 1
   («агрегация с группирующим литералом → 0 rows»). Чеки самого скилла невиновны — они
   row-returning с алиасами.
3. **Одно неверное ожидание (run 1):** `parentless-presentation` корректно поджигает **два**
   чека — L12.1 и L12.8 (сняв PRESENTED_AS, состояние перестало показывать презентацию *своей*
   ошибки). Детектор прав, ошибалась модель автора — третий раз подряд (две неверные
   экспектации в Фазе 1, одна в Фазе 2): defect-injection ловит дефекты понимания чаще, чем
   дефекты кода.

## 3-bis. Skill-level подтверждение (независимые агенты на клоне)

Исполнитель (читал только скилл-тексты) исполнил `errors` для **UC-035** «Активация промокода»
(UI-кейс: машина существует, **реальные коды прямо в требованиях**) и **UC-033** «Обработка и
монтаж видеоролика» (неудобный кейс: backend-only, эндпоинтов нет — provisional-путь с нуля,
RuntimeContract в **легаси flat-формате**). Результат: 6 ошибок UC-035 с кодами дословно из
REQ-035-04…09 (включая `410` для blocked/expired/capacity — приоритет источников «требование
дословно > таблица киндов» сработал как задуман), полный треугольник handling+презентаций;
4 ошибки UC-033 MAY_RAISE-only на 2 provisional-эндпоинтах (trigger + status, гранулярность
по операциям); направленные штампы 0+1 и 5+1; scoped-L12 PASS дважды; **0 ошибок
cypher-shell**.

**Слепой верификатор** (не видел вывода исполнителя) независимо считал граф: вердикт **PASS,
0 CRITICAL / 0 WARNING** (2 косметических INFO-наблюдения). Особо: общий stale-набор — ровно
**7 узлов** (5 задач + 2 UC), ноль фан-аута; дельты узлов/рёбер сошлись поштучно
(+18 узлов / +42 ребра).

**Прогон принёс 8 пунктов в журнал неоднозначностей; 7 стали фиксами скилл-текста:**

| # | Дефект (нашли независимые агенты) | Фикс |
|---|---|---|
| 1 | MAY_RAISE-only режим не говорил явно, что provisional-путь в нём тоже работает (а backend-UC без EXPOSES — обычный случай) | явная фраза в guard'е |
| 2 | Не задано, какие ошибки вешать на какую поверхность (trigger vs status) | правило + worked-пример: терминальные отказы → оба эндпоинта, read-side → только status |
| 3 | REQ требует `410`, таблица киндов знает только «типичный 409» — не за что заякориться | «статус из требования — verbatim-авторитет; типичная колонка — только фолбэк» |
| 4 | Повторный идемпотентный ран без изменений превращался в «MODIFY» и штамповал | no-op-правило: без фактических изменений Phase 4 пропускается целиком |
| 5 | Scoped-L12 рецепт не покрывал screen-keyed L12.7 и UC-keyed L12.9 (у них нет err.id) | скоуп по UC для обоих, WARNING/INFO — report, never block (фикс и в sa-validate) |
| 6 | `$changeKind` упоминался, но не был определён в секции errors | словарь `{'created','modified'}` назван явно |
| 7 | Триггер shared-штампа «modified» был оценочным (а `err.updated` меняется всегда) | механический триггер: diff contract-свойств против before-image Phase 1 |

8-й пункт (у confirmation-гейтов нет автономной ветки) — общесемейная конвенция всех
`nacl-sa-*`; зафиксирован в HANDOFF, сознательно не правится per-command.

Зафиксированное наблюдение без правки: L12.6a проверяет *пустоту* `message`, но не
смешение алфавитов / не язык — симметрично решению Фаз 1–2 не проверять формат id
(дисциплина — на producer).

## 4. Демо на одном реальном UC (verify-before-bulk)

До массового прогона таксономия была вручную создана для **UC-035** ровно шаблонами
producer'а: 6 реальных кодов из REQ-035-*, process-машина (Idle/Submitting/Completed/Failed),
HANDLES от Failed, русские inline-презентации. Все 11 L12-чеков = 0; L10-санити = 0; closure
достигает ERR от UC (2 hops), от эндпоинта (1 hop), от DomainAttribute (**3 hops** — шорткат
через `Module-HAS_ERROR`, открыт именно демо); по старому списку — 0; штамп — ровно 1 узел
(0 задач + 1 UC: у UC-035 нет задач — честный no-fanout); `sa_uc_errors` рендерится с
handling-таблицей. Клон после сброса — байт-в-байт 1913/2693.

## 5. Граница честности

- Канонические ошибки бенчмарка — синтетические по содержанию (`BENCH*`-коды), канонические по
  форме; **реальные** коды и презентации покрыты демо (UC-035) и skill-прогоном (UC-035 +
  UC-033), не графовым харнессом.
- `H2_shared` измерен на одной паре UC (UC-003/UC-004, мульти-owner задачи) — этого достаточно
  для семантики штампа, но статистики по парам нет.
- UC-039/UC-040 исключены из выборки осознанно (producer-guard «нет модуля → STOP»); они же —
  empirical-основание самого guard'а.
- DA-пробы есть у 18 из 29 UC (у остальных формы без `MAPS_TO` — известный долг fc-графа).
- Цифры специфичны графу family-cinema (особенно «broad»). Харнесс воспроизводим на любом графе.
