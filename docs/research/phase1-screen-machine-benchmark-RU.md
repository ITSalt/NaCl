# Бенчмарк Фазы 1: стейт-машина экрана в графе NaCl

**Дата:** 2026-06-05 · **Статус:** ПОДТВЕРЖДЕНО · **Цель:** `family-cinema` (граф 1913 узлов / 2693 рёбер), изолированный клон.

> ℹ️ **О приватности.** `family-cinema` — собственный демо-проект автора; имя, код, UC-id
> называются открыто. Абсолютные локальные пути заменены на плейсхолдеры. Клиентских данных нет.

---

## 0. Что проверяли

Фаза 1 добавила в SA-слой **детерминированную стейт-машину экрана** — первый из четырёх новых
типов артефактов (за ней Slice / DomainError / Cache). Узлы `Screen` / `ScreenState` /
`ScreenEvent` / **reified** `Transition` / `ScreenEffect` (+ минимальный `AnalyticsEvent`),
производитель — `nacl-sa-ui state-machine`, валидатор — **новый уровень L10** в `nacl-sa-validate`
(L8/L9 заняты staleness/provenance Фазы 0).

Главный риск всех «новых типов узлов» был сформулирован ещё до Фазы 0: **новые артефакты, которые
изменение не дотягивает, усиливают дрейф**. Поэтому бенчмарк измеряет не greenfield-создание, а
**сценарий изменения**: дотягивается ли «потянул за нитку» до нового экрана, остаётся ли гейт-штамп
точечным, и ловят ли новые валидаторы реальные поломки машины.

### Гипотезы (фальсифицируемые)

- **H0 (vacuous pass).** На графе **без единого** Screen-узла все L10-чеки дают ноль находок —
  проекты, не внедрившие стейт-машины, не страдают от нового уровня.
- **H1 (достижимость).** Со **старым** allow-list `sa_impact_closure` изменение UC / его
  DomainAttribute / его APIEndpoint достигает **0** узлов машины; с **новым** — полное поддерево
  машины от UC и Screen от атрибута (через мост `RENDERS`) и от эндпоинта (через `CALLS`).
- **H2 (точечный штамп).** После authoring машины staleness-штамп помечает ровно задачи UC +
  задачи транзитивных `DEPENDS_ON`-зависимых + сам UC; широкий undirected-обход по расширенному
  набору рёбер пометил бы кратно больше.
- **H3 (детекторы детектируют).** Каждый из 8 инжектированных дефектов машины поджигает ровно
  свой L10-чек (с ожидаемым счётом), без cross-talk на остальных; на целой машине все чеки — ноль.

---

## 1. Установка (воспроизводимо)

Клон тот же, что в Фазе 0: изолированный `nacl-bench-neo4j` (bolt 7798, neo4j:5-community + APOC),
потоковый APOC-экспорт рабочего fc-графа через HTTP → `cypher-shell -f` (точные команды — §1
phase0-отчёта). Перед прогоном клон **реклонирован** (паритет: 1913 узлов / 2693 ребра / 31 Decision /
37 UseCase) и применён one-time baseline `planned_from_version` (правило №2 Фазы 0).
Дополнительно применены новые constraints/indexes Фазы 1 (`sa-schema.cypher`).

**Выборка:** все **31 UC** клона с `USES_FORM` (не backend-only) — не cherry-picked. Для каждого
программно создаётся каноническая машина ровно тем Cypher, что прописан в
`nacl-sa-ui state-machine`: 4 состояния (Loading-initial / Loaded / Empty / Error), 3 события
(OnLoaded / OnLoadFailed / OnRetry), 4 перехода (2 guarded-ветки на OnLoaded), 1 load-эффект
`CALLS` эндпоинт (provisional + `EXPOSES`, т.к. в fc-графе эндпоинтов нет — см. §5). **Полный сброс**
(удаление машин + очистка staleness) между каждым измерением.

## 2. Результаты

### H0 — vacuous pass ✓

На чистом клоне (0 Screen-узлов) все 11 L10-чеков = 0 находок. Внедрение L10 не ломает
существующие проекты.

### H1 — достижимость изменения ✓ (31/31)

| Откуда тянем | Старый allow-list | Новый allow-list |
|---|---|---|
| UC (31 проб) | **0** узлов машины | **13/13** (Screen + 4 состояния + 3 события + 4 перехода + эффект) |
| APIEndpoint (31 проб) | **0** | Screen достигнут (все 13 в радиусе 6 hops) |
| DomainAttribute (18 проб*) | **0** | Screen достигнут через `DA←MAPS_TO←FormField←HAS_FIELD←Form←RENDERS←Screen` |

\* у 13 из 31 форм нет ни одного `MAPS_TO`-атрибута — проба пропущена (честно записано `null`).

Это подтверждает обязательность ребра `RENDERS`: без него экран структурно невидим для
изменения домена — что и закреплено валидатором L10.2.

### H2 — штамп остаётся точечным ✓ (31/31)

| Метрика | Значение |
|---|---|
| Помечено точечным штампом, всего (31 UC) | **58** узлов (27 задач + 31 UC) |
| Широкий undirected-радиус (Task-узлы), всего | **1324** |
| Среднее на UC: точечный / широкий | **0.87 / 42.7 задачи** (~**49×**) |

Точечный штамп после authoring машины — тот же направленный контракт, что `nacl-sa-feature` 3g:
задачи UC + задачи транзитивных `DEPENDS_ON*1..5`-зависимых + сам UC. Широкий обход по
расширенному (Фазой 1) набору рёбер фанится через общие ACTOR/Requirement/навигацию ещё сильнее,
чем в Фазе 0 (~49× против ~20×) — **новые рёбра делают broad-штамп ещё опаснее**, правило
«широкий обход только для разведки» становится жёстче.

### H3 — детекторы детектируют ✓ (8/8, без cross-talk)

| Инжектированный дефект | Ожидаемые чеки | Сработало |
|---|---|---|
| Осиротевший эффект | L10.0 + L10.1 + L10.2 | ✓ ровно они |
| Удалён `RENDERS` | L10.2 | ✓ |
| Переход на чужой ScreenState | L10.3 | ✓ |
| Два unguarded на (Loading, OnLoadFailed) | L10.4 | ✓ |
| Два `is_initial=true` | L10.5a | ✓ |
| Недостижимое состояние | L10.5b (QPP-обход) | ✓ |
| Удалён retry-переход | L10.6a + L10.1 | ✓ |
| `CALLS` перенацелен на Form | L10.7a + L10.2 | ✓ |

На целой машине перед каждой инжекцией — все чеки 0 (нет ложных срабатываний).

## 3. Что бенчмарк нашёл и что мы починили

1. **Штамп `nacl-sa-ui` расходился с каноном 3g.** Первая редакция стейт-машинного штампа брала
   только прямых (`*1`) `DEPENDS_ON`-зависимых и помечала UC-узлы зависимых; канон
   `nacl-sa-feature` 3g — транзитивные `*1..5` и второй statement только для самого изменённого UC.
   Расхождение давало бы разный L8-радиус для одного изменения в зависимости от того, какой скилл
   штамповал. **Фикс:** sa-ui приведён к контракту 3g дословно (включая guard `WHERE d IS NOT NULL`
   и `count(DISTINCT t)`).
2. **Два неверных ожидания в первом прогоне H3** (дефекты модели автора, не детекторов):
   удаление retry-перехода оставляет эффекту его `CALLS` (срабатывает L10.1 «нет родителя», а не
   L10.2); перенацеленный `CALLS` корректно поджигает **оба** чека — label-qualified L10.2 «нет
   валидного `CALLS->APIEndpoint`» и L10.7a «wrong-label target». Ожидания исправлены, повторный
   прогон — чистый. Поведение детекторов в обоих случаях правильное по построению.
3. **Несамосогласованность черновика skill-текста** (Phase 2 предлагала 3 события, но переход в
   Error — по четвёртому): исправлено до публикации — события OnLoaded/OnLoadFailed/OnRetry,
   load-эффект на retry-переходе (единственном re-entry в Loading), явная оговорка про
   Idle-состояние для экранов без немедленной загрузки.

## 4. Демо на одном реальном UC (verify-before-bulk)

До массового прогона машина была вручную создана для **UC-006 «Просмотр результата»**
(`FORM-scr-004-result`, 3 mapped-атрибута, задача `TASK-FR029-W4-T3-result-viewer`):
`SCR-ResultViewer` → 24 L10-проверки = 0 строк; `sa_screen_machine` отрисовывает 4 перехода с
guard'ами и эффектом; точечный штамп = ровно 1 задача + 1 UC; closure достигает экрана от UC
(1 hop), от атрибута (3 hops), от эндпоинта (2 hops).

## 5. Граница честности

- В fc-графе **нет узлов `APIEndpoint` и рёбер `EXPOSES`** — допущение плана «есть api-*» не
  выполнилось. Producer получил штатный путь: MERGE **provisional** эндпоинта + `EXPOSES`-якорь к
  UC (не сирота), с пометкой `provisional=true` для обогащения в `nacl-tl-plan`. Бенчмарк
  использует этот же путь — то есть H1-проба «от эндпоинта» бежит от узла, созданного самим
  производителем, а не от исторического.
- `HAS_STATE`/`TRIGGERS` namespace-делятся с BA-слоем, `NAVIGATES_TO` — с легаси-навигацией
  Form→Form. Решение: имена сохранены (зеркалим BA-паттерн, прецедент `HAS_STEP`), все запросы
  label-qualified, риск ограничен display-only ролью широкого обхода. Зафиксировано в схеме.
- L10.5b использует quantified path patterns (Neo4j ≥ 5.9); фолбэк для старых 5.x —
  `apoc.path.expandConfig`.
- Цифры специфичны графу family-cinema (особенно «broad», зависящий от плотности общих
  ролей/требований). Харнесс воспроизводим на любом графе.
