## 0. Где мы остановились и почему экраны — первыми

В Фазе 0 мы починили сценарий изменения: «потянул за UC — граф показал весь blast-radius,
заблокировал релиз до пересмотра, а "почему год назад" отвечается одним Cypher». Доказали
бенчмарком и независимыми агентами. Теперь можно делать то, ради чего всё затевалось — переносить
в NaCl четыре идеи из конкурентного mobile-фреймворка: Behavior Slice, стейт-машины экранов,
таксономию ошибок, политику кэша.

Первой пошла **стейт-машина экрана** — не потому, что самая простая, а по двум причинам:

1. **Это хаб.** Остальные три идеи крепятся к ней: Slice будет `COVERS`-ить состояния и переходы,
   DomainError будет `HANDLES`-иться состоянием экрана, CachePolicy — деградировать в состояние.
   Построй хаб первым — остальным будет к чему присоединяться.
2. **У неё самые сложные валидаторы.** Детерминизм, достижимость состояний, гарантия выхода из
   ошибки — это не «у узла есть родитель», это статический анализ машины Cypher-запросами. Отладив
   их на Фазе 1, мы де-рискуем Фазы 2–4.

И главное правило, выстраданное Фазой 0: **новый тип узла, до которого изменение не дотягивается —
это усилитель дрейфа.** Поэтому каждый шаг Фазы 1 проверялся не вопросом «создаётся ли экран?», а
вопросом «дотянется ли до экрана изменение UC, атрибута домена, эндпоинта?»

## 1. Дизайн: что именно мы кладём в граф

UI-поведение экрана обычно живёт в голове фронтендера и в тикетах: «покажи спиннер, потом список,
если пусто — заглушку, если упало — ретрай». NaCl кладёт это в граф как формальную машину:

```
(:UseCase)-[:HAS_SCREEN]->(:Screen)-[:RENDERS]->(:Form)
(:Screen)-[:HAS_STATE]->(:ScreenState)      // loading / busy / content / empty / error
(:Screen)-[:HAS_EVENT]->(:ScreenEvent)      // user / system / lifecycle
(:Screen)-[:HAS_TRANSITION]->(:Transition)  // reified-узел!
(:Transition)-[:FROM_STATE|TO_STATE]->(:ScreenState), -[:ON_EVENT]->(:ScreenEvent)
(:Transition)-[:TRIGGERS]->(:ScreenEffect)  // load/mutate → CALLS APIEndpoint,
                                            // navigate → NAVIGATES_TO Screen,
                                            // analytics → EMITS AnalyticsEvent
```

Три решения здесь — новаторства, а не оформление:

**1.1. Переход — reified-узел, а не ребро со свойствами.** В Neo4j соблазнительно сделать
`(state)-[:TRANSITIONS_TO {event, guard}]->(state)` — так устроена BA-машина состояний сущности
этажом ниже. Мы сознательно подняли переход до узла `:Transition (SCRTR-*)`, по трём причинам:
(а) валидатору нужен **стабильный id перехода** для отчёта — «переход SCRTR-ResultViewer-003
недетерминирован» полезнее, чем «какое-то ребро между Loading и Error»; (б) только узел может быть
**источником эффектов** (`TRIGGERS -> ScreenEffect`) — у ребра не бывает исходящих рёбер;
(в) reified-узел **сам попадает под orphan-чек** — осиротевший переход виден валидатору, осиротевшее
ребро не существует как сущность вовсе.

**1.2. `RENDERS` — обязательный мост из домена в UI.** Цепочка
`DomainAttribute ← MAPS_TO ← FormField ← HAS_FIELD ← Form ← RENDERS ← Screen` — это путь, по которому
изменение атрибута домена *дотягивается* до экрана. Убери `RENDERS` — и экран структурно невидим
для impact-анализа: меняешь тип поля, а машина экрана «не при делах». Поэтому `RENDERS` — не
опция, а required-ребро с CRITICAL-валидатором (исключение — честно формлес-экраны: splash, 404,
флаг `formless=true`).

**1.3. Инвариант связности как структурная гарантия.** Каждый из шести новых типов узлов обязан
иметь: required-parent ребро + required cross-layer/sibling ребро + свой валидатор. «Максимальная
связность» — не дисциплина команды, а свойство схемы: узел, который можно забыть присоединить,
поймается гейтом. Даже минимальный `AnalyticsEvent` существует ровно потому, что analytics-эффект
без цели был бы тупиком и нарушил бы инвариант.

## 2. Валидатор L10: машина проверяется статически, Cypher-ом

Новый уровень в `nacl-sa-validate` (L8/L9 заняты Фазой 0 — staleness и provenance). Самое
интересное — не orphan-чеки (они зеркалят существующие), а три проверки, которые превращают
описание экрана в **контракт**:

- **Детерминизм (L10.4).** Два перехода из одного состояния по одному событию — неоднозначность,
  *кроме* случая, когда обе ветки под guard'ом (`items > 0` / `items == 0`). Формула: пара
  `(from_state, on_event)` нарушает контракт, если unguarded-переходов больше одного ИЛИ unguarded
  смешан с guarded. Дизъюнктность guard'ов остаётся на совести автора — это честная граница
  статической проверки.
- **Достижимость (L10.5).** Ровно одно `is_initial`-состояние на экран, и каждое прочее состояние
  достижимо из него по направленной цепочке `(:ScreenState)<-[:FROM_STATE]-(:Transition)-[:TO_STATE]->()`.
  Реализовано quantified path pattern'ом (Neo4j ≥ 5.9) — направление чередуется, обычный
  `*`-обход здесь дал бы ложную симметричную достижимость.
- **Гарантия выхода из ошибки (L10.6).** Error-состояние без исходящего перехода — ловушка для
  пользователя (CRITICAL). Есть выход, но ни один не срабатывает по user-событию — нет ретрай-
  аффорданса (WARNING). Проверяем по `event_kind='user'`, а **не по имени** «OnRetry»: имена
  событий бывают на языке проекта, kind — детерминированный инвариант.

И сквозное свойство: **vacuous pass**. Граф без единого экрана проходит L10 с нулём находок —
проекты, не внедрившие стейт-машины, ничего не должны.

## 3. Реальные данные сломали два допущения плана — и это нормально

План Фазы 1 писался по схеме «у family-cinema есть FORM-*, DE-*, api-*». Первый же запрос к клону
показал: **в реальном графе нет ни одного узла `APIEndpoint` и ни одного ребра `EXPOSES`** —
проект жил с RuntimeContract'ами, а TL-слой эндпоинтов так и не материализовался. Допущение «эффект
CALLS существующий api-эндпоинт» не на что было опереть.

Решение вошло в продукт: если эндпоинта нет, producer **MERGE-ит provisional-эндпоинт**
(`provisional=true`) и сразу якорит его `EXPOSES`-ребром к UC — чтобы он не осиротел — а
`nacl-tl-plan` потом обогащает его из api-contracts. Дыра в данных превратилась в штатный путь.

Второе открытие: имена рёбер. `HAS_STATE` уже занят BA-слоем (`BusinessEntity→EntityState`),
`TRIGGERS` — BA-процессами, `NAVIGATES_TO` — легаси-навигацией форм. Мы **сознательно оставили
имена** (зеркалим BA-паттерн; прецедент `HAS_STEP` делил имя между слоями ещё до нас), но с
жёстким правилом: **каждый запрос label-qualified**, голым типом эти рёбра не матчить. Это
безопасно ровно потому, что широкий обход — display-only, а гейт-штамп направленный (правило №1
Фазы 0 здесь окупилось второй раз).

## 4. Бенчмарк: четыре фальсифицируемых гипотезы на 31 реальном UC

Та же методология, что в Фазе 0: изолированный клон реального графа (1913 узлов), выборка не
cherry-picked (все 31 UC с формами), полный сброс между измерениями, харнесс и сырой JSON — в
репозитории.

| Гипотеза | Результат |
|---|---|
| **H0** Граф без экранов проходит L10 чисто | ✓ 11/11 чеков — ноль находок |
| **H1** Старый allow-list не дотягивается до машины; новый — дотягивается | ✓ 31/31: старый — **0** узлов, новый — **13/13** от UC; Screen достижим от DomainAttribute (мост RENDERS) и от APIEndpoint |
| **H2** Штамп после authoring остаётся точечным | ✓ 31/31: точечный **0.87** задачи/UC против широкого **42.7** (~**49×**) |
| **H3** Каждый инжектированный дефект ловится своим чеком, без cross-talk | ✓ 8/8 классов дефектов; ложных срабатываний 0 |

Про H2 стоит сказать отдельно. В Фазе 0 разница точечного и широкого штампа была ~20×. После
Фазы 1 она выросла до ~49×: **каждое новое ребро делает широкий undirected-штамп ещё опаснее**,
потому что у обхода появляются новые пути фан-аута. Правило «широкий обход — только разведка»
с каждой фазой не ослабевает, а ужесточается.

H3 — это defect-injection: на эталонной машине ломаем по одной вещи (двойной initial,
недостижимое состояние, unguarded-неоднозначность, тупиковый error, `CALLS` в Form вместо
эндпоинта, осиротевший эффект, чужое состояние в переходе, снятый `RENDERS`) и проверяем, что
загорается **ровно** ожидаемый набор чеков. Валидатор, который не доказал, что умеет ловить —
это декорация.

**Честная остановка №1.** Первый прогон H3 дал 6/8: на «тупиковом error» и «CALLS в Form» мои
*ожидания* не совпали с реальностью. Разбор показал — детекторы правы, ошибалась модель в голове
автора: удаление retry-перехода оставляет эффекту его `CALLS` (загорается «нет родителя», а не
«нет вызова»), а перенацеленный `CALLS` корректно поджигает **два** чека сразу (нет валидной цели
+ цель неправильного лейбла). Ожидания исправлены, перепрогон чистый. Бенчмарк проверяет не только
код — он проверяет понимание автора.

## 5. Skill-level прогон: рекордер вместо списка — и 4 дефекта текста

Графовый бенчмарк гонял мой Cypher. Отдельный вопрос — приводит ли **текст скилла** независимого
агента к правильным записям. Схема та же, что в Фазе 0, но жёстче: **исполнитель** и
**верификатор** — два разных агента, и верификатор не видел вывода исполнителя
(test-author-isolation в обе стороны).

Исполнителю достался UC-031 «Запись и транскрипция голосовых комментариев» — **намеренно не
data-loading экран**, а рекордер: запись → загрузка (TUS) → транскрипция, два пути отказа.
Канонический шаблон скилла (Loading/Loaded/Empty/Error) сюда не ложится — и это была проверка
на прочность прозы, а не только Cypher-блоков.

Агент справился: построил машину из 6 состояний / 9 переходов / 3 mutate-эффектов, выведя её из
RuntimeContract проекта, проставил направленный штамп (ровно 3 задачи + 1 UC), прошёл все L10.
Верификатор независимо считал граф: PASS, ноль CRITICAL, штамп не задел соседние UC. Но главная
ценность — **журнал неоднозначностей** исполнителя. Четыре пункта оказались реальными дефектами
текста скилла, все починены:

| # | Дефект (нашёл независимый агент) | Фикс |
|---|---|---|
| 1 | Описан только data-loading архетип — рекордер пришлось изобретать самому | второй канонический архетип «process screen» (Idle + busy-стадии + mutate-эффекты; retry без эффекта легален) + явное «эффекты 0..n, канон — guidance, не validator law» |
| 2 | Словарь `state_kind` беден: Recording/Uploading/Transcribing огрубились в `loading` | новый kind `busy` (user-initiated operation in progress) — отличаем «грузим данные» от «работаем по команде пользователя» |
| 3 | Из `scr-031-voice-recorder` нельзя детерминированно получить имя экрана | правило деривации PascalName (`scr-031-voice-recorder` → `VoiceRecorder`) |
| 4 | «Прогони scoped L10» — а рецепт скоупа на один экран не определён | явный рецепт: пиновать `(scr:Screen {id})`-якорь; для orphan-чеков — фильтр по `-{Screen}-` id-инфиксу |

Дефект №2 — самый показательный: словарь видов состояний казался полным, пока первый же
не-CRUD-экран не заставил агента сплющить три семантически разных стадии конвейера в одно
«loading». Словари проверяются не ревью, а чужим использованием.

**Честная остановка №2.** После фиксов словаря бенчмарк перепрогнан с нуля — чтобы эталонные
результаты соответствовали отгруженным запросам, а не их вчерашней версии. Цифры не изменились
(в канонической машине нет busy-состояний), но «не изменились» — это результат прогона, а не
предположение.

## 6. Сквозные уроки

- **Хаб первым, валидаторы — самые сложные.** Порядок фаз — это управление риском: Slice/Error/Cache
  присоединятся к уже существующим и уже валидируемым состояниям экрана.
- **Reified-узел вместо ребра — когда сущности нужен id, исходящие рёбра или orphan-чек.**
  Три критерия, по которым переход стал узлом. Они переносимы на любую будущую схему.
- **Required-мост важнее самого артефакта.** Стейт-машина без `RENDERS` — корректная машина,
  невидимая для изменений. Достижимость от домена — часть определения артефакта, а не бонус.
- **Каждая фаза делает широкий штамп опаснее** (20× → 49×). Дисциплина «направленный гейт-штамп»
  — не разовый фикс Фазы 0, а инвариант роадмапа.
- **Детектор обязан доказать, что детектирует.** Defect-injection с проверкой cross-talk поймал
  не дефект кода, а дефект *модели автора* — тоже результат.
- **Словари и архетипы валидируются чужим использованием.** Один прогон независимого агента на
  неудобном (не-CRUD) примере дал больше правок текста, чем все самопроверки.

## 7. Что мы намеренно НЕ сделали (и почему это правильно)

- **Не расширили словарь состояний «на вырост».** `busy` добавлен по измеренной боли конкретного
  экрана, а не из фантазии о будущих. Следующий kind появится тем же путём — через чужой прогон.
- **Не ввели реверс-рёбра и не переименовали разделяемые типы.** Обход без стрелок +
  label-qualified запросы дешевле, чем вторая копия рёбер или каскадное переименование BA-слоя.
- **Не сделали Decision-гейт для authoring экранов.** Стейт-машина, созданная в составе фичи,
  получает provenance через FR-якорь Фазы 0; отдельный гейт на standalone-authoring добавил бы
  трение без измеренной пользы.
- **Не пушим.** 14 коммитов Фаз 0–1 лежат на локальном `main`; роадмап уезжает одним релизом.

## 8. Итог

Фаза 1 — первый тест фундамента Фазы 0 на настоящем расширении схемы: шесть новых типов узлов,
двенадцать рёбер, девять валидаторов. Тест пройден: изменение UC/атрибута/эндпоинта дотягивается
до экрана (0 → 13/13 узлов машины), гейт-штамп остался точечным (~49× против широкого), все восемь
классов поломок машины ловятся статически, а независимый агент построил по тексту скилла машину
для экрана, под который текст не писался — и его журнал неоднозначностей превратился в четыре
фикса. Хаб готов; Фаза 2 (Behavior Slice) будет крепиться `COVERS`-рёбрами к состояниям и
переходам, которые уже умеют не теряться при изменениях.
