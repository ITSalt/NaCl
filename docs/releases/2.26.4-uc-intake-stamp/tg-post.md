NaCl 2.26.4 — uc-intake-stamp

Один дефект контура сверки кондуктора — «мёртвый гейт» P-S6. UseCase-двойник фикса Task.intake_id из 2.26.3: воспроизведён RED на одноразовом Neo4j до фикса, закрыт GREEN после.

Что было. Фаза 4.5 кондуктора (nacl-tl-conductor) перед объявлением CONDUCTOR COMPLETE прогоняет шесть парных проверок согласованности артефактов. Пара P-S6 — про устаревание: «в UC-замыкании текущего интейка нет ни одного узла с review_status='stale'», то есть наверху (UC / сущность / эндпоинт) ничего не менялось без пере-синхронизации зависимых. Якорь ассерта — MATCH (uc:UseCase {intake_id:$intake})… Но intake_id на UseCase не проставлял ни один скилл: фикс 2.26.3 (DEF-B) сознательно ограничился Task.intake_id. Якорное множество всегда было пустым, count(n)=0 — тривиально истинным, и гейт проходил на любом батче, даже с реально устаревшим и не пере-синхронизированным потомком. Гейт, который не может сработать, — мёртвый гейт.

Что теперь. Штамп ложится в ТОТ ЖЕ оператор Step 2.4 nacl-tl-plan, который уже стампит Task.intake_id и уже матчит исходный UC (для ребра GENERATES и снятия staleness): одна добавленная строка `SET uc.intake_id = coalesce($intakeId, uc.intake_id)`. Опционально и null-безопасно: одиночный nacl-tl-plan передаёт null, coalesce не затирает прежнее значение. Оба стампа (Task и UC) используют ОДНУ привязку $intakeId в одном операторе — поэтому существующая пост-write проверка Step 2.4b по задачам транзитивно доказывает и UC-штамп, отдельная перечитка не нужна. Каждый UC, породивший задачу батча, получает штамп — а это ровно то «UC-замыкание интейка», по которому рассуждает P-S6. Сам P-S6 и nacl-tl-conductor не тронуты: гейт был написан правильно, ему не хватало только штампа, который он потребляет.

Почему tl-plan, а не sa-feature. Очевидный кандидат — штамповать UC там, где они создаются (sa-feature), — не работает по таймингу: sa-feature выполняется ДО кондуктора (tl-intake → sa-feature создаёт FR+UC → кондуктор стартует и чеканит intake_id в фазе 0). В момент создания UC интейка ещё не существует и разрешать $intakeId не из чего. tl-plan — первый и единственный скилл, который одновременно работает под кондуктором с --intake и трогает каждый UC батча.

Доказательства. tests/graph/regression-nav-actions-intake.sh (одноразовый Docker-Neo4j, Cypher извлекается из отгружаемого текста скилла в момент прогона) получил два кейса: uc-intake-stamp — RED на базовом дереве (uc.intake_id остаётся null после прогона Step 2.4), GREEN после фикса; uc-intake-preserve — null-$intakeId не затирает прежнее значение. Все пять прежних кейсов (nav-actions матрица + Task-пара DEF-B) зелёные на обоих деревьях. End-to-end дословным ассертом P-S6: стампованный UC со stale-потомком → count(n)=0 = FALSE (гейт срабатывает); нестампованный с тем же потомком → TRUE (тот самый вакуумный проход — дефект, показанный вживую); стампованный UC с пере-синхронизированным потомком → TRUE (чистый батч проходит).

Совместимость: поведенческая коррекция, схема и wire-формат не менялись. $intakeId опционален — одиночное планирование не затронуто. Охват тот же, что у DEF-B: путь feature/tl-plan; задачи багфиксов (tl-fix) intake_id пока не несут — отдельный отслеживаемый follow-up.
