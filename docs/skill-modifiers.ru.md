# Справочник модификаторов скиллов

Модификаторы — это аргументы, которые меняют поведение скилла при вызове. Это флаги (`--deploy`), режимы (`IMPORT_BA`), подкоманды (`sync`) и позиционные идентификаторы (`UC028`).

Данный документ — **центральный справочник** всех модификаторов по всем NaCl-скиллам.

---

## Конвенции

NaCl-скиллы используют три парадигмы вызова. Каждая подходит для определённого типа скиллов.

| Парадигма | Когда использовать | Синтаксис | Пример |
|-----------|-------------------|-----------|--------|
| **Режим** (позиционный) | Скилл имеет 2-5 взаимоисключающих ветвей workflow | `/skill MODE` | `/nacl-sa-domain IMPORT_BA` |
| **Подкоманда** | Скилл объединяет 3+ различных операций | `/skill command [args]` | `/nacl-ba-from-board sync` |
| **Флаг** | Опциональные модификаторы поведения | `/skill [target] --flag [--key value]` | `/nacl-tl-ship UC028 --deploy` |

### Правила именования

- **Флаги:** kebab-case с двойным дефисом: `--skip-verify`, `--dry-run`, `--auto-ship`
- **Значения режимов:** UPPER_CASE для CRUD-подобных ветвей: `FULL`, `CREATE`, `MODIFY`, `IMPORT_BA`. Ни��ний регистр для утилитарных режимов: `full`, `module`, `stats-only`, `internal`
- **Подкоманды:** нижний регистр: `sync`, `verify`, `stories`, `docmost`
- **Позиционные идентификаторы:** формат ID слоя: `UC028`, `FR-001`, `TECH003`, `ELE-644`

## Универсальные семейства флагов

Эти флаги встречаются в нескольких TL-скиллах. При создании нового скилла используйте существующие, прежде чем изобретать новые.

### Фильтры типа задач

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--be` | Только backend-задачи | `nacl-tl-review`, `nacl-tl-next`, `nacl-tl-status` |
| `--fe` | Только frontend-задачи | `nacl-tl-review`, `nacl-tl-next`, `nacl-tl-status` |
| `--tech` | Только инфраструктурные/TECH задачи | `nacl-tl-next`, `nacl-tl-status` |

### Фильтры фазы workflow

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--review` | Только задачи на ревью | `nacl-tl-next` |
| `--sync` | Только задачи на синхронизацию | `nacl-tl-next` |
| `--qa` | Только задачи на QA | `nacl-tl-next` |

### Селекторы области действия

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--wave N` | Ограничить волной N | `nacl-tl-next`, `nacl-tl-full` |
| `--feature FR-NNN` | Ограничить feature request | `nacl-tl-ship`, `nacl-tl-plan`, `nacl-tl-full`, `nacl-tl-conductor`, `nacl-tl-deliver` |
| `--task UC###` или `CODE` | Ограничить одной задачей | `nacl-tl-full`, `nacl-tl-reopened`, `nacl-tl-verify`, `nacl-tl-verify-code` |
| `--pr N,N` | Конкретные номера PR | `nacl-tl-release` |
| `--items X,Y,Z` | Список элементов через запятую | `nacl-tl-conductor` |
| `--branch <name>` | Явное имя ветки | `nacl-tl-conductor`, `nacl-tl-deliver` |
| `--all` | Обработать все доступные элементы | `nacl-tl-verify`, `nacl-tl-reopened` |

### Флаги пропуска

Пропуск именованной фазы workflow. Паттерн: `--skip-{фаза}`.

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--skip-verify` | Пропустить верификацию на staging | `nacl-tl-deliver` |
| `--skip-deploy` | Пропустить health check / деплой | `nacl-tl-deliver` |
| `--skip-plan` | Пропустить планирование | `nacl-tl-full` |
| `--skip-qa` | Пропустить E2E QA тестирование | `nacl-tl-full`, `nacl-tl-conductor` |
| `--skip-merge` | Пропустить мерж PR | `nacl-tl-release` |
| `--skip-deliver` | Пропустить шаг доставки | `nacl-tl-conductor` |

### Управление подтверждениями

Эти флаги — **семантические противоположности** по замыслу:

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--yes` | Пропустить подтверждения (продолжить автоматически) | `nacl-tl-full`, `nacl-tl-conductor`, `nacl-tl-reopened`, `nacl-tl-hotfix`, `nacl-tl-release` |
| `--confirm` | Добавить подтверждения (требовать явного одобрения) | `nacl-tl-fix` |

### Управление безопасностью

| Флаг | Значение | Уровень риска | Используется в |
|------|----------|---------------|----------------|
| `--dry-run` | Показать план без выполнения | Безопасно | `nacl-tl-fix`, `nacl-tl-dev`, `nacl-tl-dev-be`, `nacl-tl-dev-fe`, `nacl-tl-hotfix`, `nacl-tl-release`, `nacl-tl-reopened`, `nacl-tl-reconcile`, `nacl-init` |
| `--force` | Обойти защиты / пропустить подтверждения | Средний | `nacl-tl-reconcile` |
| `--force-push` | Пуш напрямую в main (без PR) | Высокий | `nacl-tl-hotfix` |

### Автоцепочки

Автоматически запуск следующего скилла в пайплайне после завершения.

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--deploy` | Запустить деплой после отправки | `nacl-tl-ship` |
| `--auto-ship` | Запустить отправку после исправления | `nacl-tl-fix`, `nacl-tl-reopened` |

### Версионирование

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--major` | Мажорное обновление версии | `nacl-tl-release` |
| `--minor` | Минорное обновление версии | `nacl-tl-release` |
| `--patch` | Патч-обновление версии | `nacl-tl-release` |

### Управление выводом

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--compact` | Двухстрочная сводка вместо полного отчёта | `nacl-tl-status` |
| `--list` | Показать топ кандидатов с оценками | `nacl-tl-next` |
| `--final` | Полное сканирование всей директории `src/` | `nacl-tl-stubs` |
| `--output <path>` | Записать в файл вместо терминала | `nacl-render` |

### Окружение

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--env staging\|production` | Целевое окружение деплоя | `nacl-tl-deliver` |

### Процесс разработки

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--continue` | Доработка после ревью (читает review.md) | `nacl-tl-dev`, `nacl-tl-dev-be`, `nacl-tl-dev-fe` |

### Специфичные для hotfix

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--apply` | Stash незакоммиченных изменений для hotfix | `nacl-tl-hotfix` |
| `--cherry-pick <commit\|HEAD>` | Применить существующий коммит к hotfix-ветке | `nacl-tl-hotfix` |
| `--rebase-feature` | После hotfix — ребейз исходной ветки от main | `nacl-tl-hotfix` |

### Классификация исправлений

| Флаг | Значение | Используется в |
|------|----------|----------------|
| `--l1` | Принудительная классификация L1 (только код, пропустить доки) | `nacl-tl-fix` |

## Справочник модификаторов по скиллам

### BA-скиллы

#### nacl-ba-entities

**Парадигма:** Режим (ключ=значение)

| Вызов | Режим | Описание |
|-------|-------|----------|
| `/nacl-ba-entities mode=FULL` | FULL | Интерактивное описание всех сущностей проекта |
| `/nacl-ba-entities mode=CREATE` | CREATE | Добавить одну новую сущность |
| `/nacl-ba-entities mode=MODIFY` | MODIFY | Изменить существующую сущность с анализом влияния |
| `/nacl-ba-entities mode=COLLECT` | COLLECT | Полуавтоматический сбор сущностей из шагов workflow |

#### nacl-ba-rules

**Парадигма:** Режим (ключ=значение)

| Вызов | Режим | Описание |
|-------|-------|----------|
| `/nacl-ba-rules scope=full` | full | Полный каталог: извлечение + классификация + трассируемость |
| `/nacl-ba-rules scope=add` | add | Добавить одно правило интерактивно |

#### nacl-ba-from-board

**Парадигма:** Подкоманда

| Вызов | Команда | Описание |
|-------|---------|----------|
| `/nacl-ba-from-board new <имя_проекта>` | new | Создать новую пустую доску |
| `/nacl-ba-from-board import <путь_к_файлу>` | import | Импортировать документ клиента на доску |
| `/nacl-ba-from-board analyze [путь_к_доске]` | analyze | Анализ полноты доски |
| `/nacl-ba-from-board sync [путь_к_доске]` | sync | Синхронизация элементов доски в Neo4j |
| `/nacl-ba-from-board status [путь_к_доске]` | status | Показать статус синхронизации |
| `/nacl-ba-from-board enrich [путь_к_доске]` | enrich | Обогатить сущностями/ролями/правилами |
| `/nacl-ba-from-board validate [путь_к_доске]` | validate | Запустить проверки валидации |
| `/nacl-ba-from-board handoff [путь_к_доске]` | handoff | Создать пакет передачи BA-в-SA |
| `/nacl-ba-from-board full <путь_к_файлу>` | full | Полный пайплайн |

#### nacl-ba-validate

**Парадигма:** Режим (позиционный)

| Вызов | Режим | Описание |
|-------|-------|----------|
| `/nacl-ba-validate` | full | Все уровни: L1-L8 + XL1-XL5 (по умолчанию) |
| `/nacl-ba-validate internal` | internal | L1-L8: только внутренние проверки BA |
| `/nacl-ba-validate cross` | cross | XL1-XL5: только кросс-валидация BA-SA |

#### Остальные BA-скиллы

`nacl-ba-context`, `nacl-ba-process`, `nacl-ba-workflow`, `nacl-ba-roles`, `nacl-ba-glossary`, `nacl-ba-handoff`, `nacl-ba-import-doc`, `nacl-ba-analyze`, `nacl-ba-sync`, `nacl-ba-full` — без пользовательских модификаторов.

### SA-скиллы

#### nacl-sa-domain

**Парадигма:** Режим (позиционный)

| Вызов | Режим | Описание |
|-------|-------|----------|
| `/nacl-sa-domain IMPORT_BA` | IMPORT_BA | Импорт BA-сущностей как кандидатов в DomainEntity |
| `/nacl-sa-domain CREATE <имя>` | CREATE | Создать новую DomainEntity интерактивно |
| `/nacl-sa-domain MODIFY <имя>` | MODIFY | Изменить существующую DomainEntity |
| `/nacl-sa-domain FULL <модуль>` | FULL | Создать полную доменную модель для модуля |

#### nacl-sa-roles

**Парадигма:** Режим (позиционный)

| Вызов | Режим | Описание |
|-------|-------|----------|
| `/nacl-sa-roles IMPORT_BA` | IMPORT_BA | Импорт BA-ролей как кандидатов в SystemRole |
| `/nacl-sa-roles CREATE <имя>` | CREATE | Создать новую SystemRole интерактивно |
| `/nacl-sa-roles MODIFY <имя>` | MODIFY | Изменить существующую SystemRole |
| `/nacl-sa-roles FULL` | FULL | Построить полную модель ролей |

#### nacl-sa-uc

**Парадигма:** Подкоманда

| Вызов | Команда | Описание |
|-------|---------|----------|
| `/nacl-sa-uc stories` | stories | Создать реестр UC из scope автоматизации BA |
| `/nacl-sa-uc detail <UC-ID>` | detail | Детализировать UC (activity, формы, требования) |
| `/nacl-sa-uc list` | list | Показать все UC |

#### nacl-sa-ui

**Парадигма:** Подкоманда

| Вызов | Команда | Описание |
|-------|---------|----------|
| `/nacl-sa-ui verify [модуль]` | verify | Проверить маппинг форм-домен |
| `/nacl-sa-ui components [модуль]` | components | Определить общие компоненты |
| `/nacl-sa-ui navigation` | navigation | Определить навигационную структуру |
| `/nacl-sa-ui full [модуль]` | full | Запустить все фазы |

#### nacl-sa-validate

**Парадигма:** Режим (позиционный) + Флаг

| Вызов | Описание |
|-------|----------|
| `/nacl-sa-validate` | Полная валидация (все уровни, по умолчанию) |
| `/nacl-sa-validate internal` | Только L1-L6 внутренние п��оверки SA |
| `/nacl-sa-validate ba-cross` | Только XL6-XL9 кросс-валидация BA-SA |
| `/nacl-sa-validate --scope intra-uc UC-NNN[,UC-NNN]` | Ограничить конкретными UC |
| `/nacl-sa-validate --scope intra-module mod-xxx` | Ограничить конкретным модулем |

#### nacl-sa-finalize

**Парадигма:** Режим (позиционный)

| Вызов | Режим | Описание |
|-------|-------|----------|
| `/nacl-sa-finalize` | full | Полная финализация (по умолчанию): все пять фаз |
| `/nacl-sa-finalize module <module_id>` | module | Финализация одного модуля |
| `/nacl-sa-finalize stats-only` | stats-only | Только статистика и готовность (без ADR, без глоссария) |

#### Остальные SA-скиллы

`nacl-sa-architect`, `nacl-sa-feature`, `nacl-sa-full` — без пользовательских модификаторов помимо позиционного описания.

### TL-скиллы

#### nacl-tl-plan

| Флаг | Описание |
|------|----------|
| `scope=full` | Полное планирование (по умолчанию) |
| `scope=module:<id>` | Планировать только для конкретного модуля |
| `scope=uc:<id1>,<id2>` | Планировать только для конкретных UC |
| `--feature FR-NNN` | Планировать только UC из feature request FR-NNN |
| `wave-start=N` | Начальный номер волны (по умолчанию: 0) |

#### nacl-tl-next

| Флаг | Описание |
|------|----------|
| `--be` | Только BE-задачи |
| `--fe` | Только FE-задачи |
| `--tech` | Только TECH-задачи |
| `--review` | Только задачи на ревью |
| `--sync` | Только задачи на синхронизацию |
| `--qa` | Только задачи на QA |
| `--wave N` | Только задачи из волны N |
| `--list` | Показать топ-5 кандидатов с оценками |

#### nacl-tl-status

| Флаг | Описание |
|------|----------|
| `--waves` | Показать только прогресс по волнам |
| `--be` | Показать только BE-прогресс (dev + review фазы) |
| `--fe` | Показать только FE-прогресс (dev + review фазы) |
| `--tech` | Показать только секцию TECH-задач |
| `--stubs` | Подробный отчёт по stubs (по severity, маркеру, задаче) |
| `--qa` | Результаты QA по задачам (pass/fail/pending) |
| `--blocked` | Показать блокеры с цепочками зависимостей |

## Заметки по рационализации

### Текущие несоответствия

Задокументировано для информации. **Изменений в текущем релизе не планируется** — рационализация является отдельной задачей.

1. **Синтаксис параметра режима различается:**
   - `mode=FULL` (nacl-ba-entities) — ключ=значение
   - `scope=full` (nacl-ba-rules) — ключ=значение, другое имя ключа
   - `IMPORT_BA` (nacl-sa-domain) — позиционный
   - Рекомендация для новых скиллов: использовать **позиционный**.

2. **Регистр значений режимов различается:**
   - UPPER_CASE: `FULL`, `CREATE`, `MODIFY`, `IMPORT_BA` (sa-domain, sa-roles, ba-entities)
   - lowercase: `full`, `add` (ba-rules), `full`, `module`, `stats-only` (sa-finalize)
   - Конвенция: UPPER_CASE для CRUD-подобных ветвей, lowercase для утилитарных режимов.

3. **Синтаксис `--scope`:**
   - `--scope intra-uc UC-NNN` (nacl-sa-validate) — через флаг
   - Рекомендация для новых скиллов: использовать **синтаксис флага** (`--scope`).

### Проектные решения (намеренные, не баги)

- **`--yes` и `--confirm`** — семантические противоположности: `--yes` убирает подтверждения, `--confirm` добавляет. Это намеренно.
- **Три парадигмы сосуществуют**, потому что обслуживают разные типы скиллов: режим для моделирования, подкоманда для многофункциональных инструментов, флаг для жизненного цикла.
- **`full` присутствует и в подкомандах, и в режимах** — семантически одинаково (выполнить всё), парадигма различается по типу скилла.
