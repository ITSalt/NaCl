# Сценарии

Готовые end-to-end сценарии с копируемыми командами. Команды даны в форме `/nacl-*`, как в CLI-установке; в канале плагина Claude Code Desktop те же скиллы называются `/nacl:<name>` (см. [Архитектура агентов > Установка](agents.ru.md#установка)).

## 1. Новый проект с нуля

**Когда**: начинаете проект, ничего нет.

```
/nacl-init "Название проекта"     # CLAUDE.md + config.yaml + Docker
/nacl-ba-full                        # бизнес-анализ → Neo4j граф
/nacl-sa-full                        # системная спецификация → Neo4j граф
/nacl-tl-conductor                   # планирование → разработка → QA → staging
```

`nacl-ba-full` проведёт через 10 фаз интерактивно: контекст, процессы, workflows, сущности, роли, глоссарий, правила, валидация, handoff.

`nacl-sa-full` создаст архитектуру, доменную модель, use cases, роли, UI, валидацию.

`nacl-tl-conductor` создаст задачи из графа, разработает код (TDD), проведёт ревью и QA.

## 2. Добавить фичу

**Когда**: проект существует, нужна новая функциональность.

```
/nacl-sa-feature "описание фичи"     # анализ влияния через Cypher
/nacl-tl-conductor --items FR-001    # разработка + QA + деплой
```

`nacl-sa-feature` проверяет влияние на существующие модули, домен и UC через граф.

## 3. Исправить баг

**Когда**: что-то сломалось.

```
/nacl-tl-fix "описание проблемы"
```

Автоматически определяет затронутые UC/TECH, классифицирует уровень (L0-L3), обновляет документацию ПЕРЕД кодом.

## 4. Пакетная обработка запросов

**Когда**: накопилось несколько запросов (фичи + баги + задачи).

```
/nacl-tl-intake                      # триаж: фичи → /nacl-sa-feature, баги → /nacl-tl-fix
/nacl-tl-conductor --items FR-001,FR-002,BUG-003
```

## 5. Импорт документа клиента

**Когда**: клиент прислал описание бизнес-процессов.

```
/nacl-ba-from-board import /path/to/document.docx
/nacl-sa-full
/nacl-tl-conductor
```

Парсит документ → создаёт Excalidraw-борд → синхронизирует в Neo4j-граф.

## 6. Доставка на staging

**Когда**: код готов, нужно задеплоить.

```
/nacl-tl-deliver
```

Push → PR → ожидание CI → верификация на staging (E2E) → health check.

## 7. Релиз в production

**Когда**: staging проверен, готов к релизу.

```
/nacl-tl-release
```

Полный пайплайн релиза. При `feature-branch` стратегии: собирает верифицированные PR (из YouGile "ToRelease" или GitHub), мержит их в `main` с подтверждением пользователя, ждёт CI на production, проверяет health check, затем bump версии, git tag, release notes и уведомление YouGile. При `direct` стратегии шаги мержа и деплоя пропускаются автоматически.

Мерж конкретных PR:
```
/nacl-tl-release --pr 42,45
```

Режим «только тег» не требует флага: если discovery не находит верифицированных
PR для мержа (код уже смержен вручную или через `git.strategy: direct`), скилл
сам пропускает шаги мержа и переходит к тегу и релизу. Флаг `--skip-merge`
удалён; массовые обходы — только через подписанные exceptions или emergency-режим.

## 8. Восстановление

**Когда**: всё сломано, доки устарели.

```
/nacl-tl-diagnose         # анализ git-истории, drift доков, здоровье кода
/nacl-tl-reconcile        # приводит доки в соответствие с кодом
```

## 9. Статус и планирование

```
/nacl-tl-status     # прогресс проекта из Neo4j + status.json
/nacl-tl-next       # рекомендация: что делать дальше
```

## 10. Экстренный хотфикс

**Когда**: критичный баг в production, вы на фича-ветке и нужно доставить фикс в `main` без мержа всей фичи.

```
/nacl-tl-fix "описание проблемы"
```
Исправляет баг на текущей ветке (spec-first, как всегда).

```
/nacl-tl-hotfix --apply
```
Создаёт `hotfix/`-ветку от `main`, переносит незакоммиченные изменения, открывает PR с auto-merge. После мержа возвращает на фича-ветку.

Если фикс уже закоммичен на фича-ветке:
```
/nacl-tl-hotfix --cherry-pick HEAD
```
Cherry-pick коммита на `hotfix/`-ветку от `main`, PR.

## 11. Рабочий процесс на основе целей

**Когда**: длинный детерминированный цикл с верифицируемым финишем — дренаж волны,
исправление отслеживаемого бага, валидация модуля — и вы хотите, чтобы он шёл без
ручных подтверждений между ходами. Используйте `/nacl-goal` вместо `/goal` напрямую:
он соблюдает ворота NaCl и записывает структурированный run-файл.

**Используйте `/nacl-goal`, когда:**
- Задача соответствует встроенному алиасу (`wave`, `fix`, `validate`, `reopened-drain`, `intake`, `conduct`).
- Финиш объективно верифицируется скриптом, а не человеческим суждением.
- Вы готовы оставить сессию на Tier S (до 2 ч) или Tier M (до 6 ч).

**Не используйте `/nacl-goal` для:**
- Всего, что пересекает BA-SA handoff, подтверждение фазы SA или маршрутизацию hotfix.
  Это обязательные ворота ручного подтверждения — `/nacl-goal` отказывает по коду.
- Tier L / XL без присмотра на ночь (до появления данных калибровки в 2.10.2).

```
/nacl-goal wave:5             # предпросмотр: tier, бюджет, скрипт, ворота, denylist
/nacl-goal wave:5 --start     # запускает /goal с инструкцией GOAL_PROOF
```

После завершения проверьте `.tl/goal-runs/<run_id>.md` — там находится
машиночитаемая запись, включая результат повторной проверки после завершения.

Полная документация: [docs/guides/goal-command.md](guides/goal-command.md)
