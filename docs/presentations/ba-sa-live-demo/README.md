# Полный цикл разработки через ИИ — Миф или быль? (ЛАФ 2026)

> `A1-FIRST PRACTICE` · доклад Максима Никитина на ЛАФ 2026.
> Финальная версия как её показывали — [`full-ai-dev-cycle-laf2026.pdf`](full-ai-dev-cycle-laf2026.pdf)
> (62 слайда). Этот каталог — воспроизводимый Markdown-исходник того же доклада.

Материалы выступления (90–120 мин) о скиллах бизнес-аналитика (`/nacl-ba-full`) и
системного аналитика (`/nacl-sa-full`) с **живой демонстрацией**: параллельно с
рассказом в соседнем терминале агент строит спецификацию мини-проекта
**«СТО: онлайн-запись на обслуживание автомобиля»** в Neo4j-графе — от интервью о
границах системы до спецификации, готовой к разработке.

## Структура

```
ba-sa-live-demo/
├── README.md                       # этот файл
├── full-ai-dev-cycle-laf2026.pdf   # финальный показанный PDF (62 слайда)
├── demo-script.md                  # операторский ранбук: точные входы/выходы каждой фазы
├── build-deck.sh                   # сборка slides/*.md → deck.md (Marp)
├── deck.md / deck.html             # собранный Marp-deck (генерируются из slides/)
├── assets/                         # пре-рендеренные mermaid-диаграммы (генерируются)
└── slides/                         # 62 слайда, по одному .md на слайд
```

Слайды-исходники соответствуют 62 страницам финального PDF 1:1. Разделители секций
(`07a`, `27a`, `49a`) и слайд экономики (`03a`) вставлены буквенными суффиксами,
чтобы не перенумеровывать остальные файлы.

## Как собрать слайды

```bash
cd docs/presentations/ba-sa-live-demo
bash build-deck.sh                                  # slides/*.md → deck.md
npx @marp-team/marp-cli deck.md -o deck.html        # рендер в HTML
npx @marp-team/marp-cli deck.md --pdf -o deck.pdf   # или в PDF
```

Mermaid-диаграммы: если установлен `mmdc` (`npm i -g @mermaid-js/mermaid-cli`),
`build-deck.sh` пре-рендерит их в `assets/*.svg` и подставит картинки в deck.md.
Без `mmdc` блоки останутся кодом — слайды всё равно собираются (GitHub и VS Code
рендерят mermaid нативно).

## Правила оформления слайдов

Выведены из авторского ревью — соблюдать при любых правках:

1. **Понятия вводятся до первого использования** (BA/SA/TL-слой, Cypher, MCP,
   скилл, токен, FSM, ADR…) — отдельным слайдом или пояснением на месте.
2. **Цифры — с честной базой сравнения:** что именно меряется и почему
   сравнение корректно (пример: ~150 000 токенов = чтение всей спеки,
   которое в классике приходится платить для планирования даже одного UC).
3. **Сильные утверждения — только с механизмом:** не «агент не выдумывает»,
   а конструкции, которые это обеспечивают.
4. **Английские термины — с русским переводом**; английское имя сохраняем
   (оно же — метка в графе).
5. **После каждого «агент записал в граф» — понятно, как посмотреть:**
   слайд 08a + verify-запросы в demo-script.md и в speaker notes
   («Где посмотреть: …»).
6. **Каждый символ в таблицах объяснён** (✔/—, O/P, CRUD…) — легенда под
   таблицей или в speaker notes.

## Канон демо-домена

Все примеры на слайдах и все реплики в `demo-script.md` используют **одни и те же
идентификаторы** (BP-001, OBJ-001, UC-001, FORM-BookingCreate, ERR-SlotTaken…).
Канон описан в начале `demo-script.md`. Если меняете домен — меняйте синхронно
слайды-примеры и ранбук.
