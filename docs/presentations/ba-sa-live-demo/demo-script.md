# Demo-script: операторский ранбук живой демонстрации

Пошаговый сценарий прогона `/nacl-ba-full` → `/nacl-sa-full` на демо-домене
**«СТО: онлайн-запись»** — с дословными ответами агенту, ожидаемыми выходами
и verify-запросами для Neo4j Browser. Цель: прогон воспроизводим — на репетиции
и на сцене вы отвечаете одно и то же.

---

## 0. Канон демо-домена (единый для слайдов и ранбука)

| Слой | Идентификаторы |
|------|----------------|
| Контекст | SYS-001 «СТО-Запись»; STK-01 владелец, STK-02 администратор; EXT-01 Клиент, EXT-02 СМС-шлюз |
| Процессы | GPR-01 «Запись на обслуживание»: BP-001 «Приём заявки на запись» (декомп.), BP-002 «Подтверждение записи»; GPR-02 «Выполнение обслуживания»: BP-003 «Выполнение работ» (декомп.), BP-004 «Выдача автомобиля и оплата» |
| Сущности | OBJ-001 Заявка на запись (новая→подтверждена→отменена), OBJ-002 Клиент, OBJ-003 Автомобиль, OBJ-004 Заказ-наряд (открыт→в работе→завершён→оплачен), OBJ-005 Слот расписания |
| Роли | ROL-01 Клиент, ROL-02 Администратор, ROL-03 Мастер |
| Правила | BRQ-001 занятый слот, BRQ-002 стоимость = работы+запчасти, BRQ-003 выдача после оплаты |
| SA | mod-001 Booking, mod-002 Workshop; UC-001…UC-004; DE-Booking, DE-Client, DE-Vehicle, DE-ScheduleSlot, DE-WorkOrder; FORM-BookingCreate; role-001 CLIENT, role-002 ADMIN, role-003 MECHANIC |
| 6b | SCR-BookingCreate; SLC-UC-001-HappyPath, SLC-UC-001-SlotTaken; ERR-SlotTaken, ERR-SlotsLoadFailed; CACHE-ScheduleSlots, DEG-UC-001-StaleSlots |

> Точные ID, которые присвоит агент, могут отличаться (сквозные счётчики) —
> важна структура и состав, а не цифры. На репетиции зафиксируйте фактические ID.

## 1. Pre-flight

### За день (репетиция + бэкап)

1. Полный прогон по этому ранбуку у себя. Замерьте длительность фаз.
2. **Бэкап-граф:** после успешного прогона снимите дамп:
   ```bash
   docker stop <neo4j-container>
   docker run --rm --volumes-from <neo4j-container> -v "$PWD":/backup alpine \
     tar czf /backup/sto-demo-graph.tar.gz /data
   docker start <neo4j-container>
   ```
   При сбое на сцене: восстановить том → показывать готовый граф и запросы,
   рассказ не останавливается.
3. Проверьте, что `deck.html` собран свежим (`bash build-deck.sh`).

### За час (стенд)

```bash
# 1. Граф пуст и жив
docker compose -f graph-infra/docker-compose.yml up -d
# в Neo4j Browser (http://localhost:3574):
#   MATCH (n) RETURN count(n)        → 0 (если нет — MATCH (n) DETACH DELETE n)

# 2. ПУСТАЯ папка проекта (без инициализации! /nacl-init — живой шаг демо)
mkdir -p ~/projects/sto-demo && cd ~/projects/sto-demo

# 3. MCP отвечает: в Claude Code попросить выполнить
#   mcp__neo4j__read-cypher: RETURN 1            → 1

# 4. SA Analyst Tool запущен и подключён к этому же графу
#    (команда запуска — по локальной установке тула)

# 5. Окна: слайды | терминал Claude Code | Neo4j Browser | SA Analyst Tool
```

### Как смотреть граф (шпаргалка к слайду 08a)

**Neo4j Browser** (`http://localhost:3574`): логин `neo4j`, пароль — значение
`NEO4J_PASSWORD` из `graph-infra/.env`. Вверху строка запроса → вставить
verify-Cypher из этого ранбука → `Ctrl+Enter`. Кружки = узлы, стрелки = рёбра;
клик по узлу показывает свойства; левая панель — все метки (типы узлов).
Полезно заранее: в настройках Browser включить «Connect result nodes».

**SA Analyst Tool**: тот же граф глазами аналитика — процессы, сущности, роли,
сценарии без Cypher. Использовать для «бизнесовых» фаз (2, 4, 5 BA),
Browser — для демонстрации рёбер и трассировок.

## 2. BA-прогон: `/nacl-ba-full`

### Шаг 0 — инициализация живьём (слайд 06)

Первый демонстрируемый шаг — скилл инициализации:

```
cd ~/projects/sto-demo
/nacl-init "СТО Онлайн-запись"
```

**Ожидаемо:** созданы `CLAUDE.md` (правила проекта, маршрутизация скиллов)
и `config.yaml` (порт графа, git-стратегия, команды тестов). Показать оба
файла залу: «всё, что скиллы знают о проекте, живёт здесь».

Запуск `/nacl-ba-full` — на слайде 08. Дальше по фазам: **«Слайды» = что
рассказывать, пока агент работает**.

### Фаза 1 — Контекст (слайды 09–10)

Агент спросит цели, границы, стейкхолдеров, внешние сущности. Ответы:

> **Система:** «СТО-Запись» — онлайн-запись клиентов на обслуживание автомобилей.
> **Цели:** 1) клиент записывается сам, без звонка; 2) администратор управляет
> расписанием и подтверждает записи; 3) работы учитываются через заказ-наряды.
> **Вне границ:** склад запчастей, бухгалтерия, зарплаты.
> **Стейкхолдеры:** владелец СТО — загрузка и выручка; администратор — удобное расписание.
> **Внешние:** Клиент (пользователь), СМС-шлюз (внешняя система уведомлений).

Потоки данных агент предложит сам → подтвердить.

**Ожидаемо:** SystemContext ×1, Stakeholder ×2, ExternalEntity ×2, DataFlow ×3
+ Mermaid-диаграмма. **Gate → yes.**

Verify: `MATCH (s:SystemContext)-[r]->(n) RETURN s,r,n`

### Фаза 2 — Процессы (слайды 11–12)

> Клиент выбирает время и оставляет заявку. Администратор её подтверждает.
> В день визита мастер выполняет работы по заказ-наряду. Затем администратор
> выдаёт автомобиль и принимает оплату.

На вопросы о триггерах/результатах — из канона (см. таблицу слайда 12).
**Декомпозиция: только BP-001 и BP-003.**

**Ожидаемо:** ProcessGroup ×2, BusinessProcess ×4, цепочка TRIGGERS,
предварительные роли ROL-01..03. **Gate → yes.**

Verify: `MATCH (g:ProcessGroup)-[:CONTAINS]->(bp) RETURN g.name, collect(bp.name)`

### Фаза 3 — Workflow (слайды 13–14)

**BP-001:**

> Клиент выбирает услугу и свободный слот, заполняет заявку. Система проверяет,
> что слот ещё свободен, резервирует его и отправляет СМС о приёме заявки.
> Если слот заняли — клиенту предлагаются другие слоты.

Стереотипы: все шаги «Автоматизируется» (подтвердить предложение агента).

**BP-003:**

> Мастер открывает заказ-наряд по подтверждённой записи, выполняет работы,
> вносит перечень работ и запчастей, закрывает наряд со статусом «завершён».

**Ожидаемо:** ~10 WorkflowStep, решение «слот свободен?», рёбра
PERFORMED_BY/READS/PRODUCES. **Gate (по каждому BP) → yes.**

Verify: `MATCH ({id:'BP-001'})-[:HAS_STEP]->(s) RETURN s.id, s.name ORDER BY s.id`

### Фаза 4 — Сущности (слайды 15–16) — С ЗАГОТОВЛЕННОЙ ДЫРКОЙ

Агент соберёт кандидатов из workflow → подтвердить 5 сущностей.

Атрибуты (бизнес-типы!):

> **Заявка на запись:** номер (Число), дата и время слота (Дата), услуга
> (Перечисление: ТО, диагностика, ремонт), комментарий (Текст),
> статус (Перечисление: новая, подтверждена, отменена).
> **Клиент:** ФИО (Текст), телефон (Текст), согласие на СМС (Да/Нет).
> **Заказ-наряд:** номер (Число), перечень работ (Текст), стоимость работ (Число),
> стоимость запчастей (Число), итоговая стоимость (Число),
> статус (Перечисление: открыт, в работе, завершён, оплачен).
> **Слот расписания:** дата и время (Дата), занят (Да/Нет), мастер (Ссылка).

⚠️ **Для OBJ-003 «Автомобиль» атрибуты НЕ давать:** «давайте вернёмся к нему
позже» — это заготовка для фазы 8 (слайд 23).

Состояния: Заявка — новая→подтверждена (администратор подтвердил),
новая→отменена, подтверждена→отменена; Заказ-наряд — по статусам.

**Ожидаемо:** BusinessEntity ×5, ~20 EntityAttribute, EntityState ×7,
classDiagram + CRUD-матрица. **Gate → yes** (дырку агент не видит — атрибуты
«отложены»).

### Фаза 5 — Роли (слайды 17–18)

> Администратор — отдел приёмки: подтверждает записи, управляет расписанием,
> выдаёт автомобили, принимает оплату. Мастер — цех: выполняет работы,
> ведёт заказ-наряды. Клиент — внешняя роль.

**Ожидаемо:** обогащённые ROL-01..03, матрица роль×процесс (слайд 18). **Gate → yes.**

### Фаза 6 — Глоссарий (слайд 19)

Подтверждать определения, предложенные из контекста графа. Один синоним:

> «Запись» и «Заявка на запись» — одно и то же; канонический термин — «Заявка на запись».

**Ожидаемо:** ~12 GlossaryTerm, 1 ALIAS_OF, DEFINES-рёбра. **Gate → yes.**

### Фаза 7 — Правила (слайды 20–21)

> 1) Нельзя записаться на занятый слот. 2) Итоговая стоимость заказ-наряда =
> стоимость работ + стоимость запчастей. 3) Автомобиль выдаётся только после
> полной оплаты заказ-наряда.

Подтвердить классификацию (constraint / calculation / invariant) и привязки.

**Ожидаемо:** BRQ-001..003 с рёбрами по слайду 21. **Gate → yes.**

Verify: `MATCH (r:BusinessRule)-[t]->(x) RETURN r.id, r.rule_type, type(t), x.id`

### Фаза 8 — Валидация (слайды 22–23) — КУЛЬМИНАЦИЯ ДЫРКИ

**Ожидаемо:** `L4 FAIL [CRITICAL]: OBJ-003 — 0 атрибутов` (+возможен WARNING L7).

Починка (агент предложит вернуться к сущностям):

> Атрибуты Автомобиля: гос. номер (Текст), марка и модель (Текст), год выпуска (Число).

Перезапуск валидации → **PASS**. **Gate → yes.**

### Фаза 9 — Handoff (слайды 24–25)

Агент покажет automation scope → подтвердить UC-кандидатов и приоритеты
(UC-001..003 — MVP, UC-004 — Post-MVP), модули Booking/Workshop.

**Это human gate — прокомментировать его смысл залу (слайд 24).** **Gate → yes.**

### Фаза 10 — Публикация (слайд 26)

Для экономии времени: **skip** («без публикации»). Упомянуть словами.
Итог BA — слайд 27 + общий вид графа в Browser:
`MATCH p=()-[]->() RETURN p LIMIT 200`

## 3. SA-прогон: `/nacl-sa-full`

Запуск — на слайде 28. Агент обнаружит BA-слой → подтвердить импорт.

### Фаза 1 — Архитектура (слайды 29–30)

Подтвердить модули из SUGGESTS (Booking, Workshop) и зависимости.
NFR при вопросе:

> СМС о приёме заявки — не позднее 1 минуты после создания.

**Gate → yes.** Verify: `MATCH (m:Module)-[d:DEPENDS_ON]->(m2) RETURN m.id, d.kind, m2.id`

### Фаза 2 — Домен (слайды 31–32)

Режим IMPORT_BA для обоих модулей. Подтвердить предложенные типы
(см. таблицу слайда 32: int/datetime/enum/string). Энумы агент возьмёт
из EntityState — подтвердить.

**Ожидаемо (mod-001):** DE-Booking, DE-Client, DE-Vehicle, DE-ScheduleSlot,
ENUM-BookingStatus, ENUM-ServiceType + рёбра REALIZED_AS/TYPED_AS.
**Gate (по каждому модулю) → yes.**

Verify: `MATCH (o:BusinessEntity)-[:REALIZED_AS]->(d:DomainEntity) RETURN o.name, d.id`

### Фаза 3 — Роли (слайды 33–34)

Подтвердить CLIENT/ADMIN/MECHANIC и CRUD-матрицу (слайд 34; агент выведет
черновик из графа — поправить расхождения с каноном). **Gate → yes.**

### Фаза 4 — UC-реестр (слайды 35–36)

Подтвердить UC-001..004, акторов, приоритеты, user stories, DEPENDS_ON-цепочку.
**Gate → yes.**

Verify: `MATCH (ws:WorkflowStep)-[:AUTOMATES_AS]->(uc:UseCase) RETURN uc.id, collect(ws.id)`

### Фаза 5 — Детализация UC (слайды 37–39)

**Живьём детализируем только UC-001** (см. слайд 39):

- сценарий: подтвердить 6 ActivityStep + альтернативу «слот занят»;
- форма: подтвердить FORM-BookingCreate, 5 полей + submit (action, exempt);
  проверить, что у каждого поля ввода есть MAPS_TO;
- требования: подтвердить RQ из BRQ (атомарная проверка слота, маска телефона,
  СМС после создания);
- **runtime-контракт:** агент сам объявит его обязательным (гонка за слот) —
  подтвердить транзакцию create+reserve, блокировку слота, идемпотентность
  по паре (клиент, слот).

После UC-001 на вопрос «детализировать остальные?» — **«остальные после
доклада, переходим к UI»** (честно прокомментировать залу: Post-MVP UC
останутся WARNING в валидации; UC-002/003 можно детализировать, если
остаётся время).

Verify: `MATCH (f:FormField)-[:MAPS_TO]->(a:DomainAttribute) RETURN f.id, a.id`

### Фаза 6 — UI (слайды 40–41)

Подтвердить компоненты и навигацию; проследить, чтобы для FORM-BookingCreate
появилась входящая точка (кнопка «Записаться» в CMP-MainNav). **Gate → yes.**

### Фаза 6b — Расширения (слайды 42–46)

На вопрос «принять расширения?» — **yes** (default). Делаем все 4 шага
для UC-001:

1. **Screen FSM:** подтвердить машину SCR-BookingCreate по слайду 43
   (5 состояний, выход из error обязателен).
2. **Slices:** подтвердить SLC-UC-001-HappyPath (COVERS success-состояние)
   и SLC-UC-001-SlotTaken (COVERS error-состояние).
3. **Errors:** ERR-SlotTaken (user_fixable, inline) и ERR-SlotsLoadFailed
   (retryable, toast) + MAY_RAISE/PRESENTED_AS/SHOWS.
4. **Resilience:** CACHE-ScheduleSlots (ttl 60 c, инвалидация по событию
   записи) и DEG-UC-001-StaleSlots (ON_ERROR → показать кэш с плашкой).

**Gate после каждого шага → yes.**

### Фаза 7 — Валидация (слайд 47)

**Ожидаемо:** 0 CRITICAL; WARNING на недетализированных UC-002..004 —
прокомментировать как осознанный остаток. **Gate → yes.**

### Фаза 8 — Финализация (слайд 48)

Показать статистику, traceability matrix (4 покрытия), readiness.
Decision-узлы: согласиться зафиксировать решение о двух модулях. **Gate → yes.**

### Фазы 9–10 (слайд 49)

Публикация — skip. `/nacl-tl-plan` — по запасу времени (клиффхэнгер:
показать сгенерированный task-fe.md со встроенной screen machine).

## 5. Сокращённый режим (если отстаёте)

1. BA Ф6 (глоссарий) — подтвердить всё без обсуждения (−3 мин).
2. BA Ф10, SA Ф9 — skip всегда.
3. SA Ф5 — только UC-001 (заложено в сценарий).
4. SA 6b — только шаги 1 (screens) и 3 (errors); slices/resilience — по
   слайдам 44/46 словами (−8 мин). Валидация L11/L13 пройдёт вакуумно —
   честно сказать об этом залу.
5. Полный отказ от живого демо (форс-мажор) — восстановить бэкап-граф,
   показывать verify-запросы из этого ранбука по готовому графу.

## 6. Типовые отклонения

| Симптом | Действие |
|---------|----------|
| Агент придумал лишнюю сущность/шаг | Сказать «этого нет в домене, убери» — на gate любой фазы |
| ID не совпали с каноном слайдов | Не страшно: на слайдах структура; проговорить «у нас BP-00X» |
| Фаза упала / agent error | Оркестратор предложит retry/skip → retry; resume detection продолжит с места |
| MCP/Neo4j отвалился | Перезапустить docker-контейнер; resume по графу |
| Совсем нет времени | Бэкап-граф + слайды-примеры: на каждый вход/выход есть слайд |
