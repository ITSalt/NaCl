# Методология NaCl: три слоя, один граф

NaCl -- это набор из 55 slash-command навыков для Claude Code, покрывающих полный жизненный цикл разработки ПО: от бизнес-анализа через системную спецификацию до TDD-разработки, код-ревью, QA и релиза. Отличие NaCl не в количестве навыков, а в методологии, на которой они построены: структурированном конвейере, где все артефакты бизнес- и системного анализа хранятся в графовой базе данных Neo4j и доступны для запросов как людям, так и AI-агентам.

В этом документе изложены ключевые идеи. Сопутствующие страницы раскрывают каждый слой более детально.

---

## Почему методология важна для AI-агентов

AI-агенты для написания кода, такие как Claude Code, обладают высокой производительностью, но структурно забывчивы. Каждая сессия начинается с чистого контекстного окна. На протяжении длительных проектов -- где бизнес-анализ занимает недели, а реализация растягивается на месяцы -- три проблемы нарастают лавинообразно:

1. **Потеря контекста.** Агент не помнит, какие решения он принял вчера. Каждая сессия требует повторного чтения спецификаций, восстановления соглашений и повторного определения текущего состояния проекта.

2. **Деградация спецификаций.** Традиционные спецификации живут в десятках markdown-файлов. В реальном проекте с 10-15 юзкейсами каталог docs/ легко разрастается до 70+ файлов. Чтобы спланировать один юзкейс, агент должен прочитать значительную часть этих файлов -- карты процессов, каталоги сущностей, матрицы ролей, бизнес-правила, глоссарий, диаграммы рабочих процессов, спецификации UI, доменную модель. Это примерно 150 000 токенов контекста, потраченных до начала какой-либо продуктивной работы.

3. **Отсутствие возможности запросов.** Плоские файлы не имеют структуры помимо заголовков. Невозможно спросить "какие юзкейсы затрагивают сущность Order?" без сканирования каждого файла. Невозможно проверить согласованность ("у каждого бизнес-процесса назначена хотя бы одна роль?") без написания хрупких regex-скриптов. Невозможно проследить поле UI-формы до атрибута бизнес-сущности, которую оно представляет.

NaCl решает все три проблемы, кодифицируя методологию в виде исполняемых навыков, опирающихся на графовую базу данных. Вместо markdown-файлов, которые агент читает последовательно, спецификации существуют как типизированные узлы и типизированные рёбра в Neo4j. Агент запрашивает только то, что ему нужно, и тогда, когда это нужно. Валидация -- это набор Cypher-запросов, а не ручной чек-лист. Трассируемость -- это обход графа, а не упражнение по перекрёстным ссылкам.

Результат: агент тратит бюджет контекста на продуктивную работу (генерация кода, запуск тестов, ревью диффов), а не на повторное чтение спецификаций, которые он не способен запомнить.

## Трёхслойный конвейер

NaCl организует работу в три последовательных слоя. Каждый слой имеет собственные навыки, собственный язык вывода и собственный раздел графа.

- **BA (Business Analysis)** -- 14 навыков. Язык вывода: русский (для заказчиков и доменных экспертов). Производит: системный контекст, бизнес-процессы, рабочие процессы, сущности, роли, правила и глоссарий. Всё хранится как узлы Neo4j.

- **SA (System Analysis)** -- 9 навыков. Язык вывода: русский (для технических заказчиков). Производит: модули (ограниченные контексты), доменную модель, юзкейсы с диаграммами активностей, формы, UI-компоненты, системные роли, требования. Также хранится как узлы Neo4j.

- **TL (TeamLead)** -- 25 навыков. Язык вывода: английский (для разработчиков). Потребляет SA-граф и производит файлы задач, код (через TDD), код-ревью, QA-отчёты, коммиты и релизы. Артефакты живут в файловой системе и Git.

Разделение по языкам сделано намеренно. Документы анализа читают бизнес-заказчики, работающие на русском языке. Код, файлы задач, сообщения коммитов и описания PR читают разработчики, работающие на английском. Промпт каждого навыка написан на языке вывода, поэтому Claude естественным образом создаёт артефакты на нужном языке.

```
BA Layer (14 skills)          SA Layer (9 skills)           TL Layer (25 skills)
---------------------         ---------------------         ---------------------
Stakeholders, processes,      Modules, domain model,        Tasks, waves,
entities, roles, glossary,    use cases, forms, roles,      TDD code, review,
rules, workflows              API contracts, UI             QA, ship, deploy
        |                             |                            |
        v                             v                            v
   Neo4j Graph                   Neo4j Graph                 Files + Git
   (13 node types)               (12 node types)             (.tl/tasks/, src/)
```

Слои строго последовательны. BA должен завершиться прежде, чем может начаться SA, поскольку SA-навыки читают BA-узлы для вывода юзкейсов из бизнес-процессов. SA должен завершиться прежде, чем TL сможет планировать, поскольку `nacl-tl-plan` читает юзкейсы, сущности и зависимости из SA-графа для генерации файлов задач. Межслойные рёбра (такие как `AUTOMATES_AS` от WorkflowStep к UseCase или `GENERATES` от UseCase к Task) делают эти зависимости явными и обходимыми.

## Граф как единственный источник истины

Центральное проектное решение в NaCl -- хранение спецификаций в виде графа, а не документов. Вот почему.

**Почему Neo4j.** Графовая база данных со свойствами предоставляет типизированные узлы (BusinessProcess, UseCase, DomainEntity) с типизированными рёбрами (CONTAINS, AUTOMATES_AS, GENERATES). Такая структура изначально пригодна для запросов. Чтобы найти все юзкейсы, затронутые изменением сущности Order, достаточно одного Cypher-запроса:

```cypher
MATCH (e:DomainEntity {name: "Order"})<-[:CONTAINS_ENTITY]-(m:Module)-[:CONTAINS_UC]->(uc:UseCase)
RETURN uc.id, uc.name
```

С markdown-файлами пришлось бы грепать каждую спецификацию UC в надежде, что имя сущности где-то упоминается в тексте. Граф даёт уверенность; grep даёт надежду.

**Сокращение токенов.** Навык `nacl-tl-plan` выполняет один Cypher-запрос на юзкейс -- примерно 550 токенов (50 на запрос, 500 на ответ с полным контекстом UC: шаги, формы, сущности, зависимости, требования). Альтернатива -- чтение ~70 markdown-файлов, которые порождает традиционная спецификация, -- обходится приблизительно в 150 000 токенов. Это **сокращение на 99,6%** потребления токенов на один юзкейс при планировании. Для агента с конечным контекстным окном это разница между планированием 10 юзкейсов за одну сессию и исчерпанием контекста на втором.

**Что обеспечивает граф:**

- **Анализ влияния.** Изменение DomainEntity -- и немедленный обзор всех UseCase, Form и Requirement, которые на неё ссылаются, по всем модулям.

- **Межслойная трассируемость.** Проследить FormField через его Form к UseCase, через ребро AUTOMATES_AS к WorkflowStep и вверх до BusinessProcess. Каждая связь -- это ребро в графе.

- **Валидация согласованности.** Cypher-запросы выявляют осиротевшие узлы (DomainEntity без модуля), отсутствующие связи (UseCase без актора) и пробелы в покрытии (BusinessProcess без рабочего процесса). NaCl включает 17+ проверок валидации на уровнях BA и SA.

**Определение схемы.** Схема графа определена в трёх файлах в каталоге `graph-infra/schema/`:

| File | Layer | Node types | Purpose |
|------|-------|------------|---------|
| `ba-schema.cypher` | BA | 13 | ProcessGroup, BusinessProcess, WorkflowStep, BusinessEntity, EntityAttribute, EntityState, BusinessRole, BusinessRule, GlossaryTerm, SystemContext, Stakeholder, ExternalEntity, DataFlow |
| `sa-schema.cypher` | SA | 12 | Module, UseCase, ActivityStep, DomainEntity, DomainAttribute, Enumeration, EnumValue, Form, FormField, Requirement, SystemRole, Component |
| `tl-schema.cypher` | TL | 3 | Task, Wave, APIEndpoint |

Каждый файл определяет ограничения уникальности, индексы и документирует типы связей для своего слоя. Межслойные рёбра (BA-to-SA, SA-to-TL) документируются в файле схемы нижележащего слоя.

## Принцип автономии

Навыки NaCl спроектированы для работы совместно с пользователем, а не вместо пользователя. Ключевое правило, сформулированное в `nacl-ba-full`:

> **Факты предоставляет пользователь, конструирование выполняет агент, подтверждение даёт пользователь.**

Этот принцип проводит чёткую границу между тем, что агент может делать автономно, и тем, что требует участия человека.

**Агент ДЕЛАЕТ:**

- Генерирует структурированные идентификаторы (GPR-01, BP-001, UC-101, OBJ-005) по единым соглашениям об именовании
- Создаёт структуру графа -- узлы, рёбра, свойства -- на основе фактов, предоставленных пользователем
- Строит Mermaid-диаграммы и Excalidraw-доски из данных графа
- Вычисляет матрицы трассируемости, статистику покрытия и отчёты валидации
- Предлагает разумные значения по умолчанию и допущения, явно помеченные как таковые

**Агент НЕ ДЕЛАЕТ:**

- Не изобретает бизнес-факты ("системе, вероятно, нужен модуль оплаты")
- Не додумывает детали процессов, которые пользователь не описывал
- Не фабрикует требования или критерии приёмки
- Не пропускает точки подтверждения ради ускорения

**Пользователь ПОДТВЕРЖДАЕТ:**

- Каждую крупную фазу конструирования перед переходом агента к следующей
- Каждый набор допущений, сделанных агентом (помеченных `status: "assumption"` на узлах графа)
- Область автоматизации (какие бизнес-процессы становятся юзкейсами)
- Финальный отчёт валидации перед передачей на следующий слой

Этот принцип распространяется на все навыки работы с графом в слоях BA и SA. Каждый оркестратор (`nacl-ba-full`, `nacl-sa-full`) выполняет 10 фаз, и каждая фаза завершается точкой подтверждения. Пользователь может одобрить, повторить или остановить выполнение на любой точке. Если пользователь не может ответить на все вопросы фазы, агент фиксирует известное, помечает пробелы как допущения и возвращается к ним на этапе валидации (Фаза 8), где Cypher-запросы выявляют неполные данные.

Слой TL более автономен по замыслу -- после подтверждения SA-спецификации навыки разработки (`nacl-tl-dev-be`, `nacl-tl-dev-fe`) выполняют TDD-циклы без запроса разрешения на каждый тест. Но и здесь код-ревью (`nacl-tl-review`) и QA (`nacl-tl-qa`) служат контрольными точками, доступными для инспекции человеком, перед отправкой кода.

## Иерархия оркестрации

Навыки NaCl организованы в четыре уровня абстракции: от оркестраторов полного конвейера до разделяемых справочных библиотек.

**Уровень 4: Оркестратор конвейера.**
`nacl-tl-conductor` управляет всем потоком от приёма задач до деплоя на staging. Он создаёт feature-ветки, делегирует работу оркестраторам слоёв, коммитит каждый завершённый юзкейс атомарно и координирует доставку. Одна ветка на батч, один коммит на UC.

**Уровень 3: Оркестраторы слоёв.**
`nacl-ba-full` (10 фаз), `nacl-sa-full` (10 фаз) и `nacl-tl-full` (жизненный цикл разработки на каждый UC). Каждый оркестратор выстраивает навыки Уровня 2 в правильном порядке, управляет точками подтверждения, отслеживает прогресс и обрабатывает сбои логикой повтора или пропуска.

**Уровень 2: Отдельные навыки.**
48 специализированных навыков, выполняющих фактическую работу: `nacl-ba-context` определяет границы системы, `nacl-tl-dev-be` выполняет backend TDD, `nacl-tl-qa` запускает E2E-тесты через Playwright и так далее. Каждый навык читает из чётко определённого среза графа или файловой системы и записывает в него.

**Уровень 1: Разделяемые справочники.**
`nacl-core` предоставляет соглашения о подключении к Neo4j, правила форматирования идентификаторов, расположение файлов схемы и паттерны Cypher-запросов. `nacl-tl-core` предоставляет шаблоны файлов задач, Git-соглашения и чек-листы код-ревью. Эти навыки не вызываются напрямую -- они читаются другими навыками как справочный материал.

**Паттерн делегирования.** Оркестраторы Уровня 3 запускают навыки Уровня 2 как Task-агентов -- подагентов с изолированными контекстными окнами. Это осознанное проектное решение для управления бюджетом токенов. Полный BA-конвейер порождает сотни узлов графа за 10 фаз. Если бы все фазы выполнялись в одном контекстном окне, агент исчерпал бы бюджет токенов к Фазе 3 или 4. Делегируя каждую фазу подагенту, оркестратор сохраняет собственный контекст компактным (только отслеживание прогресса и управление точками подтверждения), а каждый подагент получает свежее контекстное окно для своей конкретной работы.

**Граф как разделяемая память.** Поскольку каждый подагент записывает в Neo4j независимо, сам граф становится разделяемой памятью между фазами. Фаза 4 (сущности) не нуждается в чтении вывода Фазы 2 (процессы) из контекста -- она запрашивает у графа узлы BusinessProcess и обнаруживает их через Cypher. Именно поэтому подход "граф в первую очередь" -- это не просто выбор хранилища, а архитектурная необходимость для многофазных AI-рабочих процессов.

Этот раздел методологии содержит шесть документов. Вот как они связаны:

- **[Слой SA](sa-layer.md)** -- Детальное описание 10-фазного SA-конвейера: модули, доменная модель, юзкейсы с диаграммами активностей, формы и UI-компоненты, системные роли и фаза финализации, порождающая статистику, ADR и матрицу трассируемости.

- **[Передача между слоями](handoff.md)** -- Как слои связаны через межслойные рёбра: `AUTOMATES_AS` (шаг рабочего процесса BA к юзкейсу SA), `REALIZED_AS` (сущность BA к доменной сущности SA), `MAPPED_TO` (роль BA к системной роли SA), `GENERATES` (юзкейс SA к задаче TL). Статистика покрытия и обнаружение пробелов.

- **[Валидация](validation.md)** -- Полный каталог проверок валидации: 8 внутренних BA-проверок (L1-L8), 6 внутренних SA-проверок (L1-L6) и 9 межслойных проверок (XL1-XL9). Каждая проверка -- это Cypher-запрос с ясным критерием прохождения/непрохождения.

- **[Философия "граф в первую очередь"](graph-philosophy.md)** -- Концептуальный фундамент: почему графовая модель, почему не реляционная база данных, почему не markdown с frontmatter и как графовая модель соотносится со способом потребления и порождения структурированных данных AI-агентами.

Если вы впервые знакомитесь с NaCl, начните здесь, затем прочтите документы по слоям BA и SA по порядку. Если вы интегрируете NaCl в существующий проект, начните с [Быстрого старта](../quickstart.md) и вернитесь к методологии, когда потребуется понять, почему всё устроено именно так.
