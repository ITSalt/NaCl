# Слой бизнес-анализа: от хаоса к структуре

Слой BA преобразует неструктурированные знания заинтересованных сторон в формальную графовую модель с возможностью запросов. Четырнадцать навыков (skills) выполняют конвейер из 10 фаз, каждая из которых создает определенные типы узлов и связей в базе данных Neo4j. Результат -- полная бизнес-модель (процессы, сущности, роли, правила и глоссарий), которую слой SA может использовать без неоднозначностей.

В этом документе описаны структура конвейера, схема графа, таксономия связей, декомпозиция рабочих процессов, визуальный инструментарий, механизм возобновления и валидация.

---

## 10-фазный конвейер BA

Конвейер BA выполняется последовательно. Каждая фаза -- отдельный навык, вызываемый оркестратором (`nacl-ba-full`), и каждая фаза завершается явным шлюзом подтверждения пользователем перед началом следующей. Оркестратор делегирует каждую фазу Task-агенту -- подчиненному агенту с изолированным контекстом -- чтобы удерживать потребление токенов в рамках и предотвращать переполнение контекста между фазами.

| Фаза | Навык | Выход графа |
|------|-------|-------------|
| 1 | `nacl-ba-context` | SystemContext, Stakeholder, ExternalEntity, DataFlow |
| 2 | `nacl-ba-process` | ProcessGroup, BusinessProcess |
| 3 | `nacl-ba-workflow` | WorkflowStep (для каждого BP с `has_decomposition: true`) |
| 4 | `nacl-ba-entities` | BusinessEntity, EntityAttribute, EntityState |
| 5 | `nacl-ba-roles` | BusinessRole |
| 6 | `nacl-ba-glossary` | GlossaryTerm |
| 7 | `nacl-ba-rules` | BusinessRule |
| 8 | `nacl-ba-validate` | ValidationReport (только чтение, записей в граф нет, кроме отчета) |
| 9 | `nacl-ba-handoff` | Межслойные связи: AUTOMATES_AS, REALIZED_AS, MAPPED_TO, IMPLEMENTED_BY |
| 10 | `nacl-publish` | Markdown + Excalidraw (опционально, производный артефакт) |

**Фаза 1 -- Контекст** определяет границу системы: что входит в scope, что исключено, кто является заинтересованными сторонами и какие внешние системы обмениваются данными с целевой системой. Это задает рамки для всех последующих решений.

**Фазы 2-3 -- Процессы и рабочие потоки** формируют поведенческую модель. Фаза 2 создает карту процессов (группы, отдельные процессы, межпроцессные триггеры). Фаза 3 декомпозирует каждый процесс с пометкой `has_decomposition: true` в пошаговую диаграмму деятельности с 3 дорожками (swimlanes).

**Фазы 4-6 -- Сущности, роли, глоссарий** формируют структурную модель. Сущности -- это объекты данных, которыми оперируют процессы. Роли -- организационные участники, выполняющие шаги рабочих процессов. Глоссарий устанавливает единый язык (ubiquitous language), исключающий синонимы и двусмысленности.

**Фаза 7 -- Правила** фиксирует бизнес-ограничения, формулы расчетов, инварианты и правила авторизации. Каждое правило привязано к сущностям, которые оно ограничивает, и к процессам, в которых оно применяется.

**Фаза 8 -- Валидация** выполняет 8 проверок полноты и согласованности (L1-L8) по графу. Фаза работает строго в режиме только чтение -- она никогда не модифицирует данные, а лишь формирует отчет с указанием уровней серьезности.

**Фаза 9 -- Передача (Handoff)** создает межслойные связи между BA и SA. Эти связи (AUTOMATES_AS, REALIZED_AS, MAPPED_TO, IMPLEMENTED_BY) формируют основу трассируемости между бизнес-требованиями и системными спецификациями.

**Фаза 10 -- Публикация** генерирует человекочитаемые Markdown-документы и опциональные Excalidraw-доски из графа. Этот вывод является производным -- граф остается источником истины.

## Схема графа: 13 типов узлов

Каждый артефакт BA -- это узел с уникальным, монотонно возрастающим идентификатором. Идентификаторы никогда не переиспользуются, даже после удаления -- это гарантирует ссылочную целостность между снимками и аудиторскими следами.

```
SystemContext     SYS-NNN    (e.g., SYS-001)
Stakeholder       STK-NN     (e.g., STK-01)
ExternalEntity    EXT-NN     (e.g., EXT-01)
DataFlow          DFL-NNN    (e.g., DFL-001)
ProcessGroup      GPR-NN     (e.g., GPR-01)
BusinessProcess   BP-NNN     (e.g., BP-001)
WorkflowStep      {BP}-S{NN} (e.g., BP-001-S03)
BusinessEntity    OBJ-NNN    (e.g., OBJ-001)
EntityAttribute   {OBJ}-A{NN}(e.g., OBJ-001-A03)
EntityState       {OBJ}-ST{NN}(e.g., OBJ-001-ST02)
BusinessRole      ROL-NN     (e.g., ROL-01)
BusinessRule      BRQ-NNN    (e.g., BRQ-001)
GlossaryTerm      GLO-NNN    (e.g., GLO-001)
```

### Ключевые свойства по типам узлов

**SystemContext** -- `goals` (список), `in_scope` (список), `out_of_scope` (список), `constraints` (список), `assumptions` (список). Определяет границу системы и рамочные решения.

**Stakeholder** -- `role`, `interest`. Идентифицирует, кто заинтересован в системе и почему.

**ExternalEntity** -- `type` ("User" / "ExternalSystem" / "Organization"), `description`. Участники и системы за пределами границы.

**DataFlow** -- `direction` ("IN" / "OUT" / "BOTH"), `data_description`. Что пересекает границу системы и в каком направлении.

**ProcessGroup** -- `name`, `description`. Логическая группировка связанных бизнес-процессов.

**BusinessProcess** -- `trigger` (что запускает процесс), `result` (что он производит), `has_decomposition` (булево, существуют ли шаги рабочего процесса), `automation_level` ("manual" / "partial" / "full"). Основная поведенческая единица.

**WorkflowStep** -- `function_name`, `step_number`, `stereotype` ("Бизнес-функция" / "Автоматизируется" / "Decision"), `change_marker` ("[inherited As-Is]" / "[changed]" / "[new]"). Отдельные шаги в рамках декомпозиции процесса.

**BusinessEntity** -- `stereotype` ("Внешний документ" / "Бизнес-объект" / "Результат"), `has_states` (булево). Объекты данных, которые процессы читают, создают или изменяют.

**EntityAttribute** -- `name`, `type`. Свойства бизнес-сущности.

**EntityState** -- `name`, `description`. Дискретные состояния, в которых может находиться сущность (например, "Черновик", "Утвержден", "Архивирован").

**BusinessRole** -- `full_name`, `department`, `responsibilities` (список). Организационные участники, выполняющие шаги рабочих процессов.

**BusinessRule** -- `rule_type` ("constraint" / "calculation" / "invariant" / "authorization"), `formulation` (формулировка на естественном языке), `severity` ("critical" / "warning" / "info"). Ограничения, регулирующие поведение сущностей или выполнение процессов.

**GlossaryTerm** -- `term`, `definition`. Записи единого языка, устраняющие двусмысленности.

## Типы связей

Граф BA использует 22+ типа связей, организованных в восемь категорий. Связи несут семантическую нагрузку модели -- они выражают, кто что делает, что от чего зависит и что что ограничивает.

### Иерархия процессов

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| CONTAINS | ProcessGroup | BusinessProcess | -- |
| HAS_STEP | BusinessProcess | WorkflowStep | `order` |
| TRIGGERS | BusinessProcess | BusinessProcess | -- |
| CALLS_SUB | BusinessProcess | BusinessProcess | -- |

CONTAINS группирует процессы в логические категории. HAS_STEP связывает процесс с его декомпозированным рабочим потоком. TRIGGERS фиксирует цепочки последовательных процессов (результат одного процесса запускает другой). CALLS_SUB моделирует декомпозицию подпроцессов внутри одного процесса.

### Поток рабочего процесса

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| NEXT_STEP | WorkflowStep | WorkflowStep | `label` |
| PERFORMED_BY | WorkflowStep | BusinessRole | -- |

NEXT_STEP определяет порядок выполнения внутри рабочего процесса, включая метки ветвления для узлов принятия решений. PERFORMED_BY назначает ответственную роль для каждого шага.

### Роли и процессы

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| OWNS | BusinessRole | BusinessProcess | -- |
| PARTICIPATES_IN | BusinessRole | BusinessProcess | -- |

OWNS обозначает ответственную роль (ровно одна на процесс). PARTICIPATES_IN фиксирует дополнительные роли, участвующие в выполнении.

### Взаимодействие с сущностями

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| READS | WorkflowStep | BusinessEntity | -- |
| PRODUCES | WorkflowStep | BusinessEntity | -- |
| MODIFIES | WorkflowStep | BusinessEntity | -- |

Эти три связи формируют матрицу "сущность-процесс": какие шаги потребляют, создают или обновляют какие объекты данных. Полное покрытие (проверка L5) гарантирует, что ни одна сущность не существует без привязки хотя бы к одному процессу.

### Структура сущностей

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| HAS_ATTRIBUTE | BusinessEntity | EntityAttribute | -- |
| HAS_STATE | BusinessEntity | EntityState | -- |
| TRANSITIONS_TO | EntityState | EntityState | `condition` |
| RELATES_TO | BusinessEntity | BusinessEntity | `rel_type`, `cardinality` |

HAS_ATTRIBUTE и HAS_STATE определяют внутреннюю структуру. TRANSITIONS_TO моделирует конечный автомат с условиями перехода. RELATES_TO фиксирует межсущностные ассоциации (например, "Order RELATES_TO Customer" с кардинальностью "1:N").

### Бизнес-правила

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| CONSTRAINS | BusinessRule | BusinessEntity | -- |
| APPLIES_IN | BusinessRule | BusinessProcess | -- |
| AFFECTS | BusinessRule | EntityAttribute | -- |
| APPLIES_AT_STEP | BusinessRule | WorkflowStep | -- |

Правила привязываются к сущностям (CONSTRAINS), процессам (APPLIES_IN), конкретным атрибутам (AFFECTS) и отдельным шагам рабочего процесса (APPLIES_AT_STEP). Эта многоуровневая привязка обеспечивает трассируемость правил до точных точек их применения.

### Глоссарий

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| DEFINES | GlossaryTerm | BusinessEntity / BusinessRole / BusinessProcess | -- |
| ALIAS_OF | GlossaryTerm | GlossaryTerm | -- |

DEFINES связывает термин с узлом, который он именует. ALIAS_OF разрешает синонимы -- если заинтересованные стороны используют два слова для одного понятия, один термин становится каноническим, а другой указывает на него через ALIAS_OF.

### Контекст системы

| Связь | Источник | Цель | Свойства |
|-------|----------|------|----------|
| HAS_STAKEHOLDER | SystemContext | Stakeholder | -- |
| HAS_EXTERNAL_ENTITY | SystemContext | ExternalEntity | -- |
| HAS_FLOW | ExternalEntity | SystemContext | `direction`, `data_description` |

Эти связи формируют контекстную диаграмму: кто взаимодействует с системой, какие внешние сущности существуют и какие данные передаются между ними.

## Декомпозиция рабочего процесса в 3 дорожках

Навык `nacl-ba-workflow` декомпозирует каждый бизнес-процесс (с `has_decomposition: true`) в структурированную диаграмму деятельности, организованную в три вертикальные дорожки (swimlanes).

**Левая дорожка -- Исполнитель.** Показывает, какая BusinessRole выполняет каждый шаг. Каждый WorkflowStep имеет связь PERFORMED_BY, указывающую ровно на одну роль. Когда исполнитель меняется между последовательными шагами, визуальное представление переносит шаг в новую строку, выровненную по новой роли.

**Центральная дорожка -- Шаги рабочего процесса.** Содержит последовательность узлов WorkflowStep, соединенных связями NEXT_STEP. Каждый шаг имеет стереотип, определяющий его визуальную форму:

- **"Бизнес-функция"** -- скругленный прямоугольник. Основное ручное действие, выполняемое человеком.
- **"Автоматизируется"** -- прямоугольник с иконкой шестеренки. Шаги, которые могут быть или будут обработаны системой.
- **"Decision"** -- ромб, представляющий точку ветвления. Исходящие связи NEXT_STEP имеют свойство `label` (например, "Утверждено" / "Отклонено"), определяющее условия ветвления.

Обработка исключений моделируется выделенными узлами исключений, ответвляющимися от основного потока, что обеспечивает явное описание путей обработки ошибок.

**Правая дорожка -- Документы и сущности.** Показывает, какие узлы BusinessEntity потребляются (READS), создаются (PRODUCES) или изменяются (MODIFIES) на каждом шаге. Это создает визуальную CRUD-матрицу параллельно с потоком процесса.

### Отслеживание изменений

Каждый WorkflowStep имеет свойство `change_marker` с одним из трех значений:

- `[inherited As-Is]` -- шаг существует в текущем (as-is) процессе и остается неизменным в целевой (to-be) модели.
- `[changed]` -- шаг существует в as-is модели, но был изменен (другой исполнитель, другое взаимодействие с сущностями, другая логика).
- `[new]` -- шаг добавлен в to-be модели и не имеет as-is аналога.

Этот трехзначный маркер позволяет сравнивать as-is и to-be состояния без ведения двух отдельных графов. Один запрос Cypher может извлечь все измененные или новые шаги для формирования отчета о влиянии изменений.

## Визуальный подход: интеграция с Excalidraw

Для команд, предпочитающих совместную работу в стиле доски, NaCl поддерживает визуальный рабочий процесс через Excalidraw-доски. Три специализированных навыка обеспечивают цикл импорт-анализ-синхронизация.

### Импорт: nacl-ba-import-doc

Разбирает документ клиента (DOCX, PDF, XLSX или обычный текст) и извлекает элементы бизнес-процессов -- шаги, роли, документы, решения. Извлеченные элементы размещаются на Excalidraw-доске с раскладкой в 3 дорожках. Каждый элемент получает цвет обводки в зависимости от уверенности извлечения:

- **Зеленая обводка** -- высокая уверенность. Элемент явно указан в исходном документе.
- **Желтая обводка** -- средняя уверенность. Элемент выведен из контекста, но не указан явно.
- **Красная обводка** -- низкая уверенность. Элемент является предположением, требующим подтверждения заинтересованными сторонами.

### Анализ: nacl-ba-analyze

Читает JSON доски и выполняет 8 проверок полноты: отсутствующие исполнители, сущности-сироты, несвязанные шаги, отсутствующие метки решений, покрытие стереотипов, дублирование имен и сравнение с последним снимком. Если граф Neo4j уже существует, анализ также сравнивает доску с графом для выявления расхождений.

### Синхронизация: nacl-ba-sync

Читает доску, классифицирует каждый элемент как новый (без `nodeId`), измененный (изменен с последней синхронизации) или синхронизированный (без изменений). Для новых элементов создаются узлы Neo4j с новыми идентификаторами. Измененные элементы обновляют соответствующие узлы графа. Синхронизированные элементы пропускаются. После успешной синхронизации каждый элемент доски получает зеленую обводку, а его флаг `synced` устанавливается в `true`.

### Контракт customData в Excalidraw

Каждая фигура Excalidraw содержит структурированные метаданные в поле `customData`:

```json
{
  "nodeId": "BP-001-S03",
  "nodeType": "WorkflowStep",
  "confidence": "high",
  "sourceDoc": "process.docx",
  "sourcePage": 3,
  "synced": false
}
```

- `nodeId` -- идентификатор узла BA-графа (null до первой синхронизации).
- `nodeType` -- метка Neo4j, которой соответствует данная фигура.
- `confidence` -- уверенность извлечения из фазы импорта.
- `sourceDoc` / `sourcePage` -- отслеживание происхождения до исходного документа клиента.
- `synced` -- записан ли этот элемент в Neo4j.

### Оркестратор: nacl-ba-from-board

Полный визуальный конвейер управляется оркестратором `nacl-ba-from-board`, который выстраивает цепочку: создание доски --> импорт документа --> анализ полноты --> синхронизация с графом --> обогащение сущностей и ролей --> валидация --> передача. Это повторяет 10-фазный конвейер `nacl-ba-full`, но начинается с визуального артефакта, а не с диалогового интервью.

## Обнаружение точки возобновления

Когда `nacl-ba-full` запускается, он не выполняет все 10 фаз заново вслепую. Вместо этого он запрашивает граф для определения, какие фазы уже завершены:

```cypher
OPTIONAL MATCH (sc:SystemContext) WITH count(sc) > 0 AS phase1
OPTIONAL MATCH (gpr:ProcessGroup) WITH phase1, count(gpr) > 0 AS phase2
OPTIONAL MATCH (ws:WorkflowStep) WITH phase1, phase2, count(ws) > 0 AS phase3
OPTIONAL MATCH (be:BusinessEntity) WITH phase1, phase2, phase3, count(be) > 0 AS phase4
OPTIONAL MATCH (br:BusinessRole) WITH phase1, phase2, phase3, phase4, count(br) > 0 AS phase5
OPTIONAL MATCH (gt:GlossaryTerm) WITH phase1, phase2, phase3, phase4, phase5, count(gt) > 0 AS phase6
OPTIONAL MATCH (bru:BusinessRule) WITH phase1, phase2, phase3, phase4, phase5, phase6, count(bru) > 0 AS phase7
// ... through phase 9
```

Логика обнаружения кумулятивна: фаза N считается завершенной только при условии, что фазы с 1 по N-1 также завершены. Это предотвращает ложные срабатывания от частичного импорта или ручных правок в произвольном порядке.

Если фазы 1-5 обнаружены как завершенные, оркестратор возобновляет работу с фазы 6 (глоссарий). Никакая работа не теряется, завершенные фазы не перевыполняются. Пользователь видит сводку обнаруженных фаз и подтверждает точку возобновления перед продолжением выполнения.

Этот механизм критически важен для длительных сеансов анализа. Если контекст потерян в середине конвейера (таймаут, разрыв соединения, исчерпание токенов), аналитик может перезапустить `nacl-ba-full` и продолжить ровно с того места, где остановился. Граф выступает контрольной точкой -- а не история диалога.

## Внутренняя валидация (L1-L8)

Навык `nacl-ba-validate` выполняет 8 проверок полноты и согласованности BA-графа. Валидация работает строго в режиме **только чтение** -- она никогда не создает, не обновляет и не удаляет узлы или связи. Она только читает граф и формирует отчет.

| Уровень | Проверка | Назначение |
|---------|----------|------------|
| L1 | Все BP имеют владельца (связь OWNS) | Ответственность за процесс -- каждый процесс должен иметь ровно одну ответственную роль |
| L2 | BP с `has_decomposition: true` имеют связи HAS_STEP | Покрытие рабочих процессов -- объявленные декомпозиции должны фактически существовать |
| L3 | Каждый WorkflowStep имеет связь PERFORMED_BY | Привязка исполнителя -- ни один шаг не должен оставаться без ответственной роли |
| L4 | EntityAttribute имеет допустимые типы | Качество данных -- типы атрибутов должны соответствовать допустимым значениям |
| L5 | Покрытие READS/PRODUCES/MODIFIES | Матрица "сущность-процесс" -- нет сущностей-сирот, нет нетронутых объектов данных |
| L6 | Полнота матрицы "роль-процесс" | Нет сиротских ролей или процессов -- каждая роль участвует хотя бы в одном процессе |
| L7 | Покрытие GlossaryTerm для именованных узлов | Единый язык -- ключевые доменные понятия должны иметь записи в глоссарии |
| L8 | BusinessRule имеет связи CONSTRAINS или APPLIES_IN | Трассируемость правил -- каждое правило должно быть привязано хотя бы к одной сущности или процессу |

### Уровни серьезности

Каждое замечание классифицируется по одному из трех уровней серьезности:

- **CRITICAL** -- блокирует передачу в слой SA. Модель имеет структурный пробел, который распространит ошибки на последующие этапы. Примеры: процесс без владельца (L1), декомпозированный процесс без шагов рабочего процесса (L2).
- **WARNING** -- следует исправить перед передачей, но формально не блокирует ее. Примеры: роль, которая существует, но не участвует ни в одном процессе (L6), сущность без термина в глоссарии (L7).
- **INFO** -- рекомендация по улучшению. Модель валидна без их устранения, но исправление повышает полноту. Пример: атрибут сущности с обобщенным типом, который можно уточнить (L4).

Отчет валидации перечисляет все замечания, сгруппированные по уровням, с указанием конкретных идентификаторов и имен узлов, вызвавших каждую проверку. Это упрощает исправление -- аналитик может перейти непосредственно к проблемным узлам и исправить их перед повторным запуском валидации.

Валидация выполняется автоматически как фаза 8 в `nacl-ba-full`, но может быть также запущена независимо в любое время через `/nacl-ba-validate`. Запуск после ручных правок графа выявляет проблемы до их накопления.
