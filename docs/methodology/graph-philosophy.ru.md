# Философия "Сначала граф": почему это работает

NaCl хранит все артефакты спецификаций BA и SA в графовой базе данных Neo4j, а не в markdown-файлах. Это не технологическое предпочтение -- это структурный ответ на структурную проблему. Когда AI-агент должен работать с десятками взаимосвязанных артефактов спецификации на протяжении нескольких сессий, формат хранения определяет, потратит ли агент свой бюджет токенов на продуктивную работу или на перечитывание документов, которые он не может запомнить.

Этот документ объясняет, почему графовая модель данных является правильным форматом хранения спецификаций, потребляемых AI-агентами, какие конкретные преимущества она даёт агентам и людям, как модель двунаправленной синхронизации соединяет визуальное редактирование и машинную доступность для запросов, и где граф намеренно останавливается.

---

## Антипаттерн файлового хранения

Традиционный подход к хранению спецификаций -- это директория `docs/` с markdown-файлами. На реальном проекте с 10-15 юзкейсами эта директория разрастается до 70+ файлов: карты процессов, каталоги сущностей, ролевые матрицы, бизнес-правила, глоссарий, диаграммы рабочих процессов, спецификации UI, определения доменной модели, API-контракты и другое.

Такой подход порождает пять нарастающих проблем.

**Взрыв контекста.** Чтобы сгенерировать задачу для UC-101, AI-агент должен прочитать: спецификацию UC, доменную модель, API-контракт, описания форм, определения ролей, связанные требования. В совокупности по файлам это составляет примерно 150K токенов на один юзкейс. Для агента с конечным контекстным окном это означает, что планирование двух-трёх юзкейсов может исчерпать весь бюджет до начала какой-либо продуктивной работы.

**Расточительность токенов.** Большая часть контента, который агент читает, не имеет отношения к конкретному юзкейсу. Файл доменной модели описывает 30 сущностей, но UC-101 затрагивает 3 из них. Файл определений ролей перечисляет 8 ролей, но UC-101 задействует 2. Агент не может знать, что релевантно, не прочитав всё. Результат -- фундаментально неэффективный паттерн чтения: потребить 150K токенов, использовать 5K из них.

**Несогласованность.** Ничто не мешает файлу `docs/domain-model.md` утверждать, что "Order имеет 5 полей", в то время как `docs/uc-101.md` ссылается на 7 полей. Markdown-файлы -- это независимые текстовые документы. Они не валидируют друг друга, не имеют общей схемы и не обеспечивают ссылочную целостность. Несогласованности накапливаются незаметно, пока не всплывают в виде багов при реализации -- самом дорогом месте для обнаружения ошибок в спецификации.

**Отсутствие анализа влияния.** "Что сломается, если я переименую поле `status`?" В файловой системе для ответа на этот вопрос нужно сделать grep по 70 файлам и надеяться, что вы нашли все ссылки. Grep ищет строки, а не семантику. Он находит "status" в комментариях, в несвязанных контекстах, и пропускает ссылки с другой формулировкой. Результат -- либо ложные срабатывания, тратящие время на разбор, либо пропущенные зависимости, вызывающие регрессии.

**Устаревание контента.** По мере развития проекта одни файлы обновляются, а другие нет. Изменение доменной модели может распространиться на спецификацию UC, но не на API-контракт. Нет механизма для обнаружения рассинхронизации файлов. Набор спецификаций постепенно деградирует в коллекцию документов, зафиксированных в разные моменты времени, без способа отличить текущую истину от исторических артефактов.

Это не теоретические риски. Это состояние по умолчанию для любого нетривиального набора спецификаций, хранящегося в плоских файлах. Подход "сначала граф" устраняет все пять проблем, делая связи явными, доступными для запросов и проверяемыми.

## Что граф даёт AI-агентам

Переход на Neo4j означает, что каждый артефакт -- это узел, каждая связь -- это ребро, и каждое ограничение обеспечивается схемой. Последствия для рабочих процессов AI-агентов -- немедленные и измеримые.

**Один запрос на UC вместо чтения 70 файлов.** Cypher-запрос `sa_uc_full_context` возвращает полный подграф для юзкейса -- шаги активности, формы, поля, атрибуты домена, сущности, требования, роли -- примерно в 550 токенах. Навык `nacl-tl-plan` использует этот запрос для генерации файлов задач. Сравните 550 токенов на UC со 150 000 токенов для файлового эквивалента. Это сокращение потребления контекста на 99,6%, что напрямую выражается в способности агента планировать больше юзкейсов за сессию, удерживать больше рабочего контекста одновременно и тратить бюджет токенов на генерацию кода, а не на чтение спецификаций.

**Точный анализ влияния.** "Что сломается, если я переименую поле `status`?" превращается в один Cypher-запрос:

```cypher
MATCH (da:DomainAttribute {name: "status"})
  <-[:MAPS_TO]-(ff:FormField)
  <-[:HAS_FIELD]-(f:Form)
  <-[:USES_FORM]-(uc:UseCase)
RETURN uc.id, uc.name, f.name, ff.name
```

Результат -- точный список затронутых юзкейсов, форм и полей. Никаких ложных срабатываний от поиска строк в комментариях. Никаких пропущенных зависимостей из-за альтернативных формулировок. Граф кодирует семантические связи, а не текстовые совпадения, поэтому ответ одновременно полный и точный.

**Автоматическая валидация.** 17+ проверок валидации (L1-L8 для внутренней согласованности BA, L1-L6 для внутренней согласованности SA, XL1-XL9 для межслойной целостности) -- это Cypher-запросы, выполняющиеся за секунды. Они обнаруживают "осиротевшие" узлы -- DomainEntity без модуля, UseCase без актора, FormField, ссылающийся на несуществующий DomainAttribute. В файловой системе каждая из этих проверок потребовала бы написания отдельного парсера для конкретного формата файла, и парсеры ломались бы при каждом переформатировании заголовка или изменении структуры таблицы.

**Сквозная трассируемость.** От поля UI-формы можно проследить весь путь до бизнес-процесса, который его породил:

```
FormField -[:MAPS_TO]-> DomainAttribute -[:HAS_ATTRIBUTE]-> DomainEntity
  -[:REALIZED_AS]<- BusinessEntity -[:PRODUCES]<- WorkflowStep
  -[:HAS_STEP]<- BusinessProcess
```

Этот обход -- один Cypher-запрос. В файловой системе для этого нужно прочитать и сопоставить четыре-пять разных документов, мысленно собрать цепочку и надеяться, что каждая промежуточная ссылка написана правильно и актуальна. Граф делает эту цепочку явным, машинно-обходимым путём.

## Что граф даёт людям

Граф -- это не только оптимизация для AI. Он предоставляет возможности, которые люди не могут получить из плоских файлов ни при каких затратах токенов.

**Спецификации как база данных для запросов.** "Покажи все сущности, которые роль Менеджер может изменять" -- это Cypher-запрос, возвращающий немедленный и авторитетный ответ. Не нужно листать документы, нажимать Ctrl+F по файлам, собирать частичные ответы из множества источников. Спецификация становится базой данных, отвечающей на вопросы, а не библиотекой, требующей чтения.

**Визуальные доски.** NaCl интегрируется с Excalidraw для визуального моделирования процессов. Карты бизнес-процессов, диаграммы рабочих процессов и диаграммы связей сущностей отображаются как интерактивные доски, где заинтересованные стороны видят полную картину и могут непосредственно с ней работать. Доски синхронизируются двунаправленно с графом: отредактируйте фигуру на доске -- и соответствующий узел графа обновится; обновите узел в графе -- и доска перегенерируется, отражая изменение. Заинтересованные стороны работают с понятным им визуальным представлением, а граф поддерживает структурированную истину под ним.

**Автоматические матрицы трассируемости.** Запрос `handoff_traceability_matrix` генерирует полную таблицу перекрёстных ссылок из графа -- какие бизнес-процессы соответствуют каким юзкейсам, какие сущности реализуют какие бизнес-объекты, какие роли обладают какими системными разрешениями. В файловой системе матрица трассируемости -- это вручную составленная таблица, устаревающая в момент, когда кто-то добавляет юзкейс и забывает обновить матрицу. В графе матрица -- это результат запроса. Она не может устареть, потому что вычисляется из тех же данных, которые трассирует.

**Живая документация.** Навык `nacl-publish` рендерит граф в вики-страницы Docmost и диаграммы Excalidraw. Документация всегда производна от графа -- она не может рассинхронизироваться, потому что перегенерируется из источника истины при каждой публикации. Это инвертирует традиционное отношение между спецификациями и документацией. Вместо того чтобы документация была основным артефактом, который может отражать, а может и не отражать реальность, граф является основным артефактом, а документация -- его проекцией только для чтения.

## Модель двунаправленной синхронизации

NaCl использует два взаимодополняющих интерфейса к одним и тем же данным, каждый из которых оптимизирован для своей аудитории.

**Доски Excalidraw** -- это интерфейс для людей. Они визуальны, интуитивны и поддерживают совместное редактирование. Каждая фигура на доске несёт свойства `customData`, связывающие её с узлом графа -- ID фигуры, тип, метку и флаг `synced`, указывающий, была ли фигура отправлена в Neo4j.

**Граф Neo4j** -- это интерфейс для машин. Он структурирован, доступен для запросов и проверяем. Каждый узел имеет типизированные свойства, каждое ребро имеет определённое значение, а схема обеспечивает ограничения, предотвращающие структурные нарушения.

Цикл синхронизации соединяет эти два интерфейса:

1. **Импорт.** Клиентский документ (DOCX, PDF или текст) разбирается навыком `nacl-ba-import-doc`, который создаёт доску Excalidraw с фигурами, размещёнными по дорожкам (swimlanes). Каждая фигура имеет `synced: false` и цвет достоверности, указывающий, насколько парсер уверен в правильности извлечения.

2. **Ревью.** Аналитик проверяет доску визуально -- корректирует метки, перемещает фигуры между дорожками, добавляет пропущенные шаги, удаляет дубликаты. Это этап экспертного суждения, на котором доменная экспертиза исправляет и обогащает машинное извлечение.

3. **Синхронизация.** Запуск `nacl-ba-sync` отправляет новые и изменённые фигуры в Neo4j как узлы и рёбра. Фигуры получают `synced: true`, и их цвет обводки меняется на зелёный, обеспечивая немедленную визуальную обратную связь о том, что было зафиксировано в графе.

4. **Валидация.** Запросы валидации графа выполняются по синхронизированным данным, выявляя структурные проблемы -- отсутствующие связи, осиротевшие узлы, неполное покрытие. Проблемы возвращаются с указанием конкретных узлов.

5. **Исправление и повторная синхронизация.** Исправления можно вносить как на доске (визуальные корректировки), так и напрямую в графе (Cypher-мутации). В любом случае повторная синхронизация приводит оба представления в согласованное состояние.

Эта модель означает, что заинтересованным сторонам не нужно учить Cypher или разбираться в графовых базах данных. Они работают с визуальными досками, похожими на привычные им вайтборды. AI-агентам не нужно разбирать визуальные макеты. Они работают с графом, который даёт им типизированные, доступные для запросов данные с богатыми связями. Одни и те же данные, разные представления, каждое оптимизировано для своего потребителя.

## Когда НЕ стоит использовать граф

Граф предназначен для спецификаций, а не для кода. Слой TL намеренно хранит три категории артефактов вне графа:

**Код** живёт в файлах под управлением Git (`src/`). У кода есть собственная система версионирования с диффами, ветками и разрешением конфликтов слияния. У него есть собственные инструменты качества -- линтеры, проверка типов, раннеры тестов, статические анализаторы. У него есть собственный процесс ревью через пулл-реквесты. Дублирование кода в графе добавило бы накладные расходы на каждый коммит без какой-либо пользы, которую Git не обеспечивал бы лучше.

**Файлы задач** живут в файловой системе (`.tl/tasks/`). Каждый файл задачи -- это самодостаточный документ спецификации, сгенерированный из графа навыком `nacl-tl-plan`. После генерации файл задачи содержит всё, что нужно агенту-разработчику для реализации юзкейса: критерии приёмки, API-контракты, определения сущностей, спецификации форм, зависимости. Файл задачи -- это точка передачи, граница, где спецификация на основе графа заканчивается и начинается файловая разработка.

**Артефакты сборки** живут в CI/CD-пайплайнах. Docker-образы, отчёты о тестировании, логи деплоя -- это эфемерные, специфичные для пайплайна выходные данные, которым не место в графе спецификаций.

Логика этой границы проста. Ценность графа -- в слое спецификаций, где связи между артефактами важнее содержимого любого отдельного артефакта. "Какие юзкейсы затрагивают сущность Order?" -- это вопрос о связях, на который граф отвечает мгновенно. Код -- противоположность: содержимое -- логика, поток управления, преобразования данных -- важнее связей между файлами. Корректность функции зависит от того, что она делает, а не от того, сколько других файлов на неё ссылаются.

**Граница проведена явно:** SA-граф порождает файлы задач через `nacl-tl-plan`, а файлы задач порождают код через `nacl-tl-dev-be` и `nacl-tl-dev-fe`. После того как граф генерирует самодостаточные файлы задач, агенты-разработчики работают исключительно с файлами, а не с графом. Работа графа завершена, когда спецификация достаточно точна, чтобы быть самостоятельным рабочим заданием. Расширение графа на слой кода размыло бы эту границу, не улучшив ни спецификации, ни код.
