# Графовая инфраструктура

Docker-стек Neo4j для графовых скиллов. Обязателен для всех `nacl-*` скиллов.

## Что получите

| Сервис | Назначение | Порт по умолчанию |
|--------|-----------|-------------------|
| Neo4j | Графовая база данных | 3574 (HTTP), 3587 (Bolt) |

> **Excalidraw-борды** теперь ведёт **NaCl Analyst Tool** (`analyst-tool/`) — локальный Node.js-процесс вне Docker. Запускается отдельно: `cd analyst-tool && npm install && npm run dev` — слушает `http://localhost:3582`.

## Шаг 1: Запустить Docker-стек

```bash
cd ~/NaCl
cp graph-infra/.env.example graph-infra/.env    # при необходимости измените пароли
docker compose -f graph-infra/docker-compose.yml up -d
```

## Шаг 2: Загрузить схему

Подождите ~30 секунд, затем:

```bash
CONTAINER=$(docker ps --format '{{.Names}}' | grep neo4j | head -1)
NEO4J_PASS=$(grep NEO4J_PASSWORD graph-infra/.env | cut -d= -f2)

for f in graph-infra/schema/ba-schema.cypher graph-infra/schema/sa-schema.cypher graph-infra/schema/tl-schema.cypher; do
  docker exec -i "$CONTAINER" cypher-shell -u neo4j -p "$NEO4J_PASS" < "$f"
done
```

## Шаг 3: MCP-сервер Neo4j

Отдельного npm-пакета больше нет — `@anthropic/neo4j-mcp` был ранним npm-лаунчером и был
заброшен (холодный старт дольше 30 секунд регулярно упирался в таймаут подключения MCP, а
баннер в stdout ломал stdio JSON-RPC поток). Теперь `/nacl-init` сам скачивает официальный
Go-бинарник из релизов GitHub `neo4j/mcp`:

```
/nacl-init "Ваш проект"
```

Скачивается **версия-запиненный, проверенный по sha256** релиз-ассет (пин и контрольные
суммы лежат в `nacl-tl-core/scripts/neo4j-mcp.pin`) в `~/.neo4j-mcp-bin/neo4j-mcp`, и
`.mcp.json` записывается сразу с путём к этому бинарнику — без поиска в `PATH`, без
глобальной установки. Чтобы отслеживать другой релиз, задайте `NEO4J_MCP_VERSION=<tag>`
перед запуском `/nacl-init` (или `NEO4J_MCP_VERSION=latest`, чтобы отключить пин — тогда
отключается и проверка контрольной суммы, об этом печатается явное предупреждение).

**Ручной путь (если исходящий доступ заблокирован):** скачайте нужный ассет
`neo4j-mcp_<OS>_<ARCH>.tar.gz` для запиненной версии со страницы
`https://github.com/neo4j/mcp/releases` в браузере, распакуйте и положите бинарник
`neo4j-mcp` в `~/.neo4j-mcp-bin/neo4j-mcp` (`chmod +x`) перед запуском `/nacl-init` —
скрипт установки переиспользует уже существующий бинарник по этому пути вместо скачивания.

`.mcp.json` проекта после этого выглядит так:

```json
{
  "mcpServers": {
    "neo4j": {
      "type": "stdio",
      "command": "<home>/.neo4j-mcp-bin/neo4j-mcp",
      "args": [],
      "env": {
        "NEO4J_URI": "bolt://localhost:3587",
        "NEO4J_USERNAME": "neo4j",
        "NEO4J_PASSWORD": "neo4j_graph_dev",
        "NEO4J_DATABASE": "neo4j",
        "NEO4J_TELEMETRY": "false"
      }
    }
  }
}
```

Начните новую сессию для подключения — MCP-серверы подхватываются только при
старте сессии (проверено вживую: hot-reload нет, а `/mcp reconnect` не видит
НОВЫЕ записи в `.mcp.json`). Smoke-проверка в новой сессии — один вызов:
попросите Claude выполнить `mcp__neo4j__read-cypher "RETURN 1"` — должно
вернуться `1`.

### Claude Code Desktop

Desktop читает тот же `.mcp.json` проекта — отдельной настройки не нужно. На macOS GUI-
приложение берёт `PATH` из того, как оно было запущено, а не из профиля вашей оболочки —
если, например, `docker` работает в терминале, но не в Desktop, перезапустите приложение
Desktop после изменения профиля оболочки, либо добавьте нужный путь через встроенный
редактор переменных окружения Desktop. После того как `/nacl-init` запишет `.mcp.json`,
начните новую сессию — как и CLI, Desktop подхватывает изменения MCP-серверов только
при старте сессии. Учтите: набор `/mcp` в Desktop открывает каталог коннекторов
(добавление новых), а не панель статуса — для проверки подключения просто попросите
Claude выполнить `mcp__neo4j__read-cypher "RETURN 1"` в новой сессии.

## Шаг 4: Проверка

В Claude Code: `/nacl-ba-context` — если начнёт спрашивать о бизнес-домене, всё работает.

## Несколько проектов

Каждый проект получает свой Docker-стек с разными портами. Скилл `/nacl-init` делает это автоматически.

## Параллельные сессии и worktree

Конфиг (`config.yaml`, `.mcp.json`) и графовая инфраструктура (`graph-infra/`) живут в
**основной рабочей копии** — запускайте `/nacl-init` только оттуда, никогда из связанного
worktree (у worktree нет своего `.mcp.json`). Параллельные сессии Claude Code, включая
несколько окон Desktop, работают с одним и тем же графом проекта: одновременно запускайте
не более одного *пишущего* BA/SA-скилла, чтобы избежать гонок при параллельной записи;
скиллы только для чтения (запросы, ревью) можно запускать параллельно.

## Проверка живости

`node "$HOME/.claude/skills/nacl-core/scripts/graph-doctor.mjs"` — быстрая проверка
живости без внешних зависимостей. Печатает `NACL_GRAPH_DOCTOR: status=UP|DOWN|NOT_NACL` без
полного MCP-рукопожатия. Запуск с `--fix` чинит автоматически — стартует локальный
Docker-контейнер (режим local) или перезапускает удалённый sidecar (режим remote),
идемпотентно:

```bash
node "$HOME/.claude/skills/nacl-core/scripts/graph-doctor.mjs" --fix
```

Скиллы `nacl-*` теперь сами вызывают эту проверку и предлагают выполнить `--fix`, когда
граф недоступен, вместо того чтобы просто падать с ошибкой.

## Решение проблем

- **Neo4j не стартует**: проверьте, что Docker имеет достаточно памяти (мин. 2ГБ)
- **Схема не загружается**: откройте Neo4j Browser на `http://localhost:3574`, выполните `.cypher` файлы вручную
- **MCP не подключается**: сначала выполните `node "$HOME/.claude/skills/nacl-core/scripts/graph-doctor.mjs"` для проверки живости. Убедитесь, что бинарник существует и исполняем (`ls -l ~/.neo4j-mcp-bin/neo4j-mcp`), `.mcp.json` в корне проекта указывает на этот путь, затем начните новую сессию — MCP-серверы подхватываются только при старте сессии, а не при перезапуске внутри неё; `/mcp reconnect` новые записи `.mcp.json` тоже не видит
