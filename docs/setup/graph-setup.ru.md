# Графовая инфраструктура

Docker-стек Neo4j для графовых скиллов. Обязателен для всех `nacl-*` скиллов.

## Что получите

| Сервис | Назначение | Порт по умолчанию |
|--------|-----------|-------------------|
| Neo4j | Графовая база данных | 3574 (HTTP), 3587 (Bolt) |

> **Excalidraw-борды** теперь ведёт **NaCl Analyst Tool** (`analyst-tool/`) — локальный Node.js-процесс вне Docker. Запускается отдельно: `cd analyst-tool && npm install && npm run dev` — слушает `http://localhost:3582`.

## Шаг 1: Запустить Docker-стек

```bash
cd ~/NaCl
cp graph-infra/.env.example graph-infra/.env    # при необходимости измените пароли
docker compose -f graph-infra/docker-compose.yml up -d
```

## Шаг 2: Загрузить схему

Подождите ~30 секунд, затем:

```bash
CONTAINER=$(docker ps --format '{{.Names}}' | grep neo4j | head -1)
NEO4J_PASS=$(grep NEO4J_PASSWORD graph-infra/.env | cut -d= -f2)

for f in graph-infra/schema/ba-schema.cypher graph-infra/schema/sa-schema.cypher graph-infra/schema/tl-schema.cypher; do
  docker exec -i "$CONTAINER" cypher-shell -u neo4j -p "$NEO4J_PASS" < "$f"
done
```

## Шаг 3: MCP-сервер Neo4j

Путь через `.mcp.json` в шагах 3-4 относится только к Claude
Code/full-plugin compatibility; Codex использует доверенный проектный
`.codex/config.toml`, как описано ниже. Отдельного npm-пакета больше нет —
`@anthropic/neo4j-mcp` был ранним npm-лаунчером и был
заброшен (холодный старт дольше 30 секунд регулярно упирался в таймаут подключения MCP, а
баннер в stdout ломал stdio JSON-RPC поток). Теперь `/nacl-init` сам скачивает официальный
Go-бинарник из релизов GitHub `neo4j/mcp`:

```
/nacl-init "Ваш проект"
```

Скачивается **версия-запиненный, проверенный по sha256** релиз-ассет (пин и контрольные
суммы лежат в `nacl-tl-core/scripts/neo4j-mcp.pin`) в `~/.neo4j-mcp-bin/neo4j-mcp`, и
Claude-совместимый `.mcp.json` записывается сразу с путём к этому бинарнику — без поиска в `PATH`, без
глобальной установки. Чтобы отслеживать другой релиз, задайте `NEO4J_MCP_VERSION=<tag>`
перед запуском `/nacl-init` (или `NEO4J_MCP_VERSION=latest`, чтобы отключить пин — тогда
отключается и проверка контрольной суммы, об этом печатается явное предупреждение).

**Ручной путь (если исходящий доступ заблокирован):** скачайте нужный ассет
`neo4j-mcp_<OS>_<ARCH>.tar.gz` для запиненной версии со страницы
`https://github.com/neo4j/mcp/releases` в браузере, распакуйте и положите бинарник
`neo4j-mcp` в `~/.neo4j-mcp-bin/neo4j-mcp` (`chmod +x`) перед запуском `/nacl-init` —
скрипт установки переиспользует уже существующий бинарник по этому пути вместо скачивания.

`.mcp.json` проекта после этого выглядит так:

```json
{
  "mcpServers": {
    "neo4j": {
      "type": "stdio",
      "command": "<home>/.neo4j-mcp-bin/neo4j-mcp",
      "args": [],
      "env": {
        "NEO4J_URI": "bolt://localhost:3587",
        "NEO4J_USERNAME": "neo4j",
        "NEO4J_PASSWORD": "${NEO4J_PASSWORD}",
        "NEO4J_DATABASE": "neo4j",
        "NEO4J_TELEMETRY": "false"
      }
    }
  }
}
```

Начните новую сессию для подключения — MCP-серверы подхватываются только при
старте сессии (проверено вживую: hot-reload нет, а `/mcp reconnect` не видит
НОВЫЕ записи в `.mcp.json`). Smoke-проверка в новой сессии — один вызов:
попросите Claude выполнить `mcp__neo4j__read-cypher "RETURN 1"` — должно
вернуться `1`.

## Шаг 4: Настроить MCP в Claude Code

### Claude Code Desktop

Desktop читает тот же `.mcp.json` проекта — отдельной настройки не нужно. На macOS GUI-
приложение берёт `PATH` из того, как оно было запущено, а не из профиля вашей оболочки —
если, например, `docker` работает в терминале, но не в Desktop, перезапустите приложение
Desktop после изменения профиля оболочки, либо добавьте нужный путь через встроенный
редактор переменных окружения Desktop. После того как `/nacl-init` запишет `.mcp.json`,
начните новую сессию — как и CLI, Desktop подхватывает изменения MCP-серверов только
при старте сессии. Учтите: набор `/mcp` в Desktop открывает каталог коннекторов
(добавление новых), а не панель статуса — для проверки подключения просто попросите
Claude выполнить `mcp__neo4j__read-cypher "RETURN 1"` в новой сессии.

## Шаг 5: Smoke test

В Claude Code: `/nacl-ba-context` — если начнёт спрашивать о бизнес-домене, всё работает.

## Несколько проектов

Каждый проект получает свой Docker-стек с разными портами. Скилл `/nacl-init` делает это автоматически.

## Параллельные сессии и worktree

Конфиг (`config.yaml`, Claude `.mcp.json`, Codex `.codex/config.toml`) и графовая инфраструктура (`graph-infra/`) живут в
**основной рабочей копии** — запускайте `/nacl-init` только оттуда, никогда из связанного
worktree (у worktree нет своего `.mcp.json`). Параллельные сессии Claude Code, включая
несколько окон Desktop, работают с одним и тем же графом проекта: одновременно запускайте
не более одного *пишущего* BA/SA-скилла, чтобы избежать гонок при параллельной записи;
скиллы только для чтения (запросы, ревью) можно запускать параллельно.

## Проверка живости

`node "$HOME/.claude/skills/nacl-core/scripts/graph-doctor.mjs"` — быстрая проверка
живости без внешних зависимостей. Печатает `NACL_GRAPH_DOCTOR: status=UP|DOWN|NOT_NACL` без
полного MCP-рукопожатия. Запуск с `--fix` чинит автоматически — стартует локальный
Docker-контейнер (режим local) или перезапускает удалённый sidecar (режим remote),
идемпотентно:

```bash
node "$HOME/.claude/skills/nacl-core/scripts/graph-doctor.mjs" --fix
```

Этот путь — для **CLI-канала с симлинками**. В **канале плагина Claude Code Desktop**
`nacl-core` лежит под корнем плагина: сборка плагина переписывает ту же команду в
`node "$(nacl-home)/nacl-core/scripts/graph-doctor.mjs" --fix`, а SessionStart-хук плагина
запускает `node "${CLAUDE_PLUGIN_ROOT}/nacl-core/scripts/graph-doctor.mjs" --hook`. См.
`docs/setup/install-skills.ru.md`, раздел «Выберите канал» — на одной машине выбирается
только один канал.

Скиллы `nacl-*` теперь сами вызывают эту проверку и предлагают выполнить `--fix`, когда
граф недоступен, вместо того чтобы просто падать с ошибкой.

## Наблюдатель внутри сессии (плагин)

Установка через плагин (`/plugin install nacl@nacl`) дополнительно включает фоновый
монитор (`monitors/monitors.json` → `graph-doctor.mjs --watch`): долгоживущий процесс,
стартующий вместе с сессией и перепроверяющий порт графа каждые 30 секунд. Он дополняет
SessionStart-хук: хук ловит граф, лежащий уже на старте сессии; наблюдатель — контейнер,
умерший *после* старта (раньше это было незаметно до первого упавшего запроса к графу).

Пока состояние стабильно, наблюдатель молчит; одну строку `NACL_GRAPH_WATCH:` он печатает
только на переходе UP→DOWN или DOWN→UP. По официальной документации плагинов каждая
stdout-строка монитора доставляется Claude как уведомление прямо посреди разговора — так
Claude сам узнаёт, что граф упал (со ссылкой на `--fix`), а после восстановления — что
инструменты `mcp__neo4j__*` снова безопасны. В не-NaCl проектах (нет блока `graph:` в
`config.yaml`) наблюдатель завершается сразу и ничего не печатает.

> **Живая проверка в Desktop ещё не проводилась.** Описанный UX уведомлений основан на
> официальной документации мониторов плюс локальном end-to-end прогоне `--watch` (реальные
> TCP-переходы, одна строка на переход); уведомление UP→DOWN в живой сессии Claude Code
> Desktop **пока не наблюдалось**. Шаг владельца: остановить Neo4j-контейнер проекта
> посреди сессии (`docker stop <container>`) и убедиться, что уведомление пришло.

## Заметки по платформам

### macOS

Используйте Docker Desktop и выделите ему не менее 4 ГБ RAM.

### Linux

Docker Engine и Docker Desktop поддерживаются; для Engine пользователь должен входить в группу `docker`.

### Windows

Используйте Docker Desktop с WSL2 backend; Docker-команды удобнее запускать из WSL2-терминала.

## Решение проблем

- **Neo4j не стартует**: проверьте, что Docker имеет достаточно памяти (мин. 2ГБ)
- **Схема не загружается**: перед повторным запуском проверенного загрузчика схемы получите секрет
  проекта из настроенного runtime-источника. Для local нужен `NEO4J_PASSWORD` в окружении текущего
  runtime; для remote обязателен `graph.remote.secret_source` (`env:NEO4J_PASSWORD` или
  `server-route:<id>`). Если env или внешний провайдер недоступен, остановитесь и исправьте провайдер:
  не используйте demo/default пароль и не записывайте сырой пароль в файлы проекта
- **MCP не подключается**: сначала выполните `node "$HOME/.claude/skills/nacl-core/scripts/graph-doctor.mjs"` (CLI-канал) или `node "$(nacl-home)/nacl-core/scripts/graph-doctor.mjs"` (канал плагина Desktop) для проверки живости. Для Claude проверьте, что `.mcp.json` в корне проекта указывает на исполняемый бинарник, затем начните новую сессию. Для Codex проверьте канонический доверенный корень и стабильную запись `[mcp_servers.nacl_neo4j]` в `.codex/config.toml`, указывающую на сгенерированный launcher без секрета; откройте новую задачу и потребуйте handshake/read evidence. Claude `.mcp.json` никогда не доказывает pickup в Codex

## Режимы графа Codex

### Локальный граф проекта

По умолчанию Codex использует один контейнер Neo4j 5 Community и постоянные volumes на
проект. `nacl-init` планирует и создаёт стек, ждёт health, загружает схему, устанавливает
запиненный `neo4j-mcp`, создаёт launcher без секрета, объединяет
`[mcp_servers.nacl_neo4j]` в доверенном проектном `.codex/config.toml` и
фиксирует маршрут. Текущий
Git/full-plugin канал совместимости также может использовать ограниченный пакетный gateway.
Целевой официальный Skills-only канал запускает установленные скрипты `nacl-init`, а затем
требует новую задачу для загрузки проектного MCP; публичный NaCl MCP и вторая Git-установка
ему не нужны.

```toml
[mcp_servers.nacl_neo4j]
command = "<абсолютный путь этой машины к сгенерированному NaCl launcher>"
args = []
```

URI и ссылку на credential при запуске получает launcher, а не TOML или argv.
Путь регенерируется на каждой машине и не должен указывать в developer checkout
или plugin cache. Codex загружает проектный слой только после доверия
каноническому корню. Если новая задача видит только глобальные MCP, проверьте
канонический root/trust; на support-машине read-only диагностикой служит
`codex mcp list --json`.

### VPS-подключение и авторизация

`nacl-init` может создать или подключить контейнер проекта на доступном VPS. NaCl не предоставляет
managed graph. Сервер сейчас является границей авторизации: доступ к нему означает доступ ко всем базам
проектов на нём. Каждый проект получает отдельный контейнер Neo4j 5 Community и независимые постоянные
volumes. `project_scope` выбирает его маршрут и фиксирует provenance, но не выдаёт доступ и не является
маркером авторизации `(:Project)` в общем графе.
Опциональные remote create/connect scripts сохраняют существующую mTLS-границу доступа к серверу и
отклоняют cross-server routing. Публичный Skills-only продукт не добавляет hosted Streamable HTTP или OAuth gateway.
