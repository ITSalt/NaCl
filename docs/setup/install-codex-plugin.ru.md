# Установка плагина NaCl в Codex Desktop

Это обычный путь установки NaCl для пользователей Codex. Установка, обновление, отключение и удаление выполняются в интерфейсе Codex Plugins.

<!-- doc-key: availability -->
## Доступность

Проверенный Git-канал доступен как immutable GitHub Release [`codex-plugin-v0.1.0`](https://github.com/ITSalt/NaCl/releases/tag/codex-plugin-v0.1.0). Это публичный исходный artifact для установки через repo marketplace, но не карточка официального Plugins Directory.

Официальная цель — одна карточка **NaCl Skills-only** в Plugins Directory.
Она пока не опубликована. После публикации обычный пользователь один раз
установит эту карточку, запустит `nacl-init`, разрешит пакетным скриптам создать
Neo4j Community, проектный `neo4j-mcp`, launcher без секрета и доверенный
проектный `.codex/config.toml`, а затем откроет новую задачу. GitHub останется
каналом исходников, аудита, релизов и поддержки, но не
обязательной второй установкой.

GitHub tag фиксирует точную версию исходников. Установленная карточка остаётся источником истины для имени, plugin version, подключений и разрешений. Не устанавливайте archive или marketplace из fork, branch или tag, которые не указаны в этой инструкции.

<!-- doc-key: requirements -->
## Требования

- Codex Desktop с аутентифицированной учётной записью.
- Node.js 20 или новее, доступный runtime плагина. Node.js 24 проверен; запуск именно на Node.js 20 имеет статус `NOT_RUN`.
- Docker нужен только при выборе локального Neo4j-графа.
- macOS Keychain нужен только текущему опциональному пилоту локального графа. Живой bootstrap через Keychain имеет статус `NOT_RUN`.

Для проверки установки и первого dry run граф не требуется.

<!-- doc-key: official-skills-only -->
## Официальный Skills-only путь после публикации

Этот раздел описывает утверждённую цель, а не доступную сейчас карточку. Не
устанавливайте похожую по названию карточку, пока эта страница не содержит
ссылку на официальную публикацию.

1. В **Plugins** найдите официальную карточку **NaCl** типа Skills-only и
   нажмите **Install**.
2. Проверьте и выдайте только показанные разрешения, затем откройте новую
   задачу в проекте, который нужно инициализировать. Если Codex попросит,
   доверьте именно канонический корень проекта: alias пути или другой worktree
   образует другую границу доверия.
3. Вызовите `nacl-init`. Сначала он обязан показать read-only проверку
   предпосылок и план изменений; отказ от confirmation не должен ничего менять.
4. После подтверждения пакетные скрипты создают или подключают граф проекта,
   устанавливают запиненный и проверенный по checksum `neo4j-mcp`, создают
   launcher, получающий credentials только при запуске, и объединяют
   `[mcp_servers.nacl_neo4j]` в проектном `.codex/config.toml`, не
   перезаписывая другие настройки.
5. Когда init сообщит, что `.codex/config.toml` записан и прочитан обратно,
   остановите текущую задачу и
   откройте **новую задачу в том же проекте**. Это обязательная граница
   перезагрузки проектного MCP, а не вторая установка плагина.
6. Продолжайте init только после проверки MCP handshake, graph/schema health,
   именованного чтения, отдельно подтверждённого write canary и read-back в
   новой задаче.

Сгенерированный TOML может содержать абсолютный путь этой машины к launcher,
но не сырой секрет, путь developer checkout или plugin cache. Проектный
`.mcp.json` относится только к Claude Code/full-plugin compatibility и не
доказывает pickup в Codex. Если новая задача не видит `nacl_neo4j`, проверьте,
что она открыта в том же каноническом доверенном корне; support-диагностика
может сравнить `codex mcp list --json`, но эта terminal-проверка не является
частью обычной установки.

Точные финальные prompts и URL карточки появятся только после OpenAI review и
post-publication smoke того же загруженного дерева skills.

<!-- doc-key: install -->
## Установка через интерфейс из GitHub Release

1. Откройте [релиз `codex-plugin-v0.1.0`](https://github.com/ITSalt/NaCl/releases/tag/codex-plugin-v0.1.0) и скачайте **Source code (zip)** из раздела **Assets**.
2. Распакуйте ZIP в постоянную папку. Не перемещайте и не переименовывайте вложенные `.agents/plugins/marketplace.json` и `plugins/nacl`.
3. В Codex Desktop выберите открытие локальной папки/проекта и укажите корень распакованной папки `NaCl-codex-plugin-v0.1.0`, а не вложенный `plugins/nacl`.
4. Откройте **Plugins**, выберите marketplace **NaCl**, откройте карточку **NaCl** и нажмите **Install** или **+**.
5. Если Codex запрашивает подключения или разрешения, сравните их с карточкой и выдайте только показанные. Текущий bundle объявляет **Read** и **Write**; не подтверждайте более широкий или непонятный запрос.
6. Дождитесь, пока NaCl появится среди установленных.
7. Полностью завершите Codex Desktop, откройте снова и создайте новую задачу.

Если ваша версия Codex не показывает repo marketplace после открытия папки, добавьте тот же immutable Git tag одной командой, затем вернитесь в UI и выполните шаги 4-7:

```text
codex plugin marketplace add ITSalt/NaCl --ref codex-plugin-v0.1.0
```

Команда только регистрирует Git marketplace. Установка, разрешения и включение плагина по-прежнему выполняются в **Plugins**.

Новая задача обязательна для обнаружения. Полный перезапуск входил в верифицированный trial NaCl и также не позволяет предыдущей задаче удерживать устаревшее состояние плагина.

<!-- doc-key: verify -->
## Проверка в новой задаче

Отправьте этот точный запрос:

```text
Call nacl_installation_doctor exactly once with no arguments. Report status, mode, pluginVersion, and executionLocation. Continue only if status=VERIFIED and mode=plugin-only.
```

Проверьте все четыре поля:

- `status=VERIFIED`;
- `mode=plugin-only`;
- `pluginVersion` равен версии на установленной карточке;
- `executionLocation=installed-cache`.

Не фиксируйте и не выводите ожидаемую версию самостоятельно. При любом отличии остановитесь до работы с проектом или графом.

<!-- doc-key: lifecycle -->
## Обновление, переустановка, отключение, удаление или откат

Откройте **Plugins**, выберите установленную карточку NaCl и используйте действие, показанное Codex.

Шаги ниже проверены для текущего immutable Git/full-plugin канала. Официальная
Skills-only карточка должна отдельно пройти update, uninstall-persistence и
rollback smoke, прежде чем эта инструкция назовёт её lifecycle проверенным.

- **Обновление:** выберите **Update**, когда действие доступно, затем полностью перезапустите Codex, создайте новую задачу и сравните возвращённую версию с обновлённой карточкой.
- **Переустановка:** удалите bundle, снова откройте repo marketplace из того же GitHub tag, установите карточку, полностью перезапустите Codex и проверьте в новой задаче.
- **Отключение / включение:** переключите состояние NaCl на карточке. После повторного включения полностью перезапустите Codex и проверьте в новой задаче.
- **Удаление:** выберите **Uninstall plugin** на карточке. Это удаляет bundle, но не удаляет долговременные данные графа проекта, реестр, аудит, профили или состояние Keychain. Connectors управляются Codex отдельно и не удаляются только из-за удаления bundle плагина.
- **Откат:** выберите предыдущий явно опубликованный `codex-plugin-v*` release, запишите текущую версию карточки, удалите её, установите marketplace из выбранного immutable tag, полностью перезапустите Codex и убедитесь, что doctor сообщает версию этой карточки. Никогда не используйте случайный branch или fork.

Если update или rollback недоступен на карточке, остановитесь и запросите у владельца нужную карточку; не придумывайте URL установки.

<!-- doc-key: cache-and-persistence -->
## Кеш и сохранность данных

Установленный плагин работает из installed cache Codex. Изменение другой копии не обновляет установленный bundle. Выполните update или reinstall через карточку, затем полностью перезапустите Codex и создайте новую задачу перед сравнением результата doctor.

Заменяемый код плагина отделён от долговременного состояния. Удаление, переустановка, обновление, отключение и откат сохраняют конфигурацию проекта, реестр проектов, санитизированные audit records, Docker-тома графа, резервные копии, ссылки и значения Keychain и опциональные профили агентов проекта. В NaCl намеренно нет действия плагина, удаляющего данные графа.

<!-- doc-key: permissions-and-data -->
## Разрешения и обработка данных

Текущая карточка плагина объявляет **Read** и **Write**. Read нужен для проверки проекта и сбора evidence. Write используется только когда выбранному workflow нужна одобренная запись файла или графа; собственные confirmation gates NaCl не заменяют запрос разрешения Codex.

Текущий Git/full-plugin пакет запускает локальный Node.js MCP-процесс и
предоставляет десять публичных скиллов и ограниченные инструменты. Планируемая
официальная Skills-only карточка не использует этот пакетный MCP: `nacl-init`
запускает пакетные скрипты и записывает локальную MCP-конфигурацию выбранного
проекта. Без опционального графа чтение проекта и одобренная запись остаются в
границах разрешений задачи и проекта Codex. При включённом графе NaCl направляет
только явно разрешённый проект в loopback Neo4j endpoint. macOS-пилот получает
credential из Keychain в память; секрет не возвращается, не логируется, не
попадает в аргументы инструментов и не хранится в кеше плагина. Graph queries
именованы и параметризованы; произвольные graph statements не принимаются.

Не подтверждайте неожиданный сетевой доступ, значения секретов, несвязанные папки или destructive-data операции. Остановитесь и соберите support evidence.

<!-- doc-key: confirmations -->
## Подтверждения

Перед мутацией обязателен read-only план. Проверьте план, затрагиваемый проект, точные действия, блокеры и возвращённое подтверждение. Отправляйте только confirmation, возвращённый последним планом. Никогда не создавайте, не сокращайте, не используйте повторно и не фиксируйте token или confirmation из примера.

Если состояние изменилось после планирования, запросите новый план. `BLOCKED`, `FAILED`, `PARTIALLY_VERIFIED`, `NOT_RUN` и `UNVERIFIED` — состояния остановки, а не разрешение на слепой повтор.

<!-- doc-key: troubleshooting -->
## Диагностика

| Симптом | Безопасное действие |
|---|---|
| Нет marketplace или карточки | Убедитесь, что в Codex открыт корень распакованного release, содержащий `.agents/plugins/marketplace.json`. Если repo marketplace не появился, используйте приведённую выше команду с точным tag. Не подставляйте branch или fork. |
| NaCl отключён | Включите его на установленной карточке, полностью перезапустите Codex и используйте новую задачу. |
| Doctor показывает старую версию | Сравните с карточкой, выполните update или reinstall в UI, полностью перезапустите Codex и используйте новую задачу. |
| Doctor сообщает `both` | Остановите все workflows. Используйте [приложение о legacy-миграции](codex-legacy-compatibility.ru.md); никогда не удаляйте неизвестные артефакты. |
| Разрешение отклонено | Оставьте его отклонённым, пока не поймёте scope запроса. Повторяйте только нужное действие из свежей задачи или плана. |
| Node.js не поддерживается | Установите поддерживаемый runtime Node.js 20+, перезапустите Codex и снова вызовите doctor. Точный Node.js 20 не проверен; протестирован Node.js 24. |
| Граф остановлен или нездоров | Оставьте workflow writes заблокированными. Проверьте доступность Docker, затем запросите local graph doctor, graph health и schema status. |
| Keychain недоступен | Оставьте работу с графом заблокированной. Не вставляйте и не генерируйте credential; следуйте возвращённому recovery code. Живой bootstrap через Keychain пока не верифицирован. |
| Дублирующая или неоднозначная установка | Оставьте только один режим. Для плагина вместе с legacy-ссылками используйте ограниченную plan/apply миграцию. Для дублирующих карточек сохраните подтверждённую владельцем и удалите другую через её UI. |

<!-- doc-key: support -->
## Evidence для поддержки

Передайте владельцу или maintainer:

- снимок карточки NaCl с именем, версией, состоянием enabled и показанными разрешениями;
- `contract`, `status`, `mode`, `pluginVersion`, `executionLocation` и guidance от doctor;
- точный workflow prompt и возвращённые `status` / `code`;
- информацию о полном перезапуске и новой задаче;
- для ошибок графа — коды graph lifecycle, health и schema status.

Скройте имена учётных записей и несвязанное содержимое workspace. Никогда не передавайте credentials, значения Keychain, исходные или бизнес-данные проекта, персональные пути, содержимое графа или неограниченные логи.
