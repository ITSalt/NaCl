# Правила отслеживания заглушек (Stub Tracking Rules)

## 1. Назначение

Заглушки (stubs), TODO-маркеры и моковые реализации -- неизбежная часть разработки. Разработчик создает временную реализацию, чтобы двигаться дальше, но часто забывает вернуться и заменить её настоящим кодом. В результате:

- В продакшн уходят `throw new Error('Not implemented')`, ломая приложение при первом вызове
- Обход авторизации (`if (DEV) return true`) остается в коде и создает уязвимость
- Захардкоженные данные (`return [{ id: 1, name: 'Test' }]`) проходят QA, но ломаются в реальных условиях
- Пропущенная валидация позволяет невалидным данным попасть в базу

**Цель stub tracking** -- гарантировать, что ни одна заглушка не доживет до релиза незамеченной. Каждая заглушка регистрируется, классифицируется по критичности и блокирует продвижение задачи, если не разрешена.

---

## 2. Типы заглушек

### a. Комментарные маркеры (Comment Markers)

Текстовые маркеры в комментариях, обозначающие намерение доработать код:

| Маркер | Значение | Пример |
|--------|----------|--------|
| `// TODO:` | Функциональность запланирована, но не реализована | `// TODO: добавить пагинацию` |
| `// FIXME:` | Известная проблема, требующая исправления | `// FIXME: race condition при параллельных запросах` |
| `// STUB:` | Заглушка вместо реальной реализации | `// STUB: возвращаем пустой массив вместо запроса к API` |
| `// MOCK:` | Моковые данные вместо реального источника | `// MOCK: захардкоженный список пользователей` |
| `// HACK:` | Временное обходное решение | `// HACK: перезагружаем страницу вместо обновления стейта` |

### b. Паттерны в коде (Code Patterns)

Конструкции, которые сами по себе указывают на заглушку, даже без комментария:

```typescript
// Явное "не реализовано"
throw new Error('Not implemented');
throw new Error('TODO');
throw new NotImplementedError();

// Приведение к any для обхода типизации
return {} as any;
return [] as any;
return null as any;
return undefined as any;

// Пустые реализации
async function fetchUsers(): Promise<User[]> {
  return []; // пустой массив вместо реального запроса
}

// Заглушки в обработчиках
catch (error) {
  // TODO: обработать ошибку
}
```

### c. Моковые данные (Mock Data)

Захардкоженные данные, подменяющие реальный источник:

```typescript
// Захардкоженные массивы
const users = [
  { id: 1, name: 'John Doe', email: 'john@example.com' },
  { id: 2, name: 'Jane Doe', email: 'jane@example.com' },
];

// Фейковые ответы API
function getOrderStatus() {
  return { status: 'completed', updatedAt: '2025-01-01' };
}

// Плейсхолдеры
const description = 'Lorem ipsum dolor sit amet';
const imageUrl = 'https://via.placeholder.com/150';
const FAKE_TOKEN = 'eyJhbGciOiJIUzI1NiJ9.fake-token';
```

### d. Временные реализации (Temporary Implementations)

Упрощенная логика, осознанно пропускающая важные шаги:

```typescript
// Пропуск валидации
function createOrder(data: any) {
  // TODO: validate input
  return db.orders.create(data);
}

// Обход авторизации
function checkPermission(user: User, action: string): boolean {
  return true; // STUB: всегда разрешаем на время разработки
}

// Упрощенная бизнес-логика
function calculateDiscount(order: Order): number {
  return 0; // TODO: реализовать систему скидок
}

// Пропуск rate limiting
// TODO: добавить rate limiter
app.post('/api/orders', createOrderHandler);
```

## 3. Классификация по критичности (Severity Classification)

### CRITICAL

Заглушки, которые создают риск безопасности, потери данных или некорректной бизнес-логики:

| Паттерн | Почему критично |
|---------|-----------------|
| Обход авторизации (`return true`) | Любой пользователь получает доступ |
| Пропущенная валидация входных данных | SQL-инъекция, невалидные данные в БД |
| Заглушка в бизнес-логике (расчеты, статусы) | Неверные суммы, потеря заказов |
| Захардкоженные секреты/токены | Утечка credentials |
| `throw new Error('Not implemented')` в основном потоке | Крэш приложения при использовании |

**Действие**: БЛОКИРУЕТ ревью и релиз. Должна быть разрешена немедленно.

### WARNING

Заглушки в некритичных участках, которые не создают немедленного риска, но снижают качество:

| Паттерн | Почему важно |
|---------|--------------|
| TODO в обработке ошибок | Ошибки проглатываются молча |
| Неполная обработка edge cases | Баги при граничных условиях |
| Заглушка в UI-компонентах | Некорректное отображение |
| Пропущенное логирование | Невозможность отладки на проде |
| `return [] as any` в нестыковочных местах | Маскирует реальные проблемы типов |

**Действие**: ПРЕДУПРЕЖДЕНИЕ при ревью. Желательно разрешить до релиза.

### INFO

Заметки для будущих улучшений, не влияющие на текущую функциональность:

| Паттерн | Описание |
|---------|----------|
| TODO по оптимизации | `// TODO: добавить кэширование` |
| Заметки по документации | `// TODO: написать JSDoc` |
| Косметические улучшения | `// TODO: вынести в отдельный компонент` |
| Планы на рефакторинг | `// TODO: заменить на стратегию` |

**Действие**: ИНФОРМАЦИЯ. Фиксируется в реестре, не блокирует ничего.

## 4. Стандартный формат маркера

Все заглушки, создаваемые в рамках разработки по UC, должны использовать унифицированный формат:

```
// STUB(UC###): краткое описание
```

### Правила

1. **Тип маркера** -- одно из: `TODO`, `FIXME`, `STUB`, `MOCK`, `HACK`
2. **UC-ссылка обязательна** -- в скобках указывается ID Use Case
3. **Описание** -- лаконичное, но информативное (что заглушено и почему)

### Примеры

```typescript
// STUB(UC001): возвращаем пустой массив вместо запроса к БД
async function getOrders(): Promise<Order[]> {
  return [];
}

// TODO(UC003): реализовать фильтрацию по дате
function filterOrders(orders: Order[], filters: Filters): Order[] {
  return orders; // фильтры игнорируются
}

// MOCK(UC005): захардкоженные категории, заменить на API
const categories = ['Electronics', 'Clothing', 'Food'];

// HACK(UC002): обходим баг с рендером таблицы при 0 элементов
if (items.length === 0) {
  items = [{ id: 'empty', name: '-' }];
}

// FIXME(UC004): SQL-инъекция, нужен параметризованный запрос
const query = `SELECT * FROM users WHERE name = '${name}'`;
```

### Маркер без UC-ссылки

Если заглушка найдена без UC-ссылки -- это нарушение. Сканер помечает её как `orphaned` и требует привязки к конкретной задаче.

## 5. Реестр заглушек (stub-registry.json)

Все обнаруженные заглушки фиксируются в файле `.tl/stub-registry.json`.

### Схема

```json
{
  "$schema": "stub-registry-v1",
  "updatedAt": "2025-01-30T14:30:00Z",
  "stats": {
    "total": 12,
    "critical": 1,
    "warning": 5,
    "info": 6,
    "resolved": 8,
    "unresolved": 4
  },
  "stubs": [
    {
      "id": "STUB-001",
      "file": "src/orders/order.service.ts",
      "line": 45,
      "type": "STUB",
      "severity": "CRITICAL",
      "text": "// STUB(UC001): возвращаем пустой массив вместо запроса к БД",
      "description": "Метод getOrders() не обращается к базе данных",
      "uc": "UC001",
      "createdAt": "2025-01-30T10:00:00Z",
      "resolvedAt": null
    },
    {
      "id": "STUB-002",
      "file": "src/orders/order.controller.ts",
      "line": 12,
      "type": "TODO",
      "severity": "INFO",
      "text": "// TODO(UC001): добавить JSDoc для эндпоинта",
      "description": "Отсутствует документация API-метода",
      "uc": "UC001",
      "createdAt": "2025-01-30T10:15:00Z",
      "resolvedAt": "2025-01-30T14:00:00Z"
    }
  ]
}
```

### Поля записи

| Поле | Тип | Обязательно | Описание |
|------|-----|-------------|----------|
| `id` | string | Да | Уникальный ID: `STUB-NNN` |
| `file` | string | Да | Относительный путь к файлу |
| `line` | number | Да | Номер строки |
| `type` | enum | Да | `TODO`, `FIXME`, `STUB`, `MOCK`, `HACK` |
| `severity` | enum | Да | `CRITICAL`, `WARNING`, `INFO` |
| `text` | string | Да | Полный текст маркера из кода |
| `description` | string | Нет | Расширенное описание проблемы |
| `uc` | string | Да | ID Use Case (UC001, UC002 и т.д.) |
| `createdAt` | ISO 8601 | Да | Дата обнаружения |
| `resolvedAt` | ISO 8601 | Нет | Дата разрешения (`null` если не разрешена) |

## 6. Правила сканирования (Scanning Rules)

### Регулярные выражения для поиска маркеров

```
# Комментарные маркеры (с UC-ссылкой и без)
\/\/\s*(TODO|FIXME|STUB|MOCK|HACK)(\([A-Z]{2}\d{3}\))?:\s*(.+)

# Незавершенные реализации
throw\s+new\s+Error\(['"`](Not implemented|TODO|FIXME|not yet)['"`]\)
throw\s+new\s+NotImplementedError\(

# Приведение к any
return\s+(\{\}|\[\]|null|undefined)\s+as\s+any

# Пустые catch-блоки
catch\s*\([^)]*\)\s*\{\s*(\/\/.*?)?\s*\}

# Захардкоженные плейсхолдеры
['"`](Lorem ipsum|placeholder|TODO|FIXME|test@example|fake|dummy)['"`]

# Обход авторизации
return\s+true\s*;?\s*\/\/.*?(stub|todo|hack|temporary|dev)

# Фейковые токены и секреты
(FAKE_|MOCK_|TEST_|DUMMY_)(TOKEN|SECRET|KEY|PASSWORD)
```

### Файлы для сканирования

```
# Включать
src/**/*.ts
src/**/*.tsx
src/**/*.js
src/**/*.jsx

# Исключать
node_modules/
dist/
build/
**/*.test.ts        # тесты могут содержать моки по назначению
**/*.spec.ts
**/*.test.tsx
**/*.spec.tsx
tests/fixtures/**   # тестовые фикстуры -- это легитимные моки
__mocks__/**
```

### Правила классификации при сканировании

```
1. Если маркер STUB/MOCK в файле *.service.ts → CRITICAL
2. Если паттерн "return true" + auth/permission → CRITICAL
3. Если throw NotImplementedError в основном потоке → CRITICAL
4. Если FIXME с упоминанием injection/xss/security → CRITICAL
5. Если TODO в catch-блоке → WARNING
6. Если STUB/MOCK в *.controller.ts → WARNING
7. Если return [] as any → WARNING
8. Если TODO с упоминанием docs/jsdoc/readme → INFO
9. Если TODO с упоминанием optimize/cache/refactor → INFO
10. Если HACK без security-контекста → WARNING (по умолчанию)
```

## 7. Блокировки (Gates)

Заглушки блокируют продвижение задачи на определенных этапах:

### Перед ревью (tl-review)

| Условие | Действие |
|---------|----------|
| CRITICAL stubs > 0 | **BLOCK** -- ревью невозможно, вернуть на доработку |
| WARNING stubs > 3 | **WARNING** -- ревью возможно, но нужно обоснование |
| Orphaned stubs (без UC) | **BLOCK** -- все заглушки должны быть привязаны к UC |

### Перед QA (tl-qa)

| Условие | Действие |
|---------|----------|
| Любой STUB/MOCK в UI-компонентах | **WARNING** -- QA может дать ложные результаты |
| Моковые данные в API-ответах | **WARNING** -- тесты пройдут, но результат ненадежен |

### Перед релизом (--final)

| Условие | Действие |
|---------|----------|
| Любая неразрешенная заглушка любого severity | **BLOCK** -- релиз невозможен |
| Exception: INFO с явным обоснованием в описании | **WARNING** -- допустимо с подтверждением |

### Формат блокировки в отчете

```markdown
## Stub Gate: BLOCKED

**Этап**: pre-review
**Причина**: Обнаружены критические заглушки

| ID | Файл | Severity | Описание |
|----|------|----------|----------|
| STUB-001 | src/orders/order.service.ts:45 | CRITICAL | Пустая реализация getOrders() |
| STUB-003 | src/auth/auth.guard.ts:12 | CRITICAL | Обход авторизации |

**Действие**: Разрешите все CRITICAL-заглушки и повторно запустите сканирование.
```

## 8. Интеграция в workflow

### Этап разработки (tl-dev-be / tl-dev-fe)

1. Разработчик создает заглушки с правильным форматом маркера: `// STUB(UC###): описание`
2. После завершения TDD-цикла запускается автоматическое сканирование
3. Результат записывается в `.tl/tasks/UC###/stub-report.md`
4. Реестр `.tl/stub-registry.json` обновляется

**stub-report.md** (создается автоматически):

```markdown
# Stub Report: UC001

**Дата**: 2025-01-30
**Файлов просканировано**: 7
**Заглушек найдено**: 3

## CRITICAL (1)

| ID | Файл:строка | Маркер | Описание |
|----|-------------|--------|----------|
| STUB-001 | src/orders/order.service.ts:45 | STUB | Пустая реализация getOrders() |

## WARNING (1)

| ID | Файл:строка | Маркер | Описание |
|----|-------------|--------|----------|
| STUB-004 | src/orders/order.controller.ts:28 | TODO | Неполная обработка ошибок |

## INFO (1)

| ID | Файл:строка | Маркер | Описание |
|----|-------------|--------|----------|
| STUB-002 | src/orders/order.controller.ts:12 | TODO | Добавить JSDoc |

## Вердикт

**BLOCKED** -- 1 критическая заглушка. Ревью невозможно до разрешения.
```

### Этап ревью (tl-review)

1. Ревьюер читает `stub-report.md` из директории задачи
2. Если есть CRITICAL -- задача отклоняется автоматически
3. Если есть только WARNING/INFO -- ревьюер принимает решение
4. В `review.md` добавляется секция "Stub Check"

### Финальная проверка (tl-stubs --final)

1. Полное сканирование всей кодовой базы (не только файлов текущей задачи)
2. Проверка всех неразрешенных записей в `stub-registry.json`
3. Генерация финального отчета
4. Любая неразрешенная заглушка (кроме INFO с обоснованием) блокирует релиз

## 9. Разрешение заглушек

### Процесс разрешения

1. **Реализовать** -- написать настоящий код вместо заглушки
2. **Удалить маркер** -- убрать комментарий `// STUB(UC###): ...`
3. **Обновить реестр** -- установить `resolvedAt` в `stub-registry.json`
4. **Повторно просканировать** -- убедиться, что заглушка больше не обнаруживается

### Пример: до и после

**До (заглушка)**:

```typescript
// STUB(UC001): возвращаем пустой массив вместо запроса к БД
async function getOrders(): Promise<Order[]> {
  return [];
}
```

**После (реализация)**:

```typescript
async function getOrders(): Promise<Order[]> {
  return await this.orderRepository.findAll();
}
```

### Обновление реестра

```json
{
  "id": "STUB-001",
  "file": "src/orders/order.service.ts",
  "line": 45,
  "type": "STUB",
  "severity": "CRITICAL",
  "text": "// STUB(UC001): возвращаем пустой массив вместо запроса к БД",
  "uc": "UC001",
  "createdAt": "2025-01-30T10:00:00Z",
  "resolvedAt": "2025-01-30T14:30:00Z"
}
```

### Нельзя делать

- Удалять запись из реестра -- запись остается навсегда с `resolvedAt`
- Менять severity после регистрации -- severity фиксируется при обнаружении
- Разрешать заглушку без реализации -- удаление маркера без кода = баг
- Переносить заглушку в другой файл -- это создает новую запись, старая остается orphaned

### Массовая проверка разрешения

```bash
# Проверить, что все маркеры из реестра действительно удалены из кода
# Для каждой записи с resolvedAt != null проверить, что grep по файлу не находит маркер
```

## Чек-лист для разработчика

- [ ] Все мои заглушки используют формат `// TYPE(UC###): описание`
- [ ] Каждая заглушка привязана к конкретному UC
- [ ] CRITICAL-заглушки разрешены до отправки на ревью
- [ ] `stub-report.md` сгенерирован и не содержит CRITICAL
- [ ] `stub-registry.json` обновлен

## Чек-лист для ревьюера

- [ ] Проверен `stub-report.md` в директории задачи
- [ ] Нет CRITICAL-заглушек
- [ ] WARNING-заглушки обоснованы или запланированы к разрешению
- [ ] Нет orphaned-заглушек (без UC-ссылки)
- [ ] В `review.md` зафиксирован результат проверки заглушек
